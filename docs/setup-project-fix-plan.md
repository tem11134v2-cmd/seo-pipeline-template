## Раздел 1. Реальные баги (must fix)

### Баг #1 - кириллица в имени файла ломает pre-commit hook (KEYSTONE)

**Где:** `.claude/git-hooks/pre-commit:93`.

**Симптом:** при `quotepath=true` (дефолт git) `git diff --cached --name-only` отдаёт путь к кириллическому файлу как `".claude/handoff-requests/files/\320\227..."` - в двойных кавычках и octal-эскейпе. Ведущая `"` ломает `case`-матч по разрешённым префиксам (`.claude/handoff-requests/`), и хук считает разрешённый файл нарушением. Проверено эмпирически на этом репо.

**Корень глубже, чем setup:** баг кусает ЛЮБОЙ worktree-коммит с кириллицей в имени файла и срабатывает дважды - на собственном коммите `setup-project` и на `git commit` внутри `/handoff` (шаг 2). Прошлый агент обошёл через ручной `git -c core.quotepath=false commit`, следующий пойдёт в `--no-verify` и потеряет защиту хука.

**Фикс (1 строка):**
```bash
# было:
changed=$(git diff --cached --name-only)
# стало:
changed=$(git -c core.quotepath=false diff --cached --name-only)
```

**Сопутствующее:**
- Добавить комментарий рядом со строкой: «-c core.quotepath=false обязателен, иначе non-ASCII имена (ЗАКАЗЧИК.md) эскейпятся в octal + кавычки и не матчатся».
- Дописать абзац в `docs/adr/003-pre-commit-whitelist.md`: любой будущий код, который МАТЧИТ имена из git-вывода (diff/status/ls-files), обязан использовать `-c core.quotepath=false`.
- Проверить, что в `pre-commit` это единственное место, где имена из git-вывода матчатся (на момент разбора - да; `git status --porcelain` в `/handoff` используется только для проверки непустоты, матчинга нет - там фикс не нужен).

**Приоритет:** наивысший. Разблокирует setup-project и handoff. Делать первым.

### Баг #3 - маркеры `expected-*` выглядят мёртвым кодом (мисдиагноз)

**Где:** `setup-project/SKILL.md:39-43, 60-64` и аналогично в других скилах.

**Переосмысление:** маркеры НЕ мёртвые. Их читает `.claude/hooks/check-file.sh` (SubagentStop-хук): проверяет, что субагент создал непустой файл (для `.md` - ещё и наличие заголовка), затем удаляет маркер. Прошлый агент не нашёл потребителя, потому что это хук, а не сам скил.

**Реальные дыры:**
1. Скил нигде не объясняет контракт «маркер -> SubagentStop-хук check-file.sh».
2. `<run_id>` - неопределённый плейсхолдер; спасает только fallback в хуке (`ls -t expected-*.txt | head -1`).

**Фикс:**
- 2 строки комментария в скиле у первого маркера: «Маркер читает SubagentStop-хук `check-file.sh` - проверяет, что субагент создал непустой выходной файл. Имя: `expected-<agent>-<любой-уникальный-суффикс>.txt`; хук матчит по имени агента и берёт свежайший».
- Опционально: заменить `<run_id>` на явное «timestamp или короткий id».

**Приоритет:** низкий (документация/ясность).

## Раздел 2. Точки роста

### GP1 - client-profiler жёстко читает sitemap.xml (усиление существующего)

**Где:** `.claude/agents/client-profiler.md:30-34`.

**Сейчас:** мягкая опция «fetch sitemap.xml ИЛИ парсинг меню». На практике sitemap закрывает половину «уточнить у клиента» по перелинковке.

**Фикс:** поднять из опции в жёсткий шаг сбора:
1. `robots_txt` (или fetch `/robots.txt`) -> вытащить директиву `Sitemap:`.
2. fetch sitemap.xml (и вложенные индексы sitemap).
3. Извлечь реальные URL разделов (`/category/...`, `/selection/...`, и пр.), сгруппировать по тематике.
4. Заполнить таблицу «Перелинковка» РЕАЛЬНЫМИ URL, а не логичными паттернами.
5. Только если sitemap недоступен (нет robots, 403, пустой) - fallback на парсинг меню, и лишь потом - вопрос пользователю.

**Инструменты:** `robots_txt` (keys.so), `mcp_fetch_page`. У `client-profiler` нет ограничения `tools:` -> доступ есть.

**Приоритет:** высокий, дёшево.

### GP2 - вынести вопросы к пользователю из субагента в скил (структурный фикс + AskUserQuestion)

**Где:** `client-profiler.md:51` (вопрос про площадки в чате) + `setup-project/SKILL.md:53-56`.

**Корень (не замечен прошлым агентом):** `client-profiler` - СУБАГЕНТ, он не может по-настоящему встать на паузу и дождаться интерактивного ответа в середине прогона. Его «Спросить у пользователя в чате» скорее всего не работает как интерактив - вопрос уходит в текст возврата без ответа. То есть идея «3 вопроса автоматом» лечит не неудобство, а скрытый дефект.

**Фикс (структурный):**
1. Убрать из `client-profiler` все обращения к пользователю. Субагент только собирает что может и помечает остальное `_не заполнено_` / «под вопросом», возвращая структурированный список открытых полей.
2. В `setup-project` ПОСЛЕ возврата профиля задать вопросы через `AskUserQuestion` одним раундом:
   - **Дистрибуция** (multiSelect): VC, Хабр, Дзен, Telegram, + Other. (Раньше это спрашивал субагент - переносим сюда.)
   - **Подкатегории-URL**: после GP1 в основном автозаполнены -> вопрос вырождается в «URL подтянуты из sitemap, всё верно? [да / поправлю]».
   - **Автор**: свободный текст -> точечный вопрос (AskUserQuestion слаб на free-text; дать как один вопрос с «Other»-вводом либо отдельной строкой).
   - Любые прочие `_не заполнено_` из возврата субагента - добить в этот же раунд (лимит AskUserQuestion: до 4 вопросов).
3. Применить ответы к профилю через Edit (без повторного прогона субагента).

**Эффект:** чинит «субагент не умеет спрашивать» (баг A), реализует идею про 3 вопроса и убирает цикл «прочитал -> правки -> новый профиль».

**Приоритет:** высокий (самая мясная правка).

### GP6 - самовалидация контракта setup-meta.json

**Где:** `setup-project/SKILL.md:86-110` (шаг 5) + контракт с `handoff-process` 4.1.

**Сейчас:** скил не знает наверняка, что `handoff-process` корректно прочитает `setup-meta.json`. `handoff-process` 4.1 делает `cp source -> target` без проверки, что source существует.

**Фикс:** на шаге 5 setup-project сам себя валидирует (перечитать только что записанный JSON):
1. Обязательные ключи: `type`, `files[]` с `source`/`target`/`operation`.
2. Каждый `source` существует и непуст.
3. Если что-то не так - не финализировать, сообщить и предложить перегенерацию упавшего артефакта.

**Бонус:** ловит частичный провал (template-designer упал -> template.html нет -> поймали ДО handoff, а не в `cp` несуществующего внутри handoff-process). Это закрывает баг B (атомарность).

**Downstream-страховка:** упомянуть в финале, что `/handoff-process --dry-run` показывает план до применения.

**Приоритет:** средний.
