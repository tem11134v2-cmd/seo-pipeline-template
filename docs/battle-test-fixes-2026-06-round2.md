## ТОЧКА 1 - команда /status (обзор всех проектов) + контроль забытых worktree

> **СТАТУС: ✅ СДЕЛАНО (2026-06-04).** Скрипт `~/seo-projects/.claude/scripts/project-status.mjs` (derive из `meta.json` + `git worktree list`, 3 слоя, сортировка по свежести) + родительский скил `status` + node-врапперы; закоммичено и запушено в `seo-projects-parent`. Оппортунистическая напоминалка о живых worktree - в конце `handoff-process` (шаг 8). Авто-действий нет (трёхходовка соблюдена). Боевой прогон по 16 клиентам: вытащил застрявшие структуру+метатеги (`msk_trans_ru`, +3 коммита, НЕ ХЕНДОФНУТО), 5 worktree структуры (`spk_nhs_ru`), 14 zombie-папок (`mansband_ru`).

### Идея пользователя

Команда `/status` в родительской `~/seo-projects/` (рядом с `/new-project`). Показывает, на каком
этапе все последние (по дате) проекты: у этого - анализ+структура+тексты, у того - setup+темы+6 статей,
у третьего - только стратегия. Плюс: считать число повторных/параллельных услуг (3 структуры = 3 раза
делали) - точнее, **сколько хендофов** со структурой было (worktree без `/handoff` наружу не пробивается).
Доп.: в конце `/handoff-process` проверять, сколько worktree осталось - чтобы забытые увидеть, а с
ненужными осторожно предложить либо хендофнуть, либо удалить.

### Критический разбор: идея верная, но предложенный МЕХАНИЗМ повторяет ровно тот анти-паттерн, который раунд 1 только что убрал

Пользователь спрашивает: «нужно будет добавить новый json в шаблон, чтобы он отражал картину, и обновлять
его на `/handoff`?». **Нет - и это важно.** Это в точности конструкция «общий изменяемый реестр, в который
пишет каждый worktree», которую раунд 1 (точка 2 / ADR-013) признал багом и убрал:

- **Конфликт при параллели.** Если статус-файл общий и его правит worktree - два параллельных хендофа дадут merge-конфликт (ровно история `_index.json`).
- **Протухание / «забыли обновить».** Пользователь сам же называет эту дыру: «если просто сделать ворктри и не прописать хендоф - ветка не пробьётся». Файл, который обновляют руками/на событии, неизбежно разойдётся с реальностью.
- **Лишняя машинерия.** Запрос-на-обновление + ветка в `handoff-process` - целый новый путь данных ради того, что уже лежит на диске.

**А оно уже лежит на диске.** В каждой папке задачи есть `meta.json` - подтверждённая единая схема:
`slug`, `state`, `started`, `updated` (ISO - готовый ключ сортировки «по дате»), `completed_steps[]`,
плюс типовые поля (`domain`, `analysis_nnn`, `depth`, `drive_link`). Я уже собрал нужный срез **одним
циклом** по `{strategies,analyses,structures,metatags,topics,articles}/*/meta.json`:

```
easythb_pro            strategies=1
kadrovy_element_rf     analyses=1
mansband_ru            articles=10
riosystems_ru          analyses=1 structures=1
traffic_masters_ru     topics=1 articles=9
...
```

Это и есть ядро `/status`. Никакого нового трекаемого файла.

### Переосмысление: «сколько хендофов, а не сколько раз делали» получается БЕСПЛАТНО

Пользователь сам уточнил: считать надо хендофы (что пробилось в main), а не запуски. При derive-подходе
это свойство возникает само: **в main лежат только смерженные папки задач.** Считаешь
`structures/*/meta.json` в main - считаешь именно отданные структуры. Повторы тоже считаются верно: каждый
повторный прогон создаёт НОВУЮ нумерованную папку (`structures/002-...`), все в main, все в счёте. То, что
пользователь хотел получить руками, derive отдаёт даром.

### Что делаем - три слоя статуса, каждый из своего источника

- [ ] **Новый скрипт `project-status.mjs <parent_dir>`** (как существующие сканеры). Для каждого `<slug>/`:
  1. **Отдано (в main):** скан `<project>/{strategies,analyses,structures,metatags,topics,articles}/*/meta.json`. На каждый тип - счёт + `state` (показываем не голую галку, а состояние: «структура: awaiting-client», «анализ: revising», «статей 6: completed» - застрявшие видно сразу). Самый свежий `updated` = дата проекта для сортировки.
  2. **В работе (live worktree):** `git -C <project> worktree list --porcelain` -> для каждой живой `claude/*` - скан её `meta.json`. **Главный сигнал:** «metatags completed, НО не хендофнут» (ровно случай `msk_trans_ru` - готовый деливерабл застрял в ветке) и «article writing - в процессе». Дивергенция: `git -C <project> rev-list --count <base>..claude/<name>` (>0 = есть несмерженные коммиты).
  3. **Гигиена:** zombie-папки в `.claude/worktrees/*`, которых нет в `git worktree list` (ровно логика `handoff-process` шаг 1). «N zombie-папок - подчистит `/handoff-process` или вручную».
- [ ] **Новый скил `~/seo-projects/.claude/skills/status/SKILL.md`** (рядом с `new-project`; зона - только родитель). Зовёт скрипт, форматирует вывод, сортировка по свежести (последние сверху). Пропускает `template-project` (маркер `.is-template-root`) и не-проектные папки.
- [ ] **Реальная польза подтверждена данными фронта:** `msk_trans_ru` - live worktree диверговал (0e2bfca vs main fca3bcf), metatags+структура НЕ в main; `riosystems_ru`/`kadrovy_element_rf` - zombie-папки. То есть забытая/застрявшая работа на фронте есть прямо сейчас - `/status` её показал бы.

### Контроль забытых worktree: ПЕРЕОСМЫСЛИТЬ дом, а не только добавить в handoff-process

Пользователь предложил повесить проверку на конец `/handoff-process`. Критика: **`/handoff-process`
запускается только когда есть pending shared-запросы.** Чистая per-task работа (большинство статей,
стратегия без `/request-shared-edit`) его НЕ триггерит вообще. Значит тот, кто забыл `/handoff`, ровно так
же не запустит `/handoff-process` - проверка не сработает для той самой популяции, ради которой задумана.

Поэтому:
- [ ] **Основной дом - `/status`** (обзорный инструмент, который и так зовут из родителя «посмотреть всё»). Он видит worktree всех проектов разом.
- [ ] **Вторично, оппортунистически - конец `/handoff-process`:** дешёвая приписка «в этом проекте ещё N живых worktree» (срабатывает, когда он и так запущен). Не основной механизм.
- [ ] (опц.) конец `/handoff`: «ты сдал X; в проекте ещё N живых worktree» - но возможен шум, обсуждаемо.

### Авто-хендоф / авто-удаление забытых worktree: НЕ авто, трёхходовая классификация

Пользователь сам пометил «осторожно». Делаем осторожно через классификацию (данные: `git worktree list` +
счёт дивергенции):
- [ ] **Zombie-папка** (нет в `worktree list`): безопасно удалить - **уже** делает `handoff-process` шаг 1; `/status` может предложить то же.
- [ ] **Live worktree, БЕЗ дивергенции** (`rev-list --count base..branch == 0`): пусто/уже смержено -> предложить чистку с явным подтверждением, показать команду.
- [ ] **Live worktree С дивергенцией** (несмерженные коммиты): **НИКОГДА не трогать авто.** Это реальная работа (`msk_trans`). Только подсветить: «здесь несмерженная работа [состояния задач], открой её -> `/handoff`, либо разберись». Потеря этой ветки = потеря деливерабла.
- [ ] **Авто-`/handoff` чужого worktree - запрещаем.** Он смержит возможно-недоделанную работу из другой сессии, прогонит её pre-commit с авто-сообщением, без человека в цикле. Максимум - НАПЕЧАТАТЬ команду, которую пользователь выполнит в сессии того worktree.

### Итог точки 1

`/status` = derive из `meta.json` (3 слоя) + новый скрипт-сканер + оппортунистическая приписка в конце
`handoff-process`. **БЕЗ нового трекаемого JSON, БЕЗ записи на `/handoff`.** Где живёт canonical-копия
скила `/status` - см. точку 3 (родительские скилы вне git).

## ТОЧКА 2 - русификация команд + схема нумерации

> **СТАТУС: ✅ 2A СДЕЛАНО (2026-06-04). 2B (раскатка на клиентов) - СКИП по решению пользователя.** (A) русификация + (B) namespace `seo-<глагол>` применены: 7 папок `git mv`, `name`/H1 совпадают, ~432 замены слеш-рефов (skills/agents/scripts/hooks/CLAUDE/README/guide + `workbook.creator`), голые упоминания добиты, поймано и откатано повреждение тест-путей, смоук-тесты 14/14+13/13, `grep` старых /команд = ноль. Намеренные keeps: task-type id в handoff/handoff-process + `"type":"setup-project"`, имя тест-папки, история. Раскатка на 16 клиентов (2B) НЕ делается - только новые клоны получают новые имена; существующие клиенты пользователь синкнет сам через `/sync-from-template`, когда понадобится. (C) плюмбинг (`handoff`/`share-*`/`fix-article`/`sync`/`guide`) - английский, guard-хук не тронут.

### Идея пользователя

Все команды на англ - путают ассистентов. Русифицировать (translit): `napisanie-statyi` вместо
`write-article`. Плюс мини-система: одиночки и ветки. Стратегия - одиночка -> `seo0-strategiya`. Анализ/
структура/тексты - ветка -> `seo11-analiz`, `seo12-struktura`, `seo13-teksta`, `seo14-teksta-v2`. Статьи ->
`seo21-shablon-dlya-statyi`, `seo22-temi-dlya-statyi`, `seo23-napisanie-statyi`.

### Критический разбор: это ДВА разных решения в одном, оценивать порознь

**(A) Русификация (translit) имён** и **(B) числовая таксономия трек.шаг** (`seo11/seo12/seo21`) -
независимы. Можно сделать A без B и наоборот.

#### (A) Русификация - склоняюсь ДА

- **За:** прямо лечит заявленную боль. Потребитель имён - ассистенты-люди, которые ИХ НАБИРАЮТ; им важно само имя, а не только description. Translit - правильный выбор: имя скила/слеш-команды в Claude Code - lowercase-latin/kebab (`[a-z0-9-]`), кириллица как имя почти наверняка не пройдёт. Пользователь уже интуитивно транслитерирует - это совпадает с ограничением.
- **Дешёвая страховка перед массовым переименованием:** один скил переименовать и проверить, что слеш-команда зовётся (смоук-тест), ПОТОМ остальное. Не переименовывать 8 штук вслепую.
- **Радиус (мех., но большой): ~40+ md-файлов.** Подтверждено grep: почти все файлы `agents/` (агенты ссылаются на `/write-article`, `/seo-structure`...), все `share-*`, `MCP_MAP.md`, tests, `CLAUDE.md` x2, `guide` x2, README (50KB), печатные подсказки в `new-project`/`handoff`/`handoff-process`.
- **Детерминированное закрытие (не «вычитать глазами»):** вести явную таблицу old->new; сделать переименование; потом `grep` каждого старого имени -> **ноль остатков** (verify-шаг). Это тот же сквозной принцип, что в раунде 1.

#### (B) Числовая таксономия - ценная идея, но в текущем виде хрупкая; рефайн

- **Реальная ценность (и цель пользователя):** автокомплит-открытие С порядком. Ассистент набирает `/seo2` -> видит весь трек статей по порядку. Для нетехнических ассистентов это настоящий онбординг-выигрыш и прямой ответ на «путаются, какая команда и в каком порядке».
- **Риски текущей схемы:**
  1. **Жёсткость.** Порядок зашит в имя -> вставка услуги ломает нумерацию. Пользователь УЖЕ напоролся: пришлось рожать `seo14-teksta-v2`.
  2. **Дыра покрытия.** Схема нумерует только 8 клиентских услуг. Плюмбинг (`handoff`, `handoff-process`, `sync-from-template`, `guide`, `new-project`, `status`, `share-*` x5, `fix-article`, `rewrite-section`, `request-shared-edit`) - ~13 команд - без места в схеме. Это БОЛЬШИНСТВО командной поверхности. Нужен внятный ответ, как их звать.
  3. **Две оси в одном токене.** `seo` + цифра-трек + цифра-шаг + translit-глагол: `seo14-teksta-v2` уже трудно читается.
  4. **Ложная линейность.** Стратегия - «0» (одиночка), но это же ПЕРВЫЙ шаг пресейла; анализ - «11». Глобальная нумерация подразумевает порядок, которого нет (два параллельных трека + одиночка).
- **Альтернативы:**
  - **Alt-1 (читаемый namespace, порядок - в `/guide`):** `seo-strategiya`, `seo-analiz`, `seo-struktura`, `seo-shablon`, `seo-temi`, `seo-statya`, `seo-faq`. Автокомплит на `/seo-` всё равно ГРУППИРУЕТ; порядок живёт в `/guide` (вставка услуги ничего не ломает). Теряем порядок-с-первого-взгляда, оставляем группировку + читаемость.
  - **Alt-2 (рефайн нумерации - идея пользователя, но с залатанными дырами):** оставить трек.шаг, но (i) оставить зазоры под вставку (`seo11/seo13/seo15` или `seo110/seo120`), (ii) дать плюмбингу дом (descriptive-translit без номера: `sdat-rabotu`=handoff, `primenit-zaprosy`=handoff-process; ИЛИ band `seo9x`), (iii) не кодировать `-v2` в номер.
- **МОЯ РЕКОМЕНДАЦИЯ:** это твоя таксономия и твои ассистенты - оставляю развилкой Q2-B. Склоняюсь к **Alt-1** (стойкость к вставке + простой рассказ про плюмбинг), но **Alt-2 жизнеспособен**, если порядок-с-первого-взгляда важнее цены вставки. Одного «правильного» ответа нет.

### Связанное (важно для исполнения B и рассылки)

## ТОЧКА 4 - git-репозиторий в родителе (Вариант A, решение Q3) [НОВОЕ, из обсуждения]

> **СТАТУС: ✅ СДЕЛАНО (2026-06-04).** Родитель = git-репо: `.gitignore` (`/*` + `!/.claude/` + `!/.gitignore`; верифицировано через `git check-ignore` - 16 клиентов и `template-project` игнорируются, `.gitignore` и `.claude/` трекаются, `settings.local.json` исключён), коммит из 4 файлов (без клиентов), приватный remote `github.com/tem11134v2-cmd/seo-projects-parent` + push, `main` -> `origin/main`. ### Идея пользователя

Завести git в `~/seo-projects/` ради резервной копии, поддерживать со временем.

### Критический разбор: смысл есть, но naive init = ложный бэкап (два обязательных условия)

Смысл реальный: версионирует родительские скилы НА МЕСТЕ, закрывает Q3 без «копий в шаблоне». Но:
- **`.gitignore` «всё кроме `.claude/`» обязателен.** Иначе git запишет 16 вложенных клиентских репо как гитлинки-указатели (коммит-хэши, НЕ файлы) - ровно грабля, от которой стерегло старое правило «не делай init». Бэкап вышел бы ложным.
- **Локальный init - это НЕ бэкап.** Тот же диск -> умирает вместе. Нужен приватный remote + `push`, иначе это история/откат, а не защита от смерти диска.

### Что делаем

- [ ] `git init` в `~/seo-projects/` - это **переопределяет** прежнее правило «не делай git init»; правило ПЕРЕПИСЫВАЕМ, не игнорируем.
- [ ] `.gitignore` (3 строки): `/*` + `!/.claude/` + `/.claude/settings.local.json`. Трекается ровно: `.claude/CLAUDE.md` + `.claude/skills/{new-project,guide,status}`. Клиенты и `template-project` - под `/*` (в т.ч. будущие, без ручного добавления).
- [ ] Приватный GitHub-репо + `push`.
- [ ] **Переписать правило в обеих `CLAUDE.md`** (родитель + шаблон, секция «Что НЕ делать»): было «не делай git init в `~/seo-projects/`» -> стало «родитель - git-репо, трекает ТОЛЬКО свой `.claude/` (скилы + `CLAUDE.md`); клиентов и `template-project` НЕ `git add` - они в `.gitignore`». Это БЕЗОПАСНЕЕ прежнего запрета (игнор явный, грабля закрыта).
- [ ] Проверка: `git status` в родителе показывает только `.claude/`-изменения; ни один клиент не виден как кандидат на add.

### Что покрывает и что НЕТ

Бэкапит **только** родительский конфиг (~5 файлов). Клиентов НЕ покрывает (вложенные репо под игнором). «Запушить родительский репо» их не забэкапит - см. связанную задачу. Машинерия клиентов не задета: git всегда берёт ближайший `.git`, родительский для операций внутри клиента невидим.
