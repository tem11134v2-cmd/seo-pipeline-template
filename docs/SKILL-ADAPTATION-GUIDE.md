# Skill Adaptation Guide

> Как переносить старый claude.ai-скил (с фазами и шагами `Шаг X-Y`)
> в архитектуру этого проекта.

Этот документ — инструкция «как думать», не решение про инфраструктуру.
Инфраструктурные решения — в [docs/adr/](adr/). Эталонный пример переноса —
скил `seo-articles` → `.claude/skills/write-article/` + 11 агентов + 4 скрипта.

---

## Зачем это вообще

Старые скилы для claude.ai устроены под платформенные ограничения:
- Нет файловой системы → ручная переноска файлов между чатами
- Лимит длины ответа → текст по частям, команды «Шаг X-Y»
- Деградация качества на длинном контексте → каждая фаза = новый чат

Claude Code снимает все три. Но если механически перенести скил «один в один»
(один старый шаг → один новый субагент, остальное в скил-сценарий) — получится
рабочая, но **глупая** система: дороже по токенам, медленнее, без выигрыша
от новой архитектуры. Часть шагов надо объединить, часть — вынести в скрипты,
часть — превратить в hook.

Гайд описывает, как принимать эти решения.

---

## Главный принцип декомпозиции

Каждый шаг старого скила анализируем по двум осям:

1. **Тип работы:** творческая / детерминированная / проверка
2. **Контекст:** свой уникальный / общий с соседним шагом

Из этих двух осей вытекают четыре типа компонентов:

| Тип шага | Куда переносить |
|---|---|
| Творчество + уникальный контекст | **Субагент** в `.claude/agents/` |
| Детерминированная трансформация | **Скрипт** в `.claude/scripts/` (Node) |
| Механическая проверка / контракт | **Hook** в `.claude/hooks/` |
| Оркестрация серии шагов | **Скил** в `.claude/skills/` |

Дальше — конкретные правила для каждого типа.

---

## Когда шаг → субагент

Шаг становится **отдельным субагентом**, когда выполняется все четыре условия:

- Имеет **чёткий выход** (один или несколько файлов)
- Требует **творчества** (LLM, не функция)
- Имеет **специфичный контекст** (читает свои файлы, использует свои MCP)
- Стартует со **свежими мозгами** имеет ценность (длинный или сложный шаг)

**Примеры из article pipeline:**
- `client-profiler` — исследует сайт, заполняет ЗАКАЗЧИК.md (творческая работа с уникальным MCP — fetch)
- `jm-analyst` — JM-цикл, требует свежей фокусировки и доступа к JM MCP
- `section-writer` — пишет один раздел; **каждый H2 = отдельный вызов** для свежести контекста

---

## Когда шаг → скрипт (НЕ субагент)

Шаг выносим в **Node-скрипт**, когда:

- Это **детерминированная трансформация** (A → B по правилам)
- **Творчества не требуется** — алгоритм описуем функцией
- Хочется **скорости и предсказуемости** (без галлюцинаций)

**Примеры:**
- `assemble-html.mjs` — markdown + HTML-вставки + JSON-LD + template → итоговый HTML
- `to-excel.mjs` — JSON → XLSX
- `tilda-split.mjs` — HTML → HEAD + T123 с применением Тильда-фиксов
- `finalize-setup.mjs` — git init + первый коммит

**Антипаттерн:** делать «html-builder субагента», который собирает HTML по шаблону.
LLM здесь не нужен — это шаблонизация. Трата токенов + риск что модель что-то
«улучшит» в шаблоне.

---

## Когда шаг → hook (НЕ субагент и НЕ скрипт)

Шаг становится **hook**, когда:

- Это **проверка контракта**, не генерация
- **Должна выполняться гарантированно**, а не по доброй воле модели
- Можно описать как **ассерт** (файл существует, заголовок матчит, нет запрещённого паттерна)

**Принцип разделения слоёв:**
- **Творческая самопроверка** = часть промта субагента
  («после раздела убедись, что закрыл боль из ТЗ»)
- **Механическая самопроверка** = hook
  («файл создан, объём в допуске, нет длинного тире»)

Не путать слои. То, что должно выполняться независимо от честности модели,
не должно зависеть от промта. И наоборот — субъективные суждения в hook'е
будут хрупкими.

**Примеры из article pipeline:**
- `check-section.sh` — section-writer создал файл, один H2, нет `—`, объём в норме
- `check-jm-balance.sh` — перед `jm_text_analyze` есть лимиты ≥5
- `pause-for-review.sh` — после article-finalizer останавливает скил и выводит
  сообщение для пользователя

---

## Когда шаги объединять в один субагент

Несколько старых шагов сливаются в **один субагент**, когда:

- Работают **с одним контекстом** (читают те же файлы)
- Это **один тип работы** (одна голова, одна задача)
- Между ними **не нужна точка контроля** для человека
- **Свежий контекст для каждого шага отдельно не даёт выигрыша**

**Примеры объединения из article pipeline:**

| Старые шаги (PHASE-N) | Новый агент | Почему слили |
|---|---|---|
| 3-финал + 3-мета + 3-проверка + 3-сборка | `article-finalizer` | Всё работа со свежесобранной статьёй: заключение, метатеги, отчёт, склейка. Один контекст. |
| 4-1 (HTML-элементы) + 4-3 (FAQ + Schema) | `enhancer` | Всё «генерация HTML-улучшений по тексту». Читает те же файлы (article.md, report.md). |
| 4-4 (AI-детекция) + 4-5 (право + орфо + повествование) | `text-auditor` | Всё «вычитка». Один проход, один отчёт-список проблем. |

**Эвристика:** если для следующего шага не нужно ничего нового — **сливай**.
Если нужно прочитать другой файл / задействовать другой MCP / показать что-то
человеку посередине — **разделяй**.

---

## Когда шаги НЕ объединять — секционная запись

**Противоположный случай — намеренно разделять то, что в claude.ai было одним
«длинным шагом написания».**

Пример: написание статьи. В claude.ai был один длинный процесс «пиши раздел за
разделом, по `Шаг 3-1, 3-2…`». Реализатор может соблазниться сделать один
субагент `writer` и поручить ему всю статью.

**Не делать так.** Каждый раздел = отдельный вызов `section-writer`. Причины:

1. **Свежий контекст под каждый H2.** Не разводнённое внимание моделью
   предыдущих 5000 слов.
2. **Если упадёт — переписываем один раздел, не всю статью.**
3. **Кумулятивный счётчик в файле** (`progress.json`) — точка передачи
   между вызовами без раздувания контекста.

**Эвристика:** если генерация **длинная** и её можно **естественно разрезать
на самостоятельные куски** (раздел, глава, блок) — режь и делай subagent
per кусок. Связность обеспечивай через файлы (счётчик, предыдущие разделы
доступны для чтения).

---

## Точки контроля и паузы для человека

В старом скиле паузы выглядели как «человек проверяет вывод и пишет „ок"».
В новой архитектуре есть два паттерна:

### Пауза «подтверди ОК»

Используется, когда нужно дать человеку посмотреть промежуточный результат
и решить — продолжать или менять.

Механика:
- В скиле после делегации субагента — пауза с явным сообщением в чат
- Если важно подвесить процесс гарантированно — `SubagentStop` hook
  (паттерн `pause-for-review.sh`)

Пример: после `article-finalizer` человек читает статью, и только потом
запускает аудит и упаковку.

### Пауза «сделай что-то снаружи»

Используется, когда нужно действие в реальном мире (сгенерировать фото в
нейросети, проверить рендер в браузере, заполнить файл).

Механика:
- Скил выводит инструкцию и ждёт `/continue` (или другой явный сигнал)
- Подвешивает state в `meta.json` (например, `awaiting-photos`)

Пример: после `photo-promter` человек генерирует фото и заполняет `urls.json`.

## Что НЕ переносить из claude.ai-скилов

Это технические артефакты старой платформы, которые в новой архитектуре
не нужны:

- **«Скачай файл и загрузи в проект».** Файлы автоматически в task-папке.
- **Команды `Шаг X-Y` от пользователя.** Перенесены в делегации субагентов
  изнутри скила-оркестратора. Пользователь не знает про шаги.
- **«Новый чат для каждой фазы».** Одна сессия, разные субагенты со свежим
  контекстом — то же самое, без ручной возни.
- **Дублирование контекста между фазами.** В claude.ai каждый чат заново
  читал шаблоны и контекст. У нас агенты читают из файлов один раз за вызов.
- **«ЗАКАЗЧИК.md прикрепи к проекту».** Лежит в корне, читается всеми.
- **Тонкая ручная маршрутизация по командам.** Заменена на skill-аргументы
  (`--variant`, `--resume`, `--only-A`).

## Per-task vs shared-edit (повторение для контекста)

При переносе нового скила сразу решай — где живут результаты:

- **Артефакты одной задачи** → своя папка задачи
  (например `articles/NNN/`, `audits/NNN/`, `strategies/NNN/`).
  При `/handoff` они автоматически попадут в main.

- **Изменения общих файлов проекта** (ЗАКАЗЧИК.md, template.html,
  topics.xlsx) → через `.claude/handoff-requests/`.
  Применяются в main через `/handoff-process`.

- **Глобальная методология** (стили, жанры, эталонные шаблоны) →
  `~/.claude/seo-knowledge/`. Не дублируется по клиентам.

Подробнее — [ADR-002](adr/002-handoff-split.md), [ADR-004](adr/004-global-mcp-and-knowledge.md).

## State machine для многошаговых скилов

Если новый скил — это длинный процесс с несколькими этапами (как
`write-article`), сразу проектируй его как **state machine**:

1. **`meta.json` в папке задачи** — единственный источник истины
   о текущем состоянии.
2. **Состояния в линию** (`init → step1-done → step2-done → completed`),
   без ветвлений где это возможно.
3. **`update-meta.sh` после каждого этапа** обновляет state атомарно.
4. **`--resume` аргумент** скила — читает `meta.json` и продолжает с
   текущего состояния. Идемпотентность: каждый шаг проверяет, нет ли
   уже его выходного файла; если есть и `--resume` — пропускает.

Это даёт устойчивость к падениям сессии, повторам, и человеческим
«ой, я закрыл вкладку».

## Технические конвенции

Правила оформления и взаимодействия, которым следует **любой** новый компонент. Не методология, а механика — без неё новый скил не сработает с инфраструктурой проекта.

### Модель — всегда `opus`

Все агенты системы используют `model: opus`. Без исключений. Решение принято для:
- предсказуемости стоимости и качества
- устранения развилок «какую модель выбрать для этого агента»
- последовательности — иначе ревью и сравнение скилов становится сложнее

Если для нового агента opus кажется избыточным — это сигнал, что компонент скорее **скрипт** или **hook**, а не агент. Возвращайся к разделам «Когда шаг → скрипт» и «Когда шаг → hook».

### Frontmatter агента — минимально и достаточно

**Шаблон зависит от того, использует ли агент MCP-серверы.**

#### Вариант A: агент использует MCP (fetch, Keyso, Wordstat, JM, Webmaster и т.п.)

```yaml
---
name: <kebab-case-имя, обычно verb-noun>
description: <одна строка, что делает>
model: opus
---
```

**Поле `tools:` отсутствует.** Это критично: если указать `tools:` хоть с одним значением, субагент получает **только** этот список, а MCP-серверы родителя в него не пробрасываются. Агент молча останется без `mcp_fetch_page`, `domain_dashboard`, `jm_text_analyze` и так далее — даже если он их упоминает в теле промта.

Без `tools:` агент наследует **весь** набор тулов родительской сессии (Read/Write/Edit/Grep/Glob + все подключённые MCP).

#### Вариант B: агент чисто файловый (читает / пишет / правит, MCP не зовёт)

```yaml
---
name: <kebab-case-имя>
description: <одна строка>
tools: Read, Write, Edit
model: opus
---
```

Сужение тулов делает контракт агента нагляднее и страхует от случайных побочных действий (нет Bash → агент не сможет запустить процесс, нет WebFetch → не пойдёт в интернет вне MCP).

**Примеры в проекте:** `section-writer`, `article-finalizer`, `enhancer`, `text-auditor`, `photo-promter`, `article-fixer`, `strategy-writer`, `tariff-architect`, `template-designer` — все они только Read/Write.

#### НЕ добавлять (в любом варианте):

- `mcpServers:` — MCP подключены глобально (см. [ADR-004](adr/004-global-mcp-and-knowledge.md)). Указывать в агенте бесполезно и сбивает с толку.
- `hooks:` во frontmatter — все хуки живут в проектном `.claude/settings.json`.

#### Как решить, какой вариант твой

Перед написанием агента ответь: «вызывает ли он хоть один тул вида `mcp_*`, `wm_*`, `ym_*`, `jm_*`, `domain_*`, `keyword_*`, `arsenkin_*`, `speedyindex_*`, `wk_*`?» Если да — вариант A (без `tools:`). Если нет — вариант B (явный `tools: Read, Write[, Edit]`).

### `current-task.txt` — обязателен первым

**Любой worktree-скил** в самом начале (до первой записи или коммита) пишет:

```bash
echo "<path/to/task-dir>" > .claude/tmp/current-task.txt
```

Где `<path/to/task-dir>` — папка задачи (`articles/NNN-slug/`, `audits/NNN-slug/` и т.п.), **или** `.claude/handoff-requests/setup` для скилов вроде setup-project, которые пишут только в handoff-requests.

Без этого файла pre-commit hook **откажет в любом коммите** в worktree с сообщением «worktree-сессия не объявила текущую задачу» (см. [ADR-003](adr/003-pre-commit-whitelist.md)). Это не баг — защита от случайных коммитов вне зоны задачи.

### Маркеры expected-файлов — для проверки выхода агента

Перед делегацией субагента **скил пишет маркер** с ожидаемым путём выходного файла:

```bash
echo "<path/to/expected-output>" > .claude/tmp/expected-<agent-name>-<run_id>.txt
```

Хук `check-file.sh` (SubagentStop matcher `*`) после завершения агента читает маркер и проверяет, создан ли файл и непустой ли он. **Без маркера хук молча пропускает проверку** — агент может «успешно завершиться», ничего не написав, и это никто не заметит.

Маркер пишется **до** делегации. Удаление после успешной проверки — опционально, для чистоты `.claude/tmp/`.

### Агенты общаются ТОЛЬКО через файлы

Между скилом и агентом, и между агентами — обмен идёт **через файлы в папке задачи**, не через текст в чате.

- Делегирующий промт от скила к агенту содержит **пути** (`article_dir`, `tz_path`, и т.п.)
- Агент **читает** из этих путей
- Агент **пишет** результат в пути, заявленные в его контракте
- Возврат в чат — только короткая сводка для человека (или кумулятивный счётчик, как у `section-writer`)

Не передавать большие данные через промт — это раздувает контекст, удорожает вызов и теряется при следующей делегации.

### Naming conventions

| Тип | Шаблон | Примеры |
|---|---|---|
| Агент (файл) | `<verb>-<noun>.md` | `section-writer.md`, `tz-builder.md`, `article-finalizer.md` |
| Скрипт | `<verb>-<noun>.mjs` | `assemble-html.mjs`, `tilda-split.mjs` |
| Hook | `<verb>-<target>.sh` | `check-section.sh`, `pause-for-review.sh`, `check-jm-balance.sh` |
| Скил (папка) | `<verb>-<noun>/SKILL.md` | `write-article/`, `fix-article/`, `setup-project/` |
| Папка задачи | `<plural-noun>/<NNN>-<slug>/` | `articles/001-teploizolyatsiya/`, `audits/003-vasya-tech/` |

Кириллицу в именах не использовать — git, скрипты, хуки её не любят на разных платформах.

### Аргументы скилов

Когда новый скил принимает параметры, следовать паттерну существующих:

- **Позиционный обязательный** идёт первым: `/write-article <N>`, `/fix-article <NNN> "<правка>"`
- **Флаги через `--`**: `--resume`, `--only-A`, `--only-B`, `--both`, `--dry-run`, `--message "..."`
- **`--resume` — стандартный паттерн** для скилов с state machine: читает `meta.json`, спрашивает «продолжить с `<state>`? [Y/n]», стартует с правильного шага
- **Если параметр требует обсуждения с пользователем** — лучше не передавать в команду, а спросить в чате внутри скила

### Контракт «вход → выход» агента

Каждый агент в своём промте должен явно перечислить:

1. **Вход** — что передаётся в делегирующем промте (`article_dir`, `variant`, ...)
2. **Обязательное чтение** — какие файлы агент читает перед работой (ЗАКАЗЧИК.md, ТЗ, STYLE.md, ...)
3. **Выход** — какие файлы создаёт/обновляет
4. **Запреты** — что агент НЕ делает (не редактирует чужие файлы, не использует длинное тире, не дублирует упоминания бренда, ...)

Это не «методическая рекомендация», это контракт. По нему другие компоненты системы понимают, что от агента ждать.

### Известные особенности MCP-серверов

Здесь копится коллективная память — то, что мы один раз обожглись и не хотим повторять. При написании нового агента под нужный MCP — глянь сюда.

#### Keyso (`domain_dashboard`, `domain_competitors`, `keyword_info`, `domain_pages`, ...)

- **IDN-домены передавай в кириллице, не в Punycode.** Для `ремонт-квартир-днр.рф` нужен именно `ремонт-квартир-днр.рф`, а **не** `xn--80aeklehhe9ape7g.xn--p1ai`. Punycode возвращает «домен не найден» или нулевые метрики. Это касается и домена клиента, и доменов конкурентов в `domain_dashboard`.
- **Параметр `base` обязателен** (`msk`, `spb`, `ekb`, ...). По умолчанию `msk` — она покрывает все регионы и обновляется чаще. Региональную базу подключай если нужна локальная специфика.
- **`include_history=true` для клиента**, но **не** для каждого конкурента — иначе раздувается контекст и тратится квота.
- Город не в списке поддерживаемых баз → используй `msk` и зафиксируй в выводе «город X не в базе Keyso».

#### Wordstat (`mcp_wordstat_get_keyword_stats`, `mcp_wordstat_get_dynamics`)

- **`region_id` — региональный код Яндекса** (Москва: 213, СПб: 2, Екб: 54, ...), **не** Keyso-base. Полный список — в SKILL.md скила /strategy.
- Без `region_id` — общероссийская частотность; для регионального бизнеса это вводит в заблуждение.

#### JM (`jm_text_generate`, `jm_text_analyze`, ...)

- Перед `jm_text_analyze` проверь баланс через `jm_account` (хук `check-jm-balance.sh` это делает на уровне PreToolUse, но в больших циклах лучше следить и в промте).
- `jm_text_generate` и `jm_text_analyze` дорогие — кешируй результаты в `<task_dir>/jm/*.json` и читай оттуда, а не пересчитывай.

Если ты обнаружил новую особенность какого-то MCP — добавь сюда. Это самая дешёвая страховка для следующего адаптера.

## Алгоритм для новой сессии-адаптации

Когда открываешь новую сессию, чтобы перенести очередной claude.ai-скил,
действуй по шагам:

### 1. Понять источник

- Распаковать `.skill` (это zip)
- Прочитать **все** PHASE-файлы и references
- Понять домен: что делает скил, какие у него входы, выходы, MCP

### 2. Составить таблицу маппинга

Для каждого старого шага решить:
- **Субагент** (творчество, уникальный контекст) — имя, что читает, что пишет
- **Скрипт** (детерминированная трансформация) — имя, вход, выход
- **Hook** (механическая проверка) — когда срабатывает, что проверяет
- **Объединение** с другим шагом (если применимо) — какие шаги, в какого
  агента, почему можно слить

Показать таблицу пользователю **до начала реализации**. Ждать ОК.

### 3. Решить структуру папки задачи

Какое имя? `articles/`, `audits/`, `strategies/`, `commerce/`?
Какие промежуточные подпапки? Какой формат `meta.json`?
Какие финальные артефакты?

### 4. Решить per-task vs shared-edit

Скил создаёт **новые** общие файлы (как `setup-project`) или
**меняет существующие** (как добавление темы в `topics.xlsx`)?
→ Нужно ли расширять `handoff-process` под новый тип запросов?

### 5. Реализовать

При создании каждого компонента — следовать **«Технические конвенции»** (раздел выше): frontmatter с `model: opus`, без `mcpServers`, маркеры expected-файлов, current-task.txt первым, naming, контракты.

По стандартному паттерну:
- Создать агентов в `.claude/agents/`
- Создать скил в `.claude/skills/<name>/SKILL.md`
- Создать скрипты в `.claude/scripts/` (если нужны)
- Создать хуки в `.claude/hooks/` (если нужны)
- Обновить `settings.json` под новые хуки
- Обновить **README** (таблица скилов, таблица агентов)
- При архитектурно-новом решении — создать **ADR** в `docs/adr/`

### 6. Тест на холостом ходу

- Пройти полный цикл на простом примере
- Проверить: pre-commit hook не отказывает, `/handoff` работает,
  resume работает

### 7. Push

После прохождения теста — коммит и push.

## Эталонный пример: article pipeline

Маппинг старого скила `seo-articles` на новые компоненты. Используй как
референс для своего переноса.

### Структура старого скила

```
SKILL.md
INSTRUCTIONS.md
references/
  PHASE-0.md     — настройка проекта (4 шага: 0-1, 0-2, 0-3, 0-финал)
  PHASE-1.md     — генерация тем (1 шаг: 1-1)
  PHASE-2.md     — ТЗ (3 шага: 2-1, 2-2, 2-3)
  PHASE-3.md     — написание (6 шагов: 3-0, 3-N, 3-финал, 3-проверка, 3-мета, 3-сборка)
  PHASE-4.md     — упаковка (8 шагов: 4-1...4-8)
  STYLE.md       — правила стиля (методология)
  GENRES.md      — жанровые модули (методология)
  ENHANCEMENTS.md — HTML-элементы (методология)
  SVG-ICONS.md   — набор иконок (методология)
assets/
  CLIENT-TEMPLATE.md
  TEMPLATE.html
```

### Маппинг

| Старый шаг | Новый компонент | Тип | Что менялось |
|---|---|---|---|
| 0-1 (заполнение ЗАКАЗЧИК.md) | `client-profiler` | агент | — |
| 0-2 (шаблон вёрстки) | `template-designer` | агент | — |
| 0-3 (внешние площадки) | часть `client-profiler` | агент | слили в 0-1 (общий контекст) |
| 0-финал | `scripts/finalize-setup.mjs` | скрипт | вынесли (детерминированно) |
| 1-1 (Excel с темами) | `topic-generator` + `to-excel.mjs` | агент + скрипт | разделили: агент собирает данные, скрипт делает xlsx |
| 2-1 (JM-анализ) | `jm-analyst` | агент | — |
| 2-2 / 2-3 (ТЗ A / B) | `tz-builder` | агент | вызывается 1 или 2 раза с `variant=A|B` |
| 3-0 (инициализация) | логика в `/write-article` | skill | не агент — это просто чтение ТЗ + создание progress.json |
| 3-N (каждый раздел) | `section-writer` | агент | **отдельный вызов на каждый H2** (свежий контекст) |
| 3-финал + 3-мета + 3-проверка + 3-сборка | `article-finalizer` | агент | **слили в один** (общий контекст финализации) |
| 4-1 + 4-3 | `enhancer` | агент | **слили** (оба — HTML-улучшения) |
| 4-2 (фото) | `photo-promter` | агент | отдельно — есть внешний шаг (человек генерит фото) |
| 4-4 + 4-5 | `text-auditor` | агент | **слили** (всё — вычитка) |
| 4-6 (сборка HTML) | `scripts/assemble-html.mjs` | скрипт | вынесли (шаблонизация без LLM) |
| 4-7 (Тильда) | `scripts/tilda-split.mjs` | скрипт | вынесли (детерминированные правила) |
| 4-8 (правки) | `article-fixer` | агент | отдельно — многократный запуск с разными правками |

**Методологические файлы** (STYLE.md, GENRES.md, ENHANCEMENTS.md, SVG-ICONS.md,
TEMPLATE-MASTER.html, CLIENT-TEMPLATE.md) — в `~/.claude/seo-knowledge/`,
без проектных копий ([ADR-004](adr/004-global-mcp-and-knowledge.md)).

### Что было ДОБАВЛЕНО относительно старого скила (это нормально)

- `meta.json` + state machine + `--resume`
- `progress.json` (кумулятивный счётчик)
- pre-commit hook (защита от случайных правок общих файлов)
- handoff / handoff-process (worktree → main механика)

Эти вещи в старом скиле отсутствовали как класс — там не было файловой
системы и многозадачности. Новые скилы получают их «бесплатно» — потому
что инфраструктура уже стоит.

## Когда отступать от гайда

Гайд описывает паттерны, а не закон. Отступай, если:

- Новый домен **категорически не ложится** в существующие паттерны
  (например, нужна параллельная асинхронная работа — это новая
  архитектурная задача, не «новый скил»)
- Применение паттерна делает реализацию **запутаннее**, не проще
  (бывает на простых скилах из 1-2 шагов — необязательно тянуть
  state machine ради них)

При отступлении — **создавай ADR**. Решение должно быть зафиксировано,
не уходить в устную традицию.
