### Контекст оркестратора /seo-statya (причина «80% после 5 статей»)
1. SKILL.md = 64 410 байт (~18-20k токенов) грузится сразу.
2. Шаг 9b: оркестратор САМ генерит и публикует фото - 8-14 Skill-вызовов (image-generation + image-publishing на каждое из 4-7 фото) на КАЖДУЮ статью. Самый жирный пожиратель.
3. Цикл секций: 12-20 строк счётчиков × 6-10 секций в чат.
4. Parent-fallback финализатора: при сбое агент возвращает ПОЛНЫЙ report.md/metatags.json в чат.
5. Ретраи (по 2 на each verify-гейт) дублируют возвраты.
6. Серийный режим = всё выше × N статей, без какого-либо сброса. Resume есть (3 уровня, включая авто-resume по 529/503 через ScheduleWakeup) - это спасательный круг, но каждая новая сессия заново платит 64 КБ SKILL.md.

## Процесс одного этапа (стандарт)
1. Стратег (главная сессия) пишет спеку этапа: точные файлы, правки, критерии приёмки.
2. Ветка stage-N в template-project.
3. Имплементация субагентами: механические правки - sonnet, промптовые/архитектурные - opus; параллельно по непересекающимся файлам.
4. Независимая ревизия: opus-агент, вход - спека + дифф; ищет пропуски, битые кросс-ссылки (промпт ↔ скрипт ↔ SKILL), нарушения конвенций.
5. Тесты: .claude/tests/*/run.mjs (все) + новые тесты этапа.
6. Смоук: прогон затронутого скила на тестовом клоне.
7. Коммит (версия машинерии = хэш коммита, /status покажет отставание).
8. Раскатка: /sync-all --apply --only <пилоты> → наблюдение → все проекты на следующем этапе.
Правило трёх слоёв: каждое новое правило = промпт + машинный энфорс (скрипт/хук) + тест. Документация (guide/README) правится в том же этапе.
