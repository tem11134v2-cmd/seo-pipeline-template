# ADR-016: Новый тип задачи `faq/` для SEO-нормализации текста (/seo-faq)

**Статус:** Принято

**Дата:** 2026-06-06

## Контекст

Портируем авторскую фазу **У6-Ф2** (SEO-слой) из claude.ai-скила `seo-copywriting`: к готовой коммерческой странице добавить **отдельный SEO-блок** (FAQ + возражения + плитка тегов + перелинковка), который **нормализует текстовую оптимизацию** - добирает недостающие по семантике ключи/N-граммы под Яндекс. В оригинале это будущая фаза после У6-Ф1 (конверсия). У нас конверсия уже вынесена в `/seo-tekst` (ADR-015).

Развилки:
1. **Отдельный скил или часть /seo-tekst?** SEO-слой логически идёт после текстов, но нужен и на ЧУЖИХ живых страницах (не только свежесгенерённых).
2. **Куда артефакты?** Ещё один тип задачи.
3. **Чем «нормализовать»?** Нужен инструмент анализа пробелов текста.

## Решение

### 1. Отдельный скил `/seo-faq` + тип задачи `faq/NNN-<slug>/`

`/seo-faq` самостоятелен: источник - тексты `/seo-tekst` (`--from-tekst NNN`), готовая таблица, или живой URL. Работает и на страницах, которые не мы писали. Артефакты в `faq/NNN-<slug>/` (per-page `pages/<slug>/{faq_blocks.json, faq.html, faq.md}` + клиентский `FAQ_<slug>.docx`). **Шестое повторное применение паттерна** ADR-007. Pre-commit универсален; `current-task.txt = faq/NNN-slug/`.

### 2. JM-анализ пробелов как ядро ценности

`faq-builder` (агент, веер по страницам): `jm_text_analyze` по текущему тексту + целевым запросам -> чего не хватает относительно топ-10; затем пишет FAQ/возражения/плитку/перелинковку, **естественно** вшивая недостающие ключи/N-граммы; контроль переспама `jm_text_check` (Акварель). Балансовый гейт `jm_account >= 5` в скиле. Это не «вода ради объёма», а адресная нормализация семантики.

### 3. Schema.org + рендер - скриптом; pt-классы общие с китом

`build-faq.mjs` детерминированно собирает `faq.html` (аккордеон `<details>` + **Schema.org FAQPage JSON-LD** + плитка тегов + перелинковка) и `faq.md`. Классы `pt-` совпадают с китом `/seo-tekst` - блок ложится и в наш прототип, и в любую страницу клиента. `verify-faq.mjs` (валидность Schema, объёмы FAQ, стоп-формулы, тире, наличие `normalized_keywords`) - **скрипт, не hook** (веер + конвенция ADR-012). `settings.json` не меняется.

### 4. Общая методология читается из `seo-tekst/assets`

`faq-builder` читает блоки 33-39 из `BLOCKS.md` и стоп-таблицы из `COPY.md`, лежащих в `.claude/skills/seo-tekst/assets/` (кросс-скил read-only зависимость, заявленная в ADR-015 п.6). Дублировать срез не стали.

### 5. Drive - переиспользуем `texts_folder_id`

`FAQ_<slug>.docx` грузится в ту же папку «Тексты» (`texts_folder_id`), что и Analysis/Texts - FAQ часть текстовой выдачи. Нового Drive-ключа не заводим.

## Альтернативы
- **A. Фаза внутри /seo-tekst** (как Ф2 после Ф1). Отвергнуто: `/seo-faq` нужен и на чужих живых страницах независимо; раздувал бы `/seo-tekst`.
- **B. Отдельный `faq_folder_id`.** Отвергнуто: FAQ - текстовый деливерабл, логично в «Тексты».
- **C. Schema.org генерит агент.** Отвергнуто: микроразметка детерминирована -> скрипт (как в `enhancer`/`assemble-html` для статей).
- **D. Дублировать блоки 33-39 в seo-faq.** Отвергнуто: читаем из общего kit (ADR-015 уже допускает кросс-скил read).

## Последствия

**Хорошо:**
- Встраивается без изменений pre-commit/handoff/`settings.json`. Веер по страницам (6-8).
- Чистое разделение: `/seo-tekst` - конверсия (люди), `/seo-faq` - SEO (поиск). Каждый запускается отдельно.
- FAQ-блок переносим: один `faq.html` ложится и в прототип, и в живой сайт; Schema.org FAQPage даёт микроразметку под Яндекс.
- JM-нормализация адресная (по пробелам), а не «налить воды».

**Плохо:**
- Зависимость `/seo-faq` -> `seo-tekst/assets` (общий kit). Если seo-tekst переименуют/перенесут - поправить пути. Митигация: документировано здесь и в ADR-015.
- JM дорогой; на 10 страницах веер съест баланс. Митигация: балансовый гейт + кеш результатов в задаче.
- Качество нормализации зависит от наличия целевых запросов на входе (из текстов/таблицы). Без запросов анализ слабее - скрипт предупреждает.
