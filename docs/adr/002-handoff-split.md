# ADR-002: `/handoff` делает merge, `/handoff-process` только apply

**Статус:** Принято

**Дата:** 2026-05-25

## Контекст

В рамках [worktree-first архитектуры](001-worktree-first.md) нужно две операции:

1. **Завершить работу в worktree** — финализировать коммит, перенести результаты задачи в основную папку.
2. **Применить запросы на общие файлы** — взять накопленные handoff-requests (например «создай ЗАКАЗЧИК.md», «добавь 25 тем в xlsx»), реально модифицировать общие файлы проекта, разрешить конфликты.

Эти две операции могут жить в одной команде или в двух разных.

Промежуточный вариант (короткое время в проекте): `/handoff` делал и merge, и apply, плюс был `/process-handoffs` который дублировал apply отдельно. Получалось рассинхронизированно.

## Решение

**`/handoff`** (запускается **внутри worktree-сессии**):
1. Финальный коммит остатков.
2. `git -C <main> merge <self-branch>` — переносит файлы worktree (включая `.claude/handoff-requests/`) в main.
3. Удаляет ветку.
4. Пытается удалить worktree (с fallback на prune при file lock).

**`/handoff-process`** (запускается **в main-сессии** без worktree):
1. Читает `.claude/handoff-requests/` (которые уже физически в main после merge).
2. Анализирует каждый запрос (setup-project, new-topics, shared-edit).
3. Применяет к общим файлам — интеллектуально, при противоречиях между несколькими запросами объединяет или спрашивает пользователя.
4. Переносит обработанные запросы в `.claude/handoff-requests/processed/`.
5. Делает один обзорный коммит.

## Альтернативы

### A. Всё в `/handoff`

Worktree-сессия делает merge **и** apply за один вызов. Отвергнуто: apply требует **интеллектуального разрешения конфликтов** между параллельными worktree (две сессии добавили темы, обе пишут в xlsx — нужно дедуплицировать). В момент `/handoff` мы не видим другие несмерженные worktree, не можем сравнить.

### B. Всё в `/handoff-process`

Worktree-сессия только коммитит, merge делает main. Это **близко** к выбранному варианту, но тогда:
- Каждая worktree-сессия оставляет за собой «зомби»-ветку и worktree-папку до тех пор, пока пользователь не запустит `/handoff-process` в main.
- Если пользователь забыл `/handoff-process` — десятки несмерженных веток, путаница.

В выбранном варианте `/handoff` хотя бы вычищает за собой worktree сразу — если apply не нужен (per-task задача без handoff-requests), на этом всё заканчивается.

### C. Один универсальный скил, определяющий контекст автоматически

Запускаешь `/handoff`, он сам понимает — в main мы или в worktree, и делает нужное. Отвергнуто: магическое поведение, сложно отлаживать, пользователь не понимает что именно произошло.

## Последствия

**Хорошо:**
- Чёткое разделение: «закончил задачу» → `/handoff` в worktree. «Применить накопленное» → `/handoff-process` в main.
- `/handoff-process` обрабатывает **все** запросы со всех завершённых worktree сразу, что позволяет интеллектуально разрешать конфликты между ними.
- Per-task задачи (write-article без правок общих файлов) обходятся одним `/handoff`, второй шаг не нужен.

**Плохо:**
- Два шага вместо одного, когда есть handoff-requests.
- Пользователь должен помнить открыть main-сессию **отдельно** (без галочки worktree).
- Если несколько worktree сделали `/handoff` параллельно — main-репо может уйти в состояние, в котором последний merge столкнулся с конкурирующим. Это редкий случай (race condition между двумя merge в main), но возможный. Решение — выполнять `/handoff` последовательно, что естественно в одно-пользовательской системе.
