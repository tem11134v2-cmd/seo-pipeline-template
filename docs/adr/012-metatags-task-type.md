# ADR-012: Новый тип задачи `metatags/` для генерации метатегов (H1, Title, Description)

**Статус:** Принято

**Дата:** 2026-05-31

## Контекст

При портировании claude.ai-скила `seo-metatags` (авторская услуга У4: генерация H1/Title/Description под Яндекс) возникли развилки:

1. **Два режима исходника.** У4-стд (пакетный, дёшево, по правилам, ~5 MCP на весь пакет) и У4-топ (поштучный, дорого, 5 шагов START→STEP_LAST с анализом выдачи и «Акварелью» = `jm_text_check`). В claude.ai они были разными промтами с ручными командами «далее»/«шаг N» - техдолг платформы (лимит длины ответа, нет файловой системы, деградация на длинном контексте).

2. **Где живут артефакты?** Уже есть `articles/`, `strategies/`, `analyses/`, `structures/`. Метатеги - ещё один тип задачи (один комплект A7 на проект).

3. **Как стыковаться с `/seo-structure`?** Метатеги логически идут сразу после утверждения структуры клиентом (A6). Нужна и автоматическая сцепка, и самостоятельный запуск.

4. **Параллелизм.** В claude.ai У4-топ обрабатывал по одной странице за чат вручную. В Claude Code субагенту можно дать страницу, а писателей запустить веером.

## Решение

### 1. Новый тип задачи - `metatags/NNN-<slug>/`

Каждый запуск создаёт `metatags/NNN-<slug>/` рядом с `articles/`, `strategies/`, `analyses/`, `structures/`. Внутри - `meta.json` (state machine), промежуточные JSON (`inputs.json`, `pages.json`, опц. `audit.json`, `research.json`, `shortlist.json`, `pages/<n>.json`), финальный `A7_<slug>.xlsx`, `share.json`.

**Артефакты метатегов ВСЕГДА в `metatags/NNN/`** - и при самостоятельном запуске, и когда движок вызывается хвостом из `/seo-structure` (см. п.4). Никаких «или в структуру, или в метатеги в зависимости от старта» - всегда единообразно `metatags/`. `NNN` зеркалит номер источника, если он есть (структура 003 → метатеги 003).

Pre-commit hook менять не нужно - он универсален к task-dir (ADR-003). Скил пишет `.claude/tmp/current-task.txt = metatags/NNN-slug/` первым делом.

Это **четвёртое повторное применение паттерна** ADR-007 (после ADR-009 `analyses/`, ADR-010 `structures/`). Тривиально, новых архитектурных понятий не вводит.

### 2. Один движок вместо двух режимов, глубина - флагом `--depth deep|bulk`

Вместо двух непересекающихся промтов - **один пайплайн** с переключателем глубины:

```
Фаза 0 (опц., режим «сайт»)  site-scanner          -> audit.json  (последовательно, 1 агент)
Фаза 1                        metatag-researcher    -> research.json (последовательно, 1 агент, батч-MCP)
Фаза 2                        select-variations.mjs -> shortlist.json (скрипт)
Фаза 3                        metatag-writer        -> pages/<n>.json (ПАРАЛЛЕЛЬНО в deep, чанками в bulk)
Фаза 4                        build-metatags-xlsx   -> A7_<slug>.xlsx (скрипт)
```

- `--depth deep` (по умолчанию) = качество «топ»: per-page анализ выдачи (`arsenkin_top` + `arsenkin_parse`) + Title через Акварель (`jm_text_check`). Писатели запускаются **веером** (1 вызов = 1 страница, свежий контекст) - это паттерн `section-writer`.
- `--depth bulk` = дёшево «стд»: генерация по PLAYBOOK + батч-данные исследователя, без выдачи и Акварели. Писатель обрабатывает страницы чанками.

Дорогую частотность/коммерциализацию (`wk_check_frequency` x3, `arsenkin_commerce`, `jm_suggest`) батчим **один раз на весь проект** в Фазе 1 (а не N×4 вызова как был бы наивный перенос). Это выигрыш и в скорости, и в стоимости относительно оригинала. Оба ценника У4 (5к/10к) сохраняются - из одного кода.

**Глубину НЕ нужно запоминать пользователю.** Скил сам ставит `bulk`, если в сообщении есть намерение «быстро/дёшево/пакетно». Иначе `deep`.

### 3. Верификация выхода - скриптом `verify-metatags.mjs`, не hook'ом

Механические проверки (Title≤60, Description≤160, точное вхождение маркера в H1/Title, отсутствие длинного тире, запрещённые формулировки) - в **скрипт** `verify-metatags.mjs` (exit 0/2 + список нарушений), который скил гоняет после Фазы 3 по всем `pages/<n>.json`.

Причина (отступление от «механическая проверка → hook» в SKILL-ADAPTATION-GUIDE): hook-механизм проекта (`check-section.sh` / `check-file.sh`) основан на **маркере одного ожидаемого файла** и срабатывает на `SubagentStop` каждого субагента. Это **несовместимо с параллельным веером** писателей: при завершении писателя #1 хук проверил бы маркеры ещё-не-готовых страниц #2..#N и дал бы ложный отказ. Скрипт-верификатор после всей пачки этой проблемы лишён и **полностью совпадает с существующей конвенцией** проекта (`verify-markers.mjs`, `verify-progress.mjs` - тоже скрипты, не хуки). Следствие: **`settings.json` не меняется**, новый hook не нужен.

Параллельные писатели поэтому **не пишут expected-маркеры**; полноту обеспечивает `verify-metatags.mjs` (нет файла страницы → скил пере-делегирует только её, до 2 попыток). Последовательные одиночные агенты (`site-scanner`, `metatag-researcher`) маркеры пишут штатно - их проверяет общий `check-file.sh`.

### 4. Стыковка с `/seo-structure` - аддитивный хвост

`/seo-structure` получает флаг `--metatags deep|bulk|none` и аддитивный хвост (шаг 11) после `completed`:

- Порядок коммитов (важно из-за pre-commit): структура сначала доходит до `completed` и **коммитится** под `current-task.txt = structures/NNN`. В ту же запись `meta.json` пишется `metatags_pending = <deep|bulk|none>`. Только после этого хвост переставляет `current-task.txt = metatags/NNN-slug/` и работает там. Re-коммитить `structures/NNN` после переключения pre-commit запретил бы - поэтому состояние хвоста живёт в `metatags/NNN/meta.json`, а флаг-триггер `metatags_pending` - в структуре (для возобновляемости). Раньше план предполагал состояние `metatags-done` на структуре - отказались ровно из-за этого порядка коммитов.
- Хвост по «да»-страницам (`structure_data.json` где `target_status=="yes"`) собирает A7 в `metatags/NNN/`. В `--auto` запускается автоматически (с анонсом-оценкой стоимости), в `--review` - сначала спрашивает. `/seo-structure NNN --resume` при `completed` + `metatags_pending` дёргает хвост (возобновляемость).
- Скрипты структуры (`select-top10.mjs`, `build-structure-xlsx.mjs`, `import-structure.mjs`) **не трогаются** - правка чисто аддитивная (риск регресса минимален).
- `--metatags none` - выключить хвост; по умолчанию `deep`.

Тот же движок (агенты + скрипты, параметризованные рабочей папкой) обслуживает обе точки входа: самостоятельный `/seo-metatags` и хвост `/seo-structure`. Дублирования логики нет.

### 5. Источник страниц - вопрос пользователю, не флаги

При самостоятельном `/seo-metatags` без явного источника скил **спрашивает** (плоский вопрос в чат, как `/seo-analysis` про бриф):

1. **Сканировать сайт** (дай домен) - `site-scanner` соберёт sitemap, текущие H1/Title/Description, оценит приоритет «что править». Режим аудита.
2. **Таблицей** (URL / Тип / Маркер [/ запросы]) - готовый список.
3. **Из структуры** (укажи NNN) - читает `structures/NNN/`.

### 6. PLAYBOOK - в скиле, не в глобальной методологии

Правила H1/Title/Description по типам страниц, запрещённые формулировки, лимиты длины, пороги Comm/Geo и Акварели - в `.claude/skills/seo-metatags/PLAYBOOK.md` (рядом со скилом, коммитится в репо, self-contained), а не в `~/.claude/seo-knowledge/`. Причины: (1) CLAUDE.md просит не трогать глобальную папку из worktree, (2) PLAYBOOK специфичен для этого скила (в отличие от STYLE/GENRES/TARIFFS, общих для многих). Если позже понадобится в нескольких скилах - «повысим» в глобал отдельным решением.

## Альтернативы

### A. Сохранить два режима как два скила (`/seo-metatags-std`, `/seo-metatags-top`)
Отвергнуто: дублирование батч-слоя, два входа для пользователя, пользователь должен помнить какой режим. Один движок + NL-переключатель глубины проще и дешевле.

### B. Писать метатеги в `structures/NNN/` при сцепке
Отвергнуто пользователем явно: метатеги ВСЕГДА в `metatags/` независимо от точки старта (единообразие). См. п.1.

### C. Verify через hook (как `check-section.sh`)
Отвергнуто: hook-маркер на один файл ломается на параллельном веере писателей (ложные отказы). Скрипт-верификатор после пачки - надёжнее и совпадает с `verify-markers.mjs`/`verify-progress.mjs`. См. п.3.

### D. Один агент на всю задачу (`metatag-builder`)
Отвергнуто: раздувание контекста, теряется свежесть per-page анализа выдачи. Нарушает «когда нужен другой фокус - разделяй» (SKILL-ADAPTATION-GUIDE). Researcher (батч, общий контекст) и writer (per-page, свежий) - разные фокусы.

### E. Авто-сбор страниц с сайта как часть оркестратора
Отвергнуто: оркестратор не плодит логику разведки. `site-scanner` - отдельный последовательный агент (Фаза 0), до параллельных писателей.

## Последствия

**Хорошо:**

- `/seo-metatags` встраивается без изменений pre-commit hook, `/handoff`, `/handoff-process`, **`settings.json`** (verify - скрипт, не hook). Подтверждает многозадачность ADR-001.
- Один движок дешевле наивного переноса: дорогие MCP батчатся один раз; per-page работа параллелится.
- Автосцепка структура→метатеги даёт «обновлённую таблицу» (A7 зеркалит A6 строка-в-строку) без ручного перезапуска.
- A7 в 3 листа («Метатеги» для разработчика, «Аналитика» для SEO, «Сводка») - разнесены клиентское и внутреннее, как в A6.

**Плохо:**

- Жёсткой зависимости от структуры/анализа НЕТ (можно дать таблицу или просканировать сайт) - это плюс по гибкости, но `--from-structure` зависит от корректности `structure_data.json`/`top10.json`. Митигация: `read-metatags-input.mjs` валидирует вход и падает с понятным сообщением.
- Режим аудита (`site-scanner`) - самая хрупкая часть (разные форматы sitemap, JS-рендер, потолки на объём). Митигация: вынесен последним этапом, потолки настраиваемы, при пустом `<title>` помечает «не удалось прочитать» вместо тихого пропуска.
- Drive-якорь: в DRIVE.md пока НЕТ ключа `metatags_folder_id` (есть отдельные для structures/analyses/articles/topics/strategies). Нужно завести вручную (папка «Метатеги», anyone-with-link -> reader); до этого `/share-metatags` и шаг 8 скипают с подсказкой.
- Отступление от «verify -> hook» задокументировано здесь (п.3), не уходит в устную традицию.
