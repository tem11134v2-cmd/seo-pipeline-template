# ADR-003: Защита общих файлов через pre-commit hook с белым списком

**Статус:** Принято

**Дата:** 2026-05-24

## Контекст

Worktree-first архитектура (см. [ADR-001](001-worktree-first.md)) держится на одном правиле: «worktree трогает только свою задачу». Если правило соблюдается — конфликты при merge невозможны (math: разные части дерева). Если нарушается — каша.

Правило в CLAUDE.md — это **рекомендация**, не запрет. Claude может его нарушить:
- По невнимательности (увидел опечатку в ЗАКАЗЧИК.md и поправил)
- По устаревшей памяти от предыдущих сессий
- По плохо сформулированному пользовательскому запросу («поправь это везде»)

Нужен **технический барьер**, который сделает нарушение физически невозможным.

## Решение

**Pre-commit git hook** с белым списком путей.

`.claude/git-hooks/pre-commit` при каждом коммите проверяет:
1. В какой worktree мы — основной или дополнительной? (через сравнение `git rev-parse --git-dir` и `--git-common-dir`)
2. Если в **main worktree** — пропускает всё, ограничений нет.
3. Если в **дополнительной worktree** — читает `.claude/tmp/current-task.txt` (там лежит путь к папке задачи) и сравнивает изменённые файлы с белым списком:
   - `<task-dir>/*` — своя задача
   - `.claude/tmp/*` — служебные файлы сессии
   - `.claude/handoff-requests/*` — запросы для main
4. Любой файл вне белого списка → `exit 1` с подробным сообщением.

В `setup-project` (мини-промт установки) выполняется `git config core.hooksPath .claude/git-hooks` — это включает hook для **всех будущих worktree** одного репо.

## Альтернативы

### A. Только инструкция в CLAUDE.md

Сказать Claude «не правь общие файлы». Без технического барьера. Отвергнуто: модель ошибается, особенно при сложных запросах, тогда правило ничего не стоит.

### B. PostToolUse-хук Claude Code

Хук срабатывает после каждого Edit/Write, проверяет путь, в случае нарушения сигналит. Преимущество: реагирует в момент действия, не на коммите. Недостаток: технически сложнее настроить — нужен JSON-парсинг payload, разная структура у разных версий Claude Code. Pre-commit git hook — стандартный механизм git, работает одинаково везде. Отвергнуто как первичная защита, может быть добавлен как «второй уровень» в будущем.

### C. Sparse checkout

Делать worktree, в которой общие файлы физически отсутствуют (sparse checkout). Отвергнуто: общие файлы **нужно читать** (ЗАКАЗЧИК.md для контекста), а sparse скрывает их даже от чтения.

### D. Read-only attribute на общих файлах

`chmod -w ЗАКАЗЧИК.md` в worktree. Отвергнуто: на Windows работает плохо, плюс git сам не понимает читаемость, плюс не блокирует «удалить и пересоздать».

## Последствия

**Хорошо:**
- Жёсткая защита, **математически** исключающая ошибку.
- Подробное сообщение хука объясняет пользователю, что произошло и какой есть выход (`/request-shared-edit`).
- Стандартный git-механизм, понятный любому разработчику.

**Плохо:**
- Если Claude **должен** легитимно создать общий файл (как при первом `/setup-project`) — он не может. Решение: пишет в `.claude/handoff-requests/files/`, `/handoff-process` в main выносит в корень. Это лишний шаг.
- Escape hatch (`git commit --no-verify`) существует и может быть использован случайно или по ошибочной инициативе. Документировано как «не рекомендуется».
- `.claude/tmp/current-task.txt` должен быть **обязательно** записан скилом до первого коммита в worktree, иначе хук откажет без подсказки. Каждый worktree-скил отвечает за это.

## Замечание о non-ASCII именах файлов (правка 2026-05-31)

Хук матчит изменённые файлы по префиксам через `case`. Имена он берёт из `git diff --cached --name-only`. По умолчанию (`core.quotepath=true`) git эскейпит non-ASCII байты в octal и оборачивает весь путь в двойные кавычки: `ЗАКАЗЧИК.md` -> `".claude/handoff-requests/files/\320\227..."`. Ведущая `"` ломает `case`-матч по разрешённому префиксу `.claude/handoff-requests/`, и легитимный файл ошибочно считается нарушением. Прошлый setup-прогон наступил на это (ЗАКАЗЧИК.md в разрешённой зоне отклонялся хуком) и обходил через ручной `-c core.quotepath=false`.

Фикс: хук читает имена как `git -c core.quotepath=false diff --cached --name-only` (чистый UTF-8, без кавычек и octal). Префиксы у нас ASCII, кириллица всегда в суффиксе (матчится `*`), поэтому матч корректен независимо от локали.

**Правило для будущего кода:** любой код в машинерии (хуки, скрипты), который МАТЧИТ имена файлов из git-вывода (`diff`/`status`/`ls-files`), обязан использовать `-c core.quotepath=false`. Если git-вывод нужен только для проверки непустоты (как `git status --porcelain` в `/handoff`) - фикс не требуется.
