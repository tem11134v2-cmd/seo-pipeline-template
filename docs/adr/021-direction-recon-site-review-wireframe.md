# ADR-021: Контент-разведка направлений + кросс-страничный аудит + wireframe-прототипы (v5.1)

**Статус:** Принято (фрейминг «три слоя анти-ИИ» и «ИИ-узор сайта» пересмотрен в [ADR-022](022-commercial-copy-not-anti-ai.md): агенты direction-scanner/site-reviewer остаются, но переориентированы на боль ЦА / самоповторы, а не на анти-ИИ)

**Дата:** 2026-06-10

## Контекст

После рефакторинга ADR-020 (диета контекста писателя) глубокий аудит конвейера (61 находка, 56 подтверждено скептиками) вскрыл хвосты: mode B был логически разорван (планировщик не видел живую страницу), resume-точки в state machine были неоднозначны, маркер-хук не умел директории, кириллические регэкспы verify-copy были мёртвыми (`\b` ASCII-only). Параллельно заказчик сформулировал требования к «максимальной» версии скила: (1) доказательность КОНТЕНТА каждой страницы - что реально публикует топ-10 по этому направлению; (2) отдельный финальный агент, который проверяет, что тексты всего сайта качественные и не ИИ-шные; (3) библиотека шаблонов структуры/объёма, чтобы не писать с нуля (закрыто BLOCKS.md + blueprint в ADR-020, усилено образцовыми блоками в VOICE.md); (4) чёрно-белый прототип на согласование.

## Решение

### 1. direction-scanner - контент-разведка топ-10 по КАЖДОМУ направлению (шаг 2c)
Веерный агент (1 вызов = 1 направление, пачки 4-6, без expected-маркеров): `mcp_yandex_search` по маркеру + регион -> фильтр однотипных сайтов (агрегаторы/карты/статьи - вон) -> фетч 3-5 страниц (Chrome -> rendered, fetch -> fallback) -> тонкий `recon/<slug>.json` (<= 3 KB): published_info / offers_seen / blocks_common / features / **must_have** (публикуют все - без этого мы пустые) / **gaps** (нет ни у кого - отстройка). Потребители: block-planner (must_have -> обязательные блоки) и page-writer (свой recon - единственное «конкурентное» чтение писателя, диета сохранена). Флаг `--recon|--no-recon`, по умолчанию ВКЛ. Дополняет leader-block-scanner (тот - КОМПОЗИЦИЯ лидеров по ТИПАМ, этот - КОНТЕНТ топ-10 по направлениям).

### 2. Mode B через own_page в recon (закрытие разрыва ADR-020)
В mode B direction-scanner обязателен (даже при `--no-recon` - тогда без SERP) и дополнительно фетчит СВОЮ живую страницу -> `recon/<slug>.json.own_page` (блоки, объёмы, удачное). block-planner включает прошедшие правила блоки живой страницы с `keep:true` и лимитами от фактических объёмов ±15%; page-writer берёт из живой страницы только формулировки/фактуру - структура и объёмы ВСЕГДА по blueprint (двойная лояльность снята).

### 3. site-reviewer - кросс-страничный аудит (шаг 6d)
Один вызов на проект после копи-аудита: читает ВСЕ pages/*/page.json + VOICE.md + decisions и ловит то, что не видно на одной странице: межстраничные дубли формулировок, ИИ-узор сайта (одинаковые зачины/плашки/CTA на каждой странице), неуникальные H1/Title, дрейф decisions, конфликты цифр между страницами. Чинит вариативностью (внутри char-лимитов), пишет `site_audit.json` с вердиктом. Это третий слой анти-ИИ контроля: VOICE (профилактика, писатель) -> copy-auditor (страница) -> site-reviewer (сайт).

### 4. Wireframe-прототип по умолчанию
`theme-wireframe.css` (ч/б, монохром) - дефолтная тема прототипов: согласование фокусируется на ТЕКСТЕ, споры о цветах/вкусах исключаются. Цветные темы остаются опцией `--theme` / по запросу заказчика; `strategy.design_theme` от offer-strategist сохраняется в strategy.json для финального сайта.

### 5. State machine уточнена (resume-однозначность)
`approved -> blueprints-ready -> texts-written -> copy-audited -> site-reviewed -> texts-shared`. block-planner получил `pages_subset` (точечная доге-нерация, идемпотентный resume, шардинг по типу при > 12 страниц); guard «blueprints готовы -> не перегенерировать». Хук check-file.sh умеет директории (непустота) и чистит протухшие маркеры (> 60 мин) в fallback.

### 6. Сопутствующие фиксы аудита (56 подтверждённых)
Контракты opts/featured/вложенных массивов выровнены по всей цепочке blueprint -> page.json -> manifest -> HTML; meta.description пишет писатель (проходит весь контроль качества и попадает в docx); verify-copy получил живые кириллические регэкспы (lookaround вместо `\b`), проверку тире (п.14а), сверку blueprint-лимитов; build-tekst-docx перестал терять badge/price_note/cta/заголовки колонок; «защищены от» больше не конфликтует между шлюзом и методологией; кнопка CTA = «глагол + конкретика 15-30», а не «1 слово»; H1 выровнен на <= 60 во всех источниках; /guide узнал о существовании /seo-tekst и /seo-faq.

## Альтернативы
- **SERP-разведку в leader-block-scanner.** Отвергнуто: у него проектный уровень (типы страниц, один вызов), а разведка - попеременная (направление = запрос + свои конкуренты); смешение раздуло бы одного агента до двух ролей.
- **own_blocks.json отдельным сканом (вариант аудита).** Слито в recon/own_page: один веер вместо двух, те же инструменты, файл уже per-page.
- **site-reviewer как пункт 15 copy-auditor.** Отвергнуто: copy-auditor видит ОДНУ страницу - межстраничные узоры ему физически не видны; отдельный агент с другим срезом контекста (все тексты, ничего больше) - суть проверки.
- **Wireframe жёстко всегда.** Оставлена опция --theme: часть заказчиков просит «покажите красиво» осознанно.

## Последствия
**Хорошо:** каждая страница пишется со знанием, что публикует её топ (доказательность контента, не только структуры); mode B перестал быть фикцией; три слоя анти-ИИ; resume детерминирован; согласование по ч/б прототипу быстрее и дешевле по итерациям.
**Плохо:** +2 агента (46), +1 веер MCP-вызовов на проект (~5-7 на направление; отключаемо `--no-recon`); state machine длиннее (митигация: legacy-маппинг в SKILL.md 1a); ещё один контракт (recon) поверх blueprint. Порог «> 12 страниц -> шардинг block-planner по типам» - граничный компромисс решения «один вызов» из ADR-020.
