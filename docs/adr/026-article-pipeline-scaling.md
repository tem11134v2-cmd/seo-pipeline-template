# ADR-026: Масштабирование конвейера статей

**Статус:** Принято

**Дата:** 2026-07-16

## Контекст

`/seo-statya` - самый тяжелый скил конвейера, и при написании пачки статей одним
прогоном он упирался в контекст оркестратора и стоимость. Три узких места:

- **Шаг генерации фото (9b)** шел как 8-14 Skill-вызовов прямо в контексте
  оркестратора (генерация Nano Banana 2 + публикация Cloudinary + urls.json) - на
  каждую статью весь этот объем протаскивался через контекст.
- **Монолитный `SKILL.md` на 64 КБ** грузился целиком каждый прогон, хотя редкие
  аварийные ветки нужны в единичных случаях.
- **Серийный режим** был, но с багом: хуки (`check-section`, `mark-finalized`)
  видели только первую статью батча - они читали `current-task.txt`, общий на
  весь прогон.

## Решение

**Конвейер статей переведен на масштаб: вынос тяжелого шага в агента, распил
скила, серийная очередь и диета возвратов.**

- **Агент photo-producer (sonnet)** делает весь шаг 9b одним Task-вызовом вместо
  8-14 Skill-вызовов; у оркестратора остаются только аварийные ветки
  ожидания/`data:`-fallback. Контекст оркестратора перестал зависеть от числа
  иллюстраций.
- **Агент article-verifier (opus, шаг 10c)** - финальная вычитка собранной статьи
  ПОСЛЕ сборки HTML и ДО docx, state `verified` (частный случай паттерна
  [ADR-025](025-final-verifiers.md)).
- **Распил `SKILL.md` 64->55 КБ:** редкие ветки вынесены в `REFERENCE.md` (30 КБ,
  8 якорей), перенос без потерь (карта переноса сверена ревьюером против старого
  файла).
- **Серийный режим на очереди:** `batch-queue.mjs` (`init/next/set/status`) +
  `batch-queue.json` как единый источник; идемпотентный resume, инструкция после
  авто-компакта.
- **Диета возвратов:** section-writer <=5 строк, parent-fallback запрещен у
  article-finalizer, возвраты по [ORCHESTRATION.md](../ORCHESTRATION.md).
- **text-auditor размечает severity сам**, article-fixer-batch читает разметку.
- **Фикс серийного бага:** `current-article.txt` для хуков (fallback
  `current-task.txt`); `check-file.sh` больше не берет чужой expected-маркер при
  параллельных агентах.

## Альтернативы

- **Оставить генерацию фото инлайном.** Отвергнуто: 8-14 Skill-вызовов на статью
  затапливают контекст оркестратора, в батче нежизнеспособно.
- **Оставить монолитный `SKILL.md`.** Отвергнуто: 64 КБ каждый прогон, редкие
  ветки жгут бюджет; распил держит ядро тонким, `REFERENCE.md` грузится по нужде.
- **Серия без очереди (цикл в оркестраторе).** Отвергнуто: не переживает
  авто-компакт; очередь = единый источник + идемпотентный resume.

## Последствия

**Хорошо:** конвейер статей масштабируется на батч; контекст оркестратора ровный
независимо от числа фото; серия возобновляема; дешевле (sonnet-фото, диета
возвратов).
**Плохо:** больше подвижных частей (движок очереди, распил SKILL/REFERENCE) со
своими тестами (+batch-queue 14, +skill-split 4); карту якорей распила надо
держать в синхроне с ядром.
