# ADR-020: Диета контекста писателя (block-planner + VOICE.md + распил COPY.md) и анти-ИИ слой

**Статус:** Принято (анти-ИИ-слой ОТМЕНЁН в [ADR-022](022-commercial-copy-not-anti-ai.md) - был категориальной ошибкой для коммерции; диета контекста и block-planner/blueprint остаются в силе)

**Дата:** 2026-06-10

## Контекст

Первый боевой прогон /seo-tekst после порта на Claude Code: заказчик забраковал тексты («полная ерунда»). Аудит показал два корня:

- **Root A - перегруз контекста писателя.** `page-writer` перед первым словом текста читал ~55-65k токенов: BLOCKS.md (56 KB, каталог 45 блоков), COPY.md (45 KB, формулы + стоп-листы + чек-листы), fragments-manifest.json (14 KB), полный audience.json (все портреты со сценами), strategy.json, leader_blocks.json. В этом состоянии модель одновременно выбирала блоки, адаптировала ЦА, соблюдала char-метрики, формулы и стоп-листы - и письмо вырождалось в механическое заполнение слотов. Чем больше правил в контексте, тем «среднестатистичнее» проза.
- **Root B - анти-ИИ слой не был портирован.** Исходный браузерный скил держался на правилах человекоподобности; при порте они потерялись: STYLE.md (маркеры-слова ИИ, принцип неидеальности, запрет тире) читает только конвейер статей, в seo-tekst его не читал никто, и ни один из 13 пунктов copy-auditor не проверял «звучит ли как нейросеть». Дополнительный фактор успеха старой версии: писатель работал в почти пустом контексте (один A9_F1 + один шаблон) - правила соблюдались, потому что им ничего не мешало.

## Решение

Принцип: **писатель пишет, выбирает - планировщик, проверяет - контролёр.** Контракты между агентами - тощие.

### 1. block-planner - выбор блоков вынесен из писателя
Новый агент (один вызов на проект, шаг 6a, после клиентского гейта). Читает BLOCKS.md + leader_blocks.json + fragments-manifest + strategy/summary ЦА и пишет `blueprints/<page-slug>.json` (~2-3 KB на страницу): блоки в порядке, цель и тон каждого, какие боли закрывает, точные слоты и char-лимиты. Materials-gate и стадия прогретости применяются здесь.

### 2. Распил COPY.md по читателям
- **VOICE.md** (новый, для page-writer) - голос владельца бизнеса, анти-ИИ слой (типографика, слова-маркеры, структурная симметрия блоков, ритм), боевые правила старой браузерной версии (5 запрещённых паттернов + стиль), сжатые формулы письма (выгода «за счёт», факт+цифра+доказательство, Hero 5 шагов, H1/H2), тесты перед сдачей.
- **COPY-AUDIT.md** (новый, для copy-auditor) - стоп-листы v4 + таблицы замен + чек-лист, расширенный до **14 пунктов**: новый п.14 «маркеры ИИ-текста» (тире, слова-маркеры, штампы, симметрия карточек, пресс-релизный тон).
- **COPY.md** остаётся полным справочником для offer-strategist (формулы офферов, стадии) и faq-builder. В шапке - предупреждение о синхронизации выжимок.

### 3. page-writer - чистый писатель
Чтение сжато до: свой blueprint + VOICE.md + audience.json (summary + только свои портреты) + decisions/тезисы из strategy + facts.json + inputs. **Явный запрет читать BLOCKS.md / COPY.md / fragments-manifest / leader_blocks.** Контекст ~12-15k токенов вместо ~60k. Промт перестроен: голос и вживание в ЦА - первым приоритетом, механика слотов - вторым. Состав блоков менять запрещено.

### 4. copy-auditor - контролёр с пунктом 14
Читает COPY-AUDIT.md вместо COPY.md. У него минимальный контекст - именно он «свежими глазами» лучше всех ловит нейронность; п.14 чинит подачу, сохраняя факты и конкретику.

## Альтернативы
- **Только вернуть анти-ИИ правила, не трогая контекст.** Отвергнуто: правила старой версии (штампы) уже были в COPY.md и не соблюдались - тонули в перегрузе. Лечить надо оба корня.
- **Блок-план веером (на страницу).** Отвергнуто: выбор блоков требует целостного взгляда (anti-duplication между страницами одного типа, единый порядок) и читает одни и те же 70 KB каталогов - один вызов дешевле и консистентнее.
- **Писатель пишет свободный текст, маппинг в слоты отдельным проходом.** Отвергнуто: +1 проход на страницу и риск потерь при маппинге; слоты в blueprint достаточно компактны, чтобы не душить прозу.
- **Audience-сводку выносить в отдельный файл.** Пока нет: audience.json умеренного размера, инструкция «summary + свои портреты» решает; вернёмся, если audience.json распухнет.

## Последствия
**Хорошо:** писатель работает в чистом контексте (как в успешной браузерной версии), анти-ИИ слой существует и проверяется отдельным проходом; выбор блоков стал консистентным по всему проекту (один планировщик вместо N писателей); правки методологии локализованы по читателям.
**Плохо:** +1 агент и +1 последовательный шаг перед веером (один вызов, терпимо); три файла методологии вместо одного - риск рассинхронизации выжимок (митигация: предупреждение в шапке COPY.md); blueprint - ещё один контракт, который надо поддерживать при изменении BLOCKS/fragments. Граничный компромисс «одного вызова»: при > 12 страниц block-planner шардируется пачками по типу страниц с `pages_subset` (ADR-021).
