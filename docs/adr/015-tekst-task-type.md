# ADR-015: Новый тип задачи `texts/` для конверсионных текстов + HTML-прототип (/seo-tekst)

**Статус:** Принято

**Дата:** 2026-06-06

## Контекст

Портируем две авторские claude.ai-услуги в один скил `/seo-tekst`:
- **`seo-copywriting`** (У5 подготовка ТЗ + У6-Ф1 конверсионный текст) - библиотека из 45 блоков с метриками, глубокий анализ ЦА, написание копии блок-за-блоком.
- **`seo-prototype-v3`** (бриф -> research -> audience -> стратегия -> копирайтинг по формулам Сургая -> **сборка HTML-прототипа**) - продвинутый анализ и «термоядерные» формулы, но многочатовость, дубль research и **LLM руками печатает 72 КБ HTML**.

Развилки:
1. **Куда складывать?** Уже есть `articles/`, `strategies/`, `analyses/`, `structures/`, `metatags/`, `audits/`. Тексты+прототип - ещё один тип задачи.
2. **Как генерить HTML?** Гайд (SKILL-ADAPTATION-GUIDE) прямо называет «html-builder субагента» антипаттерном: сборка HTML = скрипт. Но заказчик хочет, чтобы тексты писались первыми, а потом ОТДЕЛЬНЫЙ субагент собирал прототип, и «не с нуля каждый раз».
3. **Параллелизм.** Заказчик хочет вести ~10 страниц одновременно.
4. **Согласование с клиентом.** Заказчик хочет автономный прогон с обязательной паузой: промежуточный файл (анализ ЦА + вводные) уходит клиенту на согласование (docx -> Google Doc), цикл правок, дальше.
5. **Где живёт общая методология** (45 блоков, копи-формулы, HTML-кит, дизайн-системы, legal)?

## Решение

### 1. Новый тип задачи - `texts/NNN-<slug>/`

Каждый запуск создаёт `texts/NNN-<slug>/` рядом с прочими. Внутри - `meta.json` (state machine), `inputs.json`, `pages.json`, `audience.json`, `strategy.json`, клиентские `Analysis_<slug>.docx` / `Texts_<slug>.docx`, и `pages/<page-slug>/{page.json, manifest.json, prototype.html}`. Pre-commit универсален к task-dir (ADR-003); скил пишет `current-task.txt = texts/NNN-slug/` первым делом. **Пятое повторное применение паттерна** ADR-007 (после 009/010/012/014). Тривиально.

### 2. HTML = манифест-JSON + детерминированный сборщик + тонкий субагент-сборщик (осознанное отступление от гайда)

LLM **пишет текст и выбирает блоки** (`page-writer` -> `page.json`). Затем `prototype-builder` (субагент) **достраивает рендер-решения** (тема, legal, opts, meta) в `manifest.json` и запускает **детерминированный** `build-prototype.mjs`, который защёлкивает копию в обкатанный kit (shell + component-CSS на токенах + 14 фрагментов со слотами + 6 тем + vanilla JS + legal). Сам сборщик - чистая шаблонизация (рекурсивный mini-template: вложенные `REPEAT`/`IF`/`CLASS`/`@index`), CSS/JS LLM не пишет.

Это **отступление** от «сборка HTML = только скрипт» в гайде, зафиксированное здесь (гайд требует ADR при отступлении). Причины:
- Заказчик явно хочет «генерацию HTML в отдельном субагенте» с порядком «сначала тексты, потом прототип». `prototype-builder` несёт реальные суждения (выбор фрагмента/темы/opts, заполнение legal, цикл `verify -> fix`), а не печатает HTML.
- Kit (CSS/JS/фрагменты/темы) **взят целиком** из `seo-prototype-v3` как переиспользуемый ассет - так сохраняется «крутой HTML» коллеги, но без хрупкости LLM-печати 72 КБ: детерминированно, проверяемо, **параллелится**, дёшево.
- Анти-паттерн гайда касался агента, который ПЕЧАТАЕТ HTML по шаблону. Здесь печатает скрипт; агент только маршрутизирует копию в фрагменты и проверяет результат. Дух правила соблюдён.

### 3. Клиентский гейт согласования (revising-цикл, как /seo-analiz)

После проектного анализа (`audience-analyst` + `offer-strategist`) собирается `Analysis_<slug>.docx`, грузится в Drive (Google Doc), и скил **обязательно паузится**: заказчик отправляет ссылку клиенту, получает «согласовано» или правки. Правки -> точечно в `audience.json`/`strategy.json` или пере-делегирование агента -> пере-сборка docx -> новая ревизия. Цикл до approved, затем пишем тексты. Это единственная обязательная пауза (прогон в остальном автономен); `--review` добавляет вторую паузу после текстов. Плюс финальный `Texts_<slug>.docx` -> Google Doc (выдача), прототипы - локальные `.html` (Drive не конвертирует HTML в Google-формат).

### 4. Двухуровневый веер: проектный анализ один раз + постраничный параллелизм

`audience-analyst` и `offer-strategist` - **проектного уровня** (один прогон на задачу, общий контекст для всех страниц). `page-writer` и `prototype-builder` - **постраничные**, запускаются веером пачками по 6-8 (паттерн `seo-metategi`/`audit-onpage`). Для 10 страниц: 2 проектных агента + 10 писателей + 10 сборщиков. Параллельные агенты **не пишут expected-маркеры** (single-marker hook ломается на веере - см. ADR-012 п.3); полноту проверяет наличие `page.json`/`prototype.html` + `verify-prototype.mjs`.

### 5. Verify скриптом, не hook'ом; `settings.json` не меняется

`verify-prototype.mjs` (структурные инварианты по HTML + контентные по manifest: одна форма в финале, header/footer/tel/cookie, без фреймворков, без длинного тире, стоп-формулы, H1/маркер). Hook несовместим с параллельным веером (ложные отказы). Совпадает с `verify-metatags.mjs`/`verify-markers.mjs`. **Новых hook'ов нет, `settings.json` не трогаем.**

### 6. Методология + kit - в папке скила, коммитится (как PLAYBOOK ADR-012)

`BLOCKS.md` (45 блоков), `COPY.md` (формулы), `KIT-SPEC.md`, `LEGAL.md`, `prototype.css/js`, `PROTOTYPE-MASTER.html`, `fragments/`, `themes/`, `legal/` - всё в `.claude/skills/seo-tekst/assets/`, в репозитории, self-contained (новые машины получают всё с `git clone`). НЕ в `~/.claude/seo-knowledge/` - та папка не коммитится в шаблон (переносится вручную), а CLAUDE.md просит не писать в неё из worktree. Будущий `/seo-faq` читает общий срез (блоки 33-39, COPY.md) из папки seo-tekst (read-only кросс-скил-зависимость, документирована).

## Альтернативы

- **A. LLM печатает HTML напрямую** (как у коллеги). Отвергнуто: токенно-затратно, хрупко на 10 страницах, тяжело верифицировать, ломает параллелизм, CSS «улучшается» моделью. Манифест+сборщик даёт тот же выход надёжнее.
- **B. Чистый скрипт без субагента-сборщика** (как `assemble-html.mjs`). Отвергнуто по явному пожеланию заказчика (отдельный субагент генерации) и потому что выбор фрагмента/темы/opts + цикл `verify->fix` требуют суждения. Скрипт остаётся ядром.
- **C. Методология в глобал `~/.claude/seo-knowledge/`** (как STYLE/GENRES, ADR-004). Отвергнуто: глобал не коммитится в шаблон. Skill-local (ADR-012) - self-contained.
- **D. Объединить с /seo-statya.** Отвергнуто: статья = блог (другой шаблон nx-article), тут = коммерческий лендинг (Hero/тарифы/форма). Разные шаблоны и жанры.
- **E. /seo-faq внутри /seo-tekst.** Отложено в отдельный скил/ADR-016: SEO-нормализация (JM N-граммы) работает и на чужих живых страницах, не только на свежесгенерённых; отдельный тип задачи `faq/`.

## Последствия

**Хорошо:**
- Встраивается без изменений pre-commit, `/handoff`, `/handoff-process`, **`settings.json`**. Подтверждает многозадачность ADR-001.
- Kit коллеги переиспользован целиком - «крутой HTML» без его хрупкости; per-page параллелизм для ~10 страниц.
- Не пере-анализируем конкурентов/SERP - ингестим `analyses/NNN` + `structures/NNN` (выигрыш скорости).
- Клиентский гейт даёт заказчику видимость работы и точку правок до дорогого этапа написания.

**Плохо:**
- Отступление от «HTML=скрипт» - задокументировано (п.2), не уходит в устную традицию. Граница «суждение vs шаблонизация» между `prototype-builder` и `build-prototype.mjs` требует дисциплины (агент не должен печатать HTML руками).
- Кросс-скил-зависимость `/seo-faq` -> `seo-tekst/assets` (общий срез блоков/COPY). Митигация: документировано; при росте - вынести общий слой отдельным решением.
- Drive не конвертирует HTML -> прототипы отдаются файлами (не Google-ссылкой). Приемлемо (клиент открывает .html в браузере).
- `texts_folder_id` в DRIVE.md - новый ключ (TODO до создания папки «Тексты»); до этого шаги 5/7 скипают загрузку с подсказкой (как было у metatags).
- 5 новых агентов + 5 скриптов + kit - рост машинерии. Митигация: чёткие контракты, README обновлён.
