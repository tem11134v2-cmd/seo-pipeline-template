# ADR-010: Новый тип задачи `structures/` для построения структуры сайта на базе предпроектного анализа

**Статус:** Принято

**Дата:** 2026-05-29

## Контекст

При портировании claude.ai-скила `seo-structure` (старое имя — «ПРОМТ-СТРУКТУРА», У3 в авторской методологии: мастер-список страниц → маркерные запросы → JM semantic_pack → топ-10 + каннибализация → A6.xlsx → A6.md) возникли три развилки.

1. **Где живут артефакты?** Уже есть `articles/NNN/`, `strategies/NNN/`, `analyses/NNN/`. Структура сайта — ещё один тип задачи: один комплект (`A6.xlsx` → клиенту, потом `A6.md` в проект). Либо новая папка-тип, либо подкаталог в `analyses/`.

2. **Как стыковаться с `/seo-analysis`?** Старый скил на входе читал A2.md (markdown-отчёт) и A3.md (стоп-лист) — это были артефакты из «ручного» процесса У1. Но в нашей архитектуре `/seo-analysis` уже собирает **машиночитаемые JSON** (`brief.json` с `client_pages` и `keyso_base`, `competitors.json` с `direct[]` и `leaders_top3`, `serp.json` со списком `stop_list`, `leader_scan.json`). Парсить markdown — это терять данные, которые `analysis-writer` тщательно складывал в структурированный вид.

3. **Шаг 4 «топ-10 + каннибализация» — агент или скрипт?** Часть работы детерминирована (фильтр стоп-слов, сортировка по частоте, отбор N=10, детекция дублей). Часть требует творчества — разрешение конфликтов («равнозначные — оставить там, где маркер ближе по смыслу») и формулировка рекомендаций по расширению.

## Решение

### 1. Новый тип задачи — `structures/NNN-<domain-slug>/`

Каждый запуск `/seo-structure <NNN>` создаёт папку `structures/NNN-<domain-slug>/` рядом с `articles/`, `strategies/`, `analyses/`. Внутри — `meta.json` (state machine), промежуточные JSON (`master_list.json`, `markers.json`, `semantic_pack.json`, `top10.json`, `cannibalization.json`), `A6_<slug>.xlsx` для клиента, `client_filled.xlsx` (правленая клиентом версия), `structure_data.json` (распарсенная клиентская версия) и финальный `A6.md`.

Pre-commit hook менять не нужно — он универсален к task-dir (ADR-003). Скил пишет `.claude/tmp/current-task.txt = structures/NNN-slug/` первым делом.

Это **третье повторное применение паттерна** ADR-007 (после ADR-009 для `analyses/`). Решение тривиально и не вводит новых архитектурных понятий.

### 2. Вход скила — `<NNN>` существующего анализа, не сырой бриф

`/seo-structure <NNN>` (обязательный позиционный аргумент) — где `<NNN>` это номер существующей папки `analyses/NNN-*/`. Скил читает не A2.md/A3.md, а **JSON-источники**:

- `analyses/NNN/brief.json` — даёт `slug`, `keyso_base`, `client_pages`, `assortment`, `client_target_queries`
- `analyses/NNN/competitors.json` — даёт `direct[]` (6-10 конкурентов) и `leaders_top3` (для базы анализа)
- `analyses/NNN/serp.json` — даёт `stop_list` (для исключения агрегаторов)
- `analyses/NNN/leader_scan.json` — даёт контекст лидерских блоков (информационно для рекомендаций)
- `analyses/NNN/A3.md` — параллельно проверяем актуальность стоп-листа

Если папки `analyses/NNN-*/` нет — скил отказывает с подсказкой запустить `/seo-analysis`. Это правильно: предпроектный анализ — обязательный предшественник, всегда. Запускать построение структуры без анализа — путь к ошибкам в типизации, конкурентах и регионах.

Fallback `--brief <path>` (для legacy-данных или клиентов без полного анализа) **не делается** — увеличивает поверхность скила вдвое без явной ценности. Если ситуация возникнет — добавим позже отдельной развилкой.

### 3. Шаг 4 — гибрид: скрипт `select-top10.mjs` + агент `cannibalization-resolver`

Скрипт делает 80% детерминированной работы:

- Загружает `semantic_pack.json` (топ-30 от JM по каждой странице).
- Фильтрует стоп-слова по правилам (бренды конкурентов из A3.md, дубли форм слова, орфографические ошибки по эвристикам).
- Сортирует по `freq_exact` убыванием.
- Отбирает топ-10 на страницу.
- **Детектит дубли:** собирает все запросы из топ-10 всех страниц, находит запросы, которые попали в топ-10 двух или более страниц.
- Выдаёт `top10.json` + `cannibalization.json` (список конфликтов с описанием альтернатив из топ-30).

Агент `cannibalization-resolver` (Read/Write, без MCP) принимает решения только там, где скрипт не может:

- Какую страницу оставить при равнозначной частотности маркера (смысловая близость маркера к запросу).
- Формулирует рекомендации по расширению (запросы, под которые у конкурентов есть отдельные страницы, а у нас нет).

Это экономит токены (модель не сортирует числа) и делает работу предсказуемой. Принцип SKILL-ADAPTATION-GUIDE: «детерминированная трансформация → скрипт; творчество → агент».

## Альтернативы

### A. Вписать структуру в `analyses/NNN/structure/`

Подкаталог внутри уже существующей папки анализа. Отвергнуто:
- Семантическая путаница (анализ = аналитика, структура = технический документ для разработки сайта).
- `meta.json` стал бы вложенным с двумя state machines — анти-паттерн.
- Расшаривание разное: A2 = Google Doc, A6 = Google Sheet, в DRIVE.md разные якоря.
- В будущем один клиент может иметь несколько структур (рестайл, новый раздел) — отдельные `structures/NNN/` это поддержат, а вложение в анализ — нет.

### B. Принимать сырой бриф (как у `/seo-analysis`)

Сделать `/seo-structure` самодостаточным: бриф → 6 шагов → A6. Отвергнуто:
- Дублирование работы `brief-structurer` и `competitor-finder` — два набора агентов решают одну задачу.
- При рассинхронизации (например, в анализе типизация конкурентов одна, а в структуре — другая) клиент получает противоречивые документы.
- Старый скил так делал не потому что это правильно, а потому что в claude.ai не было файловой системы. У нас она есть — используем.

### C. Один большой агент `structure-builder` на всю задачу

Творческое решение всех 6 шагов в одном вызове. Отвергнуто:
- Раздувание контекста: агент читал бы JM-результаты, страницы конкурентов, спаривание, фильтры — всё одновременно.
- Свежесть пропадает: после JM-расширения для каждой из 15-20 страниц нужна отдельная фокусировка на каннибализацию.
- Нарушение принципа SKILL-ADAPTATION-GUIDE: «если нужен другой фокус — разделяй».

### D. Шаг 6 (A6.md) — только скрипт без LLM

Распарсить xlsx, пересобрать markdown по шаблону. Отвергнуто (мягко — оставлено как возможный возврат):
- A6.md содержит «причина исключения» для отложенных страниц — там нужна формулировка (если клиент написал «обсудить», агент решает как переформулировать; если клиент вообще не заполнил — нужна сводка).
- Шапка с метаданными проекта (регион, лидеры) требует подстановки из JSON.

Скрипт `import-structure.mjs` парсит xlsx и собирает `structure_data.json` (детерминированно). Финальный markdown пишет агент `structure-writer` (Read/Write) — он гибкий к нюансам клиентских комментариев и формирует «Отложенные» секцию с причинами.

## Последствия

**Хорошо:**

- `/seo-structure` встраивается в существующую механику без изменений pre-commit hook, `/handoff`, `/handoff-process`, settings.json. Подтверждает мультизадачность ADR-001.
- Стыковка с `/seo-analysis` через JSON — точная, без потери данных в markdown-парсинге.
- A6.md в `structures/NNN/A6.md` — машиночитаемый артефакт, готовый для будущих скилов У5 (ТЗ верстальщика) и У6 (контент-план).
- Drive якорь «Структуры» — клиент заполняет колонку «Целевая?» в Google Sheet онлайн, не качает локально (если папка в DRIVE.md заведена).
- Гибрид скрипт + агент на шаге 4 — образец для будущих скилов с похожей развилкой.

**Плохо:**

- Жёсткая зависимость от `/seo-analysis`: пользователь не может запустить структуру без анализа. Это правильно архитектурно, но новые пользователи могут попробовать «прыгнуть через шаг». Митигация: на старте скил даёт чёткую ошибку с подсказкой `/seo-analysis`.
- Drive якорь `structures_folder_id` нужно завести вручную в Drive (как `analyses_folder_id` в своё время). До этого `share-structure` скипает. Документировано в DRIVE.md.
- Старый скил рассчитывал на A6.md как **единственный** артефакт для У4-У8. У нас A6.md остаётся, но дополнительно есть `structure_data.json` (машиночитаемый) — будущие скилы могут читать оба, и обновлять структуру при изменении одного из них надо синхронно. Митигация: скрипт `import-structure.mjs` пересобирает оба из xlsx (single source of truth — xlsx).
