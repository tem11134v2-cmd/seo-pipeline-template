# ADR-009: Новый тип задачи `analyses/` для предпроектного анализа конкурентов

**Статус:** Принято

**Дата:** 2026-05-28

## Контекст

При портировании claude.ai-скила `seo-analysis` (предпроектный анализ конкурентов: бриф клиента → A2-отчёт + A3-стоп-лист) возникли три развилки:

1. **Куда складывать артефакты?** Уже есть `articles/NNN/` (статьи) и `strategies/NNN/` (стратегии + сметы). Предпроектный анализ — это ещё один тип задачи: один комплект документов на запуск (A2.md + A3.md + опц. .docx), без множественной структуры внутри. Либо новая папка-тип, либо вписать в существующие.

2. **Переиспользовать ли агентов `/strategy`?** В `/strategy` уже есть `strategy-scanner` (скан клиента) и `competitor-analyst` (поиск конкурентов + типизация + вердикт). На первый взгляд они покрывают часть работы предпроектного анализа.

3. **Активация — как у исходника (триггер «ПРОМТ-АНАЛИЗ»)?** В claude.ai была команда-префикс + восемь шагов с командой «дальше» между ними. Это техдолг платформы. В Claude Code этот паттерн заменяется slash-командой + state machine.

## Решение

### 1. Новый тип задачи — `analyses/NNN-<domain-slug>/`

Каждый запуск `/seo-analysis` создаёт папку `analyses/NNN-<domain-slug>/` рядом с `articles/` и `strategies/`. Внутри — `meta.json` (state machine), исходный бриф, промежуточные JSON и финальные артефакты (`A2.md`, `A3.md`, опц. `A2_<slug>.docx`).

Pre-commit hook менять не нужно — он универсален к task-dir (см. ADR-003). Скил пишет `.claude/tmp/current-task.txt = analyses/NNN-slug/` первым делом.

Это **повторное применение паттерна** ADR-007 (где появился `strategies/`). Решение тривиально и не вводит новых архитектурных понятий.

### 2. НЕ переиспользуем `strategy-*` агентов

Создаём 5 новых агентов: `brief-structurer`, `competitor-finder`, `serp-verdict`, `leader-scanner`, `analysis-writer`. Причины:

- **Разные выходные схемы.** `brief.json` имеет 16 параметров с категоризацией УТП (технические / сервисные / социальные) и блоками «⛔ запрещённые формулировки», «🚫 чего нет в ассортименте», «Данные для анализа ЦА». Это специфично для предпроектного анализа и не нужно в `/strategy.inputs.json`.
- **Разный словарь вердикта.** `seo-analysis` использует `ИДЁМ / КОРРЕКТИРУЕМ ТИП / МЕНЯЕМ СТРАТЕГИЮ / ИДЁМ С ОГОВОРКАМИ` (4 варианта). `/strategy` использует `ИДЁМ / РАСШИРЯЕМ / ИНФОКОНТЕНТ / С_ОГОВОРКАМИ / НОВЫЙ_САЙТ` (5 вариантов). Семантика частично пересекается, но рекомендации и формулировки разные.
- **Глубокий «скан смыслов» топ-3 (Э2-лайт)** — отсутствует в `/strategy`. Это отдельная работа: по 3-4 страницы каждого лидера через `mcp_fetch_page`, с фиксацией блоков, посылов, фишек, и сводкой паттернов.
- **Разный вход.** `/strategy` ожидает URL + интерактивный Q&A. `seo-analysis` ожидает свободный бриф (текст, заполненный опросник, расшифровка разговора) и парсит из него 16 параметров.

Попытка натянуть `/strategy`-агенты потребовала бы ветки внутри них с if-else по «откуда вызвали», что нарушает контракт «вход → выход» агентов (см. SKILL-ADAPTATION-GUIDE). Лучше отдельные агенты с похожей внутренней методологией.

### 3. Активация — `/seo-analysis`, без триггер-фраз

Скил вызывается командой `/seo-analysis [бриф или путь к файлу с брифом]` и проходит все 6-7 шагов сам через state machine. `--resume` поддерживается стандартно. Шаги «дальше» / «шаг N» из исходника исчезают как класс — это техдолг claude.ai.

Шаг 7 (генерация .docx) — опциональный. Скил в конце шага 6 спрашивает пользователя «сгенерировать .docx для клиента?» и либо делает шаг 7, либо завершает.

## Альтернативы

### A. Вписать предпроектный анализ в `strategies/`

Сделать подтип внутри стратегий: `strategies/NNN-<domain>-analysis/`. Отвергнуто:
- Семантически путает — стратегия (с тарифами и сметой) ≠ предпроектный анализ (без тарифов, для последующих этапов У3/У5).
- Один клиент может иметь и анализ, и стратегию — это два независимых документа.

### B. Переиспользовать `strategy-scanner` + `competitor-analyst`

Аргумент: «не плодим агентов, общая методология». Отвергнуто по причинам в Решении 2. Дополнительный фактор: при последующих изменениях одного из скилов агенты придётся разводить — лучше развести сразу.

### C. Перенести «ПРОМТ-АНАЛИЗ» как описание триггера

Через description скила в frontmatter. Отвергнуто: Claude Code не поддерживает префиксные триггеры в текстовых сообщениях — только slash-команды. Любые попытки эмулировать «дальше» через инструкции в чате — нагромождение.

### D. Полностью склеить с `/strategy` и сделать общий «pre-project research» скил

Аргумент: «у обоих общая фаза изучения клиента и конкурентов, давайте сделаем один скил с веткой». Отвергнуто:
- Резко увеличивает сложность одного скила (и так не маленький).
- У них **разные финальные артефакты** — стратегия+смета vs A2+A3+опц.docx.
- Пользователь должен иметь возможность сделать только анализ, не запуская стратегию (например, для будущих этапов У3/У5 без коммерческого предложения).

## Последствия

**Хорошо:**

- `/seo-analysis` встраивается в существующую механику без изменений pre-commit hook, `/handoff`, `/handoff-process`, settings.json. Подтверждает, что архитектура многозадачна (как заявлено в ADR-001).
- Параллель: можно одновременно вести `articles/`, `strategies/`, `analyses/` в разных worktree.
- Паттерн «новая папка-тип + N агентов + 1 скрипт» применён повторно (после ADR-007) — это уже устоявшийся способ расширять систему. Образец для будущих скилов (`/audit-tech`, `/commerce-offer`).
- Артефакты A2.md + A3.md — машиночитаемые, готовы для передачи в будущие скилы (У3 — структура страниц, У5 — ТЗ на тексты).

**Плохо:**

- Дублирование методологии: `competitor-finder` и `strategy-*.competitor-analyst` решают похожие задачи. Если методология поиска конкурентов меняется (например, новый источник кроме Keyso) — надо править оба. Митигация: подробные промты с указанием источника правды (Keyso база, путь А/Б/В/Г), при изменении методологии — обновлять оба синхронно.
- 5 новых агентов плюсом к 16 существующим — общее количество растёт. Митигация: каждый агент имеет чёткий контракт (вход → выход), список агентов в README обновлён.
- `build-analysis-docx.mjs` дублирует часть логики `build-strategy-docx.mjs` (стили Arial, тёмно-синие заголовки, таблицы с альтернативной заливкой). Митигация: пока скрипты простые (рендер plain-JSON в .docx), общий слой не выносим — это была бы преждевременная абстракция. Если появится третий похожий скрипт — задумаемся.
