# ADR-004: MCP-серверы и общая методология — глобально, не в проекте

**Статус:** Принято

**Дата:** 2026-05-24

## Контекст

В системе есть **два слоя общих ресурсов**, не связанных с конкретным клиентом:

1. **MCP-серверы** (JM, Wordstat, Keys.so, Arsenkin, Webmaster, Yandex, Fetch, Sheets) — внешние сервисы, к которым обращаются агенты.
2. **Методология SEO** (STYLE.md, GENRES.md, ENHANCEMENTS.md, SVG-ICONS.md, TEMPLATE-MASTER.html, CLIENT-TEMPLATE.md) — авторское знание, как писать живой текст, какие жанры применять, какие HTML-элементы вставлять.

Оба слоя одинаковы для всех клиентов. Вопрос — где их хранить.

## Решение

Оба слоя — **глобально на машине пользователя**, не в проекте клиента.

### MCP-серверы

Подключены в **глобальной конфигурации Claude Code Desktop** (через настройки приложения). Имена MCP — это UUID-серверов, доступ к ним получают все сессии независимо от папки. Проектный `.mcp.json` **не используется** (присутствует только как `.mcp.json.example` для документации формата).

В frontmatter агентов нет секции `mcpServers:` — без неё агент получает доступ ко всем MCP, доступным в текущей сессии.

### Методология SEO

Лежит в `~/.claude/seo-knowledge/`:
```
~/.claude/seo-knowledge/
├── STYLE.md
├── GENRES.md
├── ENHANCEMENTS.md
├── SVG-ICONS.md
├── TEMPLATE-MASTER.html
└── CLIENT-TEMPLATE.md
```

Агенты читают эти файлы по абсолютному пути.

## Альтернативы

### A. MCP в `.mcp.json` каждого проекта

Стандартный способ Claude Code — проектный `.mcp.json` с URL'ами и токенами. Отвергнуто:
- URL'ы (`https://<vps>/mcp/jm`) и токены везде одинаковые — дублирование.
- При смене URL VPS пришлось бы редактировать `.mcp.json` в каждом клиенте.
- MCP уже глобально подключены к Claude Code Desktop — двойная конфигурация запутывает.

### B. Методология копируется в каждый проект

В `~/seo-projects/<client>/.claude/knowledge/` — копия STYLE.md и пр. Отвергнуто:
- При улучшении методологии нужно проходить по всем клиентам.
- При git clone из template-репо methodology попадает в каждый клиент → клиентов десятки, дубли × 10.
- Это **общая методология автора**, не пер-клиентское знание.

### C. Методология как git submodule

`~/seo-projects/<client>/seo-knowledge/` — submodule на отдельный репо. Отвергнуто:
- Submodule-ы в git — головная боль (нужно помнить `--recursive` при клонировании).
- Не нужно никому кроме автора — методология не публичная.

## Последствия

**Хорошо:**
- Один источник истины. Обновил STYLE.md — все клиенты сразу получают новое.
- Меньше дублей, меньше места.
- Проекты клиентов в репо template чище — нет ничего лишнего.

**Плохо:**
- **Хрупкость переноса на другую машину.** Если пользователь меняет компьютер — нужно вручную перенести `~/.claude/seo-knowledge/`. Это решаем отдельным git-репо (см. README этого репо или [ADR-006](006-github-distribution.md) для аналогичного механизма).
- Клиент-проект сам по себе не запускается на чужой машине без global knowledge — но это сознательное ограничение (методология — приватная).
- Если в `~/.claude/seo-knowledge/` нет нужного файла — агент молча использует фолбэк (если он есть) или падает с непонятной ошибкой. Документировано в README → требования.
