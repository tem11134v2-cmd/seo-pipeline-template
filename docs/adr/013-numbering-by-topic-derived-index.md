# ADR-013: Нумерация папок статей по номеру темы + случайный суффикс; `_index.json` - производный кеш

**Статус:** Принято

**Дата:** 2026-06-04

**Уточняет:** [ADR-003](003-pre-commit-whitelist.md) (убирает исключение `_index.json` из белого списка pre-commit).

## Контекст

Боевой прогон вскрыл класс гонок при параллельной работе двух worktree над статьями.

1. **Гонка счётчика номера.** `write-article` вычислял `NNN = max(существующих nnn) + 1`. Две worktree читают каждая свой `articles/_index.json` (в своей ветке, до коммита соседа), получают одинаковый `max+1` → коллизия номера и/или имени папки.

2. **Затирание записей в индексе.** `update-index.mjs` ключевал запись по числовому префиксу (`^(\d{2,4})-`). Две статьи одной темы (`005-...-a`, `005-...-b`, разные жанры) дают один ключ `005` → вторая затирает первую.

3. **Merge-конфликт по общему реестру (корень).** `articles/_index.json` - git-трекаемый файл, в который дописывает КАЖДАЯ статья, и он был **единственным санкционированным исключением** из инварианта ADR-001 (через белый список pre-commit, ADR-003). Две параллельные worktree обе дописывают в массив `articles[]` → гарантированный merge-конфликт на `/handoff`, как папки ни переименовывай.

Переименование папок (идея «`001-dko`») чинит только 1 и 2. Конфликт по `_index.json` (3) остаётся - это корень, и он требует архитектурного решения, а не косметики.

## Решение

### 1. Идентичность статьи: номер темы + случайный суффикс

- Номер папки `TTT = zero-pad(topic_id, 3)` - **детерминирован** (= номер темы из колонки № в `topics.xlsx`), `max+1` не вычисляется → гонки счётчика нет.
- Имя папки: `articles/<TTT>-<slug>-<rand3>/`, где `rand3` - 3 случайные строчные латинские буквы. Развязывает коллизию имени для нескольких статей одной темы (разные жанры/площадки) и для двух worktree на одной теме.
- **`NNN` (= `TTT`) больше НЕ уникален.** Уникален полный basename папки. Команды, принимавшие «номер статьи», принимают теперь номер темы ИЛИ полный id папки и разруливают неоднозначность через `resolve-article-dir.mjs` (не «берут первую молча»): `fix-article`, `rewrite-section`, `share-article`, `write-article --rebuild-docx`.

### 2. `_index.json` - производный кеш, а не источник истины

- Источник истины - per-folder `meta.json` (он и так уникален на папку, конфликтовать не может).
- `_index.json` **gitignored** и пересобирается из `articles/*/meta.json` скриптом `rebuild-index.mjs <root>`. Любой читатель индекса (`write-article` шаг 1a, `new-topics` дедуп) сначала пересобирает, затем читает.
- `update-index.mjs` остаётся как инкрементальный апдейтер (вызывается из `update-meta.sh`), но ключует по полному basename (`key`), а не по `nnn`. Пишет в локальный gitignored файл - конфликтовать не с чем.
- Из белого списка pre-commit убраны `articles/_index.json` / `strategies/_index.json` / `analyses/_index.json` (последние два были легаси - их никто не писал). Инвариант ADR-001 восстановлен: worktree трогает только свою папку → merge без конфликтов.

## Альтернативы

- **Только переименование папок (`001-dko`), индекс оставить трекаемым.** Отвергнуто: чинит гонку счётчика и коллизию имени, но НЕ merge-конфликт по `_index.json` - даёт ложное ощущение решённости.
- **CRDT/построчный merge `_index.json` через `.gitattributes merge=union`.** Отвергнуто: union-merge оставляет дубли/битый JSON, требует ручной чистки; производный кеш проще и надёжнее.
- **UUID вместо номера темы.** Отвергнуто: теряется человекочитаемая связка «тема №5 = папка 005», которую пользователь хочет (CLI-цифра = номер темы = номер папки).

## Последствия

- **Плюс:** ноль merge-конфликтов при параллельных статьях; предсказуемая связка номер темы ↔ папка; несколько статей одной темы сосуществуют корректно.
- **Минус/миграция:** существующие клиентские проекты, где `_index.json` уже закоммичен, при первом `/sync-from-template` + чтении получат `git rm --cached <root>/_index.json` (индекс перейдёт в gitignored-кеш). Делается один раз на проект.
- **Связанное (отложено):** та же логика «детерминированный номер + производный кеш» применима к `topics/`, `strategies/`, `analyses/`, `structures/`, `metatags/` (тоже `max+1` + свои `_index.json`). Они 1-на-клиента и параллелятся редко → отдельный заход. Нумерация батча тем (`topics/NNN`) - это НЕ `topic_id` (номер темы внутри `topics.xlsx`), поэтому идентичность статьи от неё не зависит.
