# ADR-014: Новый тип задачи `audits/` для технического SEO-аудита

**Статус:** Принято

**Дата:** 2026-06-05

## Контекст

Портируется claude.ai-скил `seo-techaudit` (технический SEO-аудит сайта под Яндекс: разведка/карточка → индексация и тех-здоровье → URL/мета/Schema/JS → аналитика/поведенческие/ссылки → отчёт A12.md с проблемами 🔴🟡🟢, чеклистом разработчику и динамическими приложениями → конвертация в A12.docx через Python-шаблон `docx_template.py`). ~50 проверок через Вебмастер, Метрику, Keyso, Арсенкин, fetch.

Развилки при переносе:

1. **Куда складывать артефакты.** Уже есть `articles/`, `strategies/`, `analyses/`, `structures/`, `metatags/`. Техаудит - ещё один тип задачи (один комплект документов на запуск).
2. **Переиспользовать ли агентов `/seo-analiz`?** У него тоже есть «разведка» и MCP-тяжёлая работа.
3. **Источник истины для отчёта.** A12 заметно более структурирован, чем A2 (`/seo-analiz`): таблицы проблем с автоподсчётом, чеклист с перекрёстными ссылками на приложения, сами приложения с блоками кода / diff / таблицами. Парсить такое из markdown в .docx (как `build-analysis-docx.mjs`) - хрупко.
4. **Шаблон .docx - Python.** Исходник конвертирует через `python-docx` (`templates/docx_template.py`), а проект - на Node (`docx`).
5. **Активация.** В исходнике - триггер `ПРОМТ-ТЕХАУДИТ` + ручная навигация `дальше`/`шаг N`. Техдолг платформы.

## Решение

### 1. Новый тип задачи - `audits/NNN-<domain-slug>/`

Каждый запуск `/seo-tehaudit` создаёт `audits/NNN-<domain-slug>/`. Внутри: `meta.json` (state machine), 4 промежуточных JSON разведки, единый `audit_data.json`, финальные `A12.md` + `A12_<slug>.docx`, `share.json`.

Pre-commit hook не меняется (универсален к task-dir, см. ADR-003). Скил пишет `.claude/tmp/current-task.txt = audits/NNN-slug/` первым делом. Это **4-е повторение** паттерна «новый тип задачи» (после ADR-007 `strategies/`, ADR-009 `analyses/`, ADR-010 `structures/`, ADR-012 `metatags/`). Гайд `SKILL-ADAPTATION-GUIDE.md` прямо приводит `audits/NNN/` как канонический пример.

### 2. НЕ переиспользуем агентов `/seo-analiz`

Создаём 5 новых агентов: `audit-recon`, `audit-indexing`, `audit-onpage`, `audit-analytics`, `audit-writer`. Причины:

- **Другой домен.** `/seo-analiz` изучает рынок/конкурентов (Keyso + SERP). Техаудит изучает **технику собственного сайта клиента** через его же доступы (Вебмастер + Метрика). Это разные кластеры MCP и разные выходные схемы.
- **Доступы клиента.** Аудит запрашивает Вебмастер/Метрику клиента (`wm_*`, `ym_*`), которых `/seo-analiz` принципиально не трогает (предпроектный этап). Поэтому это пост-онбординговая услуга (трек Б, после `/seo-shablon`).
- **Граф деления 1:1 с фазами SEO-аудита.** recon → indexing → onpage → analytics - классические независимые фазы техаудита; натянуть на них агентов конкурентного анализа = ветки if-else по «откуда вызвали», что нарушает контракт «вход → выход».

Деление на 4 сборщика + 1 writer - повтор устоявшейся формы `/seo-analiz` (brief/competitor/serp/leader + analysis-writer).

### 3. `audit_data.json` - единый структурированный источник истины; двойной рендер

`audit-writer` собирает **только** `audit_data.json` (вся творческая работа: дедуп, приоритизация, автоподсчёт, чеклист, генерация приложений). Из него - **два детерминированных рендера** скриптами:

- `render-audit-md.mjs` → `A12.md` (markdown для репо/handoff/будущих услуг);
- `build-audit-docx.mjs` → `A12_<slug>.docx` (клиентский документ, дизайн TIMUR SEO).

Плюс `verify-audit.mjs` - механическая проверка контракта `audit_data.json` (счётчики = длины массивов, каждая ссылка `appendix: N` в чеклисте указывает на реальное приложение, нет открытых плейсхолдеров `{...}`).

Это **осознанное отступление** от `/seo-analiz` (где агент пишет A2.md, а docx парсит markdown). Обоснование:

- Документ A12 структурно богаче (приложения с кодом/diff/таблицами рендерятся надёжно только из структуры, не из распарсенного markdown).
- Единый источник → нет рассинхрона между .md и .docx.
- Схема `audit_data.json` уже спроектирована в исходнике (шаг 6, словарь `data`) - переиспользуем готовый контракт.
- Творчество (что 🔴 vs 🟡, какие приложения) остаётся в агенте; шаблонизация (.md/.docx) - в скриптах. Это ровно разделение слоёв из `SKILL-ADAPTATION-GUIDE` («антипаттерн - html-builder субагент»).
- §5.6 «финальная самопроверка отчёта» из исходника становится настоящим детерминированным `verify-audit.mjs`, а не только промтом.

### 4. Порт `docx_template.py` → `build-audit-docx.mjs` (Python → Node)

Дизайн-токены TIMUR SEO (COLORS/FONT/SIZES) и сборщики секций переносятся на `docx` (npm), по образцу `build-analysis-docx.mjs` / `build-strategy-docx.mjs`. Это повтор решения ADR-007 (портировать Python-шаблоны на Node, не тянуть `python-docx` в стек). Сам `docx_template.py` в репо не входит - только как референс.

### 5. Хуки и settings.json - без изменений

Переиспользуем generic `check-file.sh` (SubagentStop `*` уже ловит всех новых агентов через маркер expected-файла) и `update-meta.sh`. Новых хуков 0, `settings.json` не трогается. Творческие самопроверки исходника → промты агентов; механические → `verify-audit.mjs`.

### 6. Заливка в Drive + цикл правок клиента

A12.docx - клиентский деливери, поэтому он заливается в Drive (Google Doc) и проходит цикл `client-review → revising → approved`, как `/seo-analiz`, `/seo-struktura`, `/seo-metategi`. Helper `/share-audit` для перезаливки. Нужен `audits_folder_id` в глобальном `~/.claude/seo-knowledge/DRIVE.md` (папка создаётся вручную один раз). Если ID нет - скил деградирует мягко (собирает A12.md+docx локально, подсказывает добавить ID), не падает.

### 7. Активация - `/seo-tehaudit <domain>`, без триггер-фраз

Скил вызывается `/seo-tehaudit <domain> [--resume] [--no-share]` и проходит все шаги сам через state machine. `ПРОМТ-ТЕХАУДИТ`, `дальше`, `шаг N`, `present_files`, генерация `/tmp/make_a12.py` - исчезают как класс (техдолг claude.ai).

### 8. Параллелизм сбора; Арсенкин - строго последовательно

`recon → indexing` последовательны (host_id, база Keyso, список URL и доноров - вход для следующих). `onpage` и `analytics` **независимы** (оба читают только `recon.json` + `indexing.json`) → оркестратор диспатчит их одним параллельным заходом. State machine остаётся линейной для чистого `--resume`; агенты идемпотентны (пропускают шаг, если их JSON уже есть).

**Арсенкин (`arsenkin_domains`) не работает при параллельных вызовах.** Его использует **только** `audit-recon` (один последовательный вызов whois на возраст домена), который выполняется отдельно и до параллельной пары. Параллельная пара `onpage ∥ analytics` Арсенкин не трогает. Правило зафиксировано в `MCP_MAP.md` и промте `audit-recon`: arsenkin_domains - один вызов, никогда не в параллель.

### 9. On-page аудит - шардами (масштабирование охвата страниц)

Шаг 3 (URL/мета/Schema) разнесён на три части ради охвата многих страниц без раздувания одного контекста (изначально было 8-12 страниц в одном агенте):

- **Выборка + url_structure** - детерминированный скрипт `select-audit-pages.mjs` (типизация страниц из `indexing.sitemap.all_urls`, round-robin по типам до `--pages N`, шардинг на батчи по 8; ЧПУ/глубина/длина/мультислеш как строковые проверки). Без MCP.
- **Per-page извлечение** - K параллельных `audit-onpage` (по батчу каждый, свежий контекст) пишут `onpage_<k>.json`. Запускаются в один заход вместе с `audit-analytics`.
- **Межстраничное** (Title-заглушка >=50%, дубли Title/H1 между батчами, `schema_summary`) - скрипт `merge-onpage.mjs` по всем шардам -> `onpage.json` (та же схема, что читает `audit-writer`).

Творческое (парсинг messy HTML) - в шардах; детерминированное (отбор, строковые проверки URL, межстраничная математика) - в скриптах. Дефолт 24 страницы, масштаб `--pages` до 80. Цена - потеря Keyso-приоритета выборки (отбор структурный по типам из sitemap, не по трафику); для техаудита охват типов важнее трафик-приоритета. `audit-onpage` из единого агента стал per-batch воркером.

## Альтернативы

- **Вписать в `analyses/`.** Отвергнуто: предпроектный анализ (рынок, без доступов) ≠ техаудит (своя техника, с доступами клиента). Один клиент может иметь оба.
- **Переиспользовать `/seo-analiz`-агентов с веткой.** Отвергнуто: разные MCP-кластеры и схемы; ветки if-else ломают контракт агента.
- **Агент пишет A12.md, docx парсит markdown (как `/seo-analiz`).** Отвергнуто как основной путь (см. Решение 3): хрупкий рендер приложений, сложная сверка счётчиков. Оставлено в уме как fallback, если структурный путь окажется тяжелее в поддержке.
- **Слить шаги 1-4 в один «collector»-агент.** Отвергнуто: разные MCP-кластеры, свежий контекст ценен для «самого объёмного» шага 2, карточка после шага 1 - точка показа человеку.
- **Тянуть `python-docx`.** Отвергнуто: в проекте нет Python-стека для генерации, всё на Node (`docx`).

## Последствия

**Хорошо:**
- `/seo-tehaudit` встраивается без изменений pre-commit, `/handoff`, `/handoff-process`, `settings.json`. Подтверждает многозадачность (ADR-001).
- Можно параллельно вести `articles/`, `analyses/`, `structures/`, `metatags/`, `audits/` в разных worktree.
- A12.md машиночитаем для будущих услуг (повторный аудит, дев-приёмка).
- Паттерн «новый тип задачи + N агентов + скрипты» применён 4-й раз - устоявшийся способ расширения.

**Плохо:**
- 5 новых агентов (всего 35) - рост числа. Митигация: чёткие контракты «вход → выход», список агентов в README обновлён.
- `build-audit-docx.mjs` - 3-й похожий docx-генератор (после strategy/analysis). Общий слой пока не выносим (преждевременная абстракция); если появится 4-й - вынести `_docx-helpers.mjs`.
- Двойной рендер (md + docx из одного JSON) - чуть больше скриптов, чем у `/seo-analiz`. Митигация: оба рендера тонкие и детерминированные, покрыты `verify-audit.mjs`.
- Аудит зависит от доступов клиента (Вебмастер/Метрика). При их отсутствии часть проверок пропускается - заложена graceful-degradation на уровне промтов агентов и раздел «Не удалось проверить» в отчёте.
