# ADR-028: Intake с провенансом, вопросы заказчику в A2, импорт ответов из Google Doc

**Статус:** Принято

**Дата:** 2026-07-16

## Контекст

`/seo-analiz` вычитывал всю вводную фактуру (бриф, транскрибации, файлы,
ЗАКАЗЧИК.md) прямо в контекст оркестратора - это и распухание контекста, и
отсутствие провенанса: по готовому A2 нельзя было сказать, какой факт откуда взят.
При этом у анализа не было структурного способа задать заказчику открытые вопросы
(варианты «либо так, либо так»), а ответы возвращались свободным текстом без пути
для импорта обратно в анализ.

## Решение

**Отдельный intake-агент с провенансом + раздел вопросов в A2 + импорт ответов из
Google Doc.**

- **Агент intake-analyst (opus, без MCP, шаг 1.5, state `intake-done`):** вся
  вводная фактура -> `intake.json` (факты с `source` + `quote`, `decision_impact`)
  + `ВВОДНЫЕ.md`. Оркестратор фактуру в свой контекст НЕ читает - передает пути.
- **brief-structurer поверх `intake.json`** (fallback `brief_raw.txt` для legacy);
  `brief.json` остается плоским - **контракт `/seo-struktura` не тронут**.
- **Редизайн A2:** новый раздел «0. Вопросы к вам» ДО Executive Summary - 3-7
  вопросов с вариантами а/б/в и рекомендацией (в docx рекомендованный подсвечен);
  `questions.json` - единый источник; раздел 1.7 = указатель на раздел 0.
- **Режим `--answers`:** автопоиск анализа -> `readGoogleDoc` -> `client_doc.md`
  -> агент answer-extractor (sonnet) -> `apply-answers.mjs` (classifyAnswer:
  «согласен с рекомендованным» = none) -> `rerun_plan.json` -> `revising`; fallback
  ручной вставки сохранен.
- **Агент analysis-verifier (opus, шаг 6b, state `analysis-verified`)** -
  финальная сверка A2 (см. [ADR-025](025-final-verifiers.md)).
- **`validate-analysis-inputs`** зовется в конце скила + warn-only проверка
  intake/questions (блок-контракт не тронут, legacy проходит).
- **MCP_MAP:** снят ложный запрет Drive, добавлен `readGoogleDoc`.

## Альтернативы

- **Читать фактуру в оркестраторе.** Отвергнуто: распухание контекста и нет
  провенанса; агент + пути держат контекст ровным, а каждый факт - привязанным к
  `source`+`quote`.
- **Ответы клиента свободным текстом.** Отвергнуто: нет надежного импорта;
  структурный `questions.json` + круг через Google Doc + extractor дают
  детерминированный путь применения.
- **Поменять форму `brief.json`.** Отвергнуто: сломало бы `/seo-struktura`; intake
  надстроен сверху, `brief.json` остается плоским.

## Последствия

**Хорошо:** провенанс у каждого факта intake; контекст оркестратора ровный;
вопросы заказчику - первоклассная сущность с обраткой из Google Doc; контракт
нижележащих скилов цел.
**Плохо:** больше агентов/скриптов (intake-analyst, answer-extractor,
`_questions.mjs`, `apply-answers.mjs`); риск двух источников (`intake.json` +
legacy `brief_raw`) - митигирован fallback'ом и warn-only валидацией.
