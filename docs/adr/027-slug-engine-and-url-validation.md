# ADR-027: Единый slug-движок и валидация URL структуры

**Статус:** Принято

**Дата:** 2026-07-16

## Контекст

Логика построения URL/slug была размазана по нескольким скриптам
(`build-structure-xlsx`, `select-top10`) с дублирующимися картами транслита, и в
одной из копий жил боевой баг: URL получались гигантскими - из склеенных
синонимов маркерного запроса и комментариев в скобках. Вдобавок structure-writer
**сам генерировал URL** (LLM-печать), что недетерминированно и трудно проверяемо,
и грозило дрейфом между JSON-данными и таблицей A6.

## Решение

**Один модуль `_slug.mjs` на весь конвейер + валидация «Адрес страницы» на входе
и на выходе; writer больше не генерит URL.**

- **`_slug.mjs`:** одна карта транслита; `buildPageUrl` с приоритетом источника
  (client_current_url -> migration_target_url -> маркерный запрос -> name без
  скобок), стоп-словами, ограничением не длиннее 5 слов / 60 символов по границе
  слова и разведением коллизий; `validateUrl`. Это чинит баг гигантских URL в
  одном месте.
- **`slugifyBase` бит-в-бит эквивалентен старому `slugifyId`** - id в
  `decisions.json` не сдвигаются (сверено ревьюером против старой реализации,
  закреплено тестом-стражем).
- **`build-structure-xlsx` и `select-top10` переведены на модуль** - дублирующие
  карты убраны.
- **structure-writer НЕ генерит URL** - копирует из `structure_data.json`.
- **`import-structure` валидирует колонку «Адрес страницы»** (латиница/длина/
  скобки); нарушения выводятся в `exit 3` отдельной секцией, молча не чинятся.
- **`verify-structure.mjs` (шаг 9г)** - правила URL, полнота A6 против JSON, дубли
  маркеров; клиентский кириллический URL = warn (уважаем ввод), новый = блок. Плюс
  агент structure-verifier (шаг 9д, см. [ADR-025](025-final-verifiers.md)).

## Альтернативы

- **Оставить slug-логику в каждом скрипте.** Отвергнуто: дублирующие карты
  дрейфуют, баг гигантских URL жил ровно в одной копии.
- **Оставить генерацию URL за structure-writer.** Отвергнуто: LLM-генерация URL
  недетерминированна и плохо валидируется; детерминированный модуль + копирование
  writer'ом надежнее.
- **Молча чинить кривые URL на импорте.** Отвергнуто: прячет правки клиента/
  человека; лучше поднять их в `exit 3` и показать.

## Последствия

**Хорошо:** одна правда про URL, детерминированно и тестируемо (регрессия на баг
гигантских URL, страж бит-в-бит id); импорт валидируется; клиентский кириллический
URL = warn, новый = блок.
**Плохо:** writer теряет способность (обязан копировать) - но это и было целью;
модуль обязан оставаться бит-в-бит совместимым по id с legacy (держит тест-страж).
