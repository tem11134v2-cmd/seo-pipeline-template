# ADR-019: Синхронизация машинерии в существующие клоны (sync-from-template + sync-all)

**Статус:** Принято

**Дата:** 2026-06-08

## Контекст

Шаблон распространяется как GitHub-репо, новые проекты создаются `git clone` (ADR-006). Но ADR-006 в разделе «Плохо» оставил нерешённым: **существующие клиенты не получают обновления автоматически**, а `git pull origin main` небезопасен (затронет клиентские файлы и историю). Предложено «на будущее» - скил `/update-template`, аккуратно переносящий только машинерию.

К моменту принятия этого ADR накопилось 19+ клиентских клонов, разъехавшихся по версиям машинерии (часть - ещё с дорусификационными именами скилов). Болевые точки:
- единственная «механическая» операция без детерминированного скрипта - синк делался руками по инструкции (`git diff` + copy + commit «на глаз»), без атомарности и отката;
- два источника истины: `new-project` клонирует с GitHub, а синк копировал из локального `template-project` - расходятся, если локальный не запушен;
- нет версионной метки - не ответить «кто отстал» (HEAD клиента бесполезен, у каждого своя история задач);
- раскатка по N проектам - вручную, по одному;
- обновление, требующее починки данных (а не только копирования файла), не выполнялось (напр. вывод `_index.json` из индекса, ADR-013, делался вручную).

## Решение

Один детерминированный **движок** + тонкие слои поверх.

- **Движок** `.claude/scripts/sync-from-template.mjs` - точное зеркало машинерии: `.claude/{scripts,agents,skills,hooks,git-hooks,migrations,tests}` + `package.json` + `.gitignore` + `.claude/settings.json`. Флаги `--template/--target/--apply/--json/--no-delete/--no-commit/--no-migrations/--force`. Dry-run по умолчанию, точный отчёт +/~/-. Проверки: target в main (не worktree), не самосинк, чистота дерева машинерии, сверка шаблона с его origin (мягкое предупреждение). Пишет `.claude/.machinery-version`, прогоняет невыполненные миграции, сам коммитит с фиксацией prev HEAD для отката. `--target` позволяет навести движок из шаблона на любой проект (снимает «курицу и яйцо» при первой раскатке).
- **Один проект:** скил `/sync-from-template` (main клиента) - тонкая обёртка.
- **Все проекты:** родительский скил/скрипт `/sync-all` (`~/seo-projects/.claude/scripts/sync-all.mjs`) - матрица «кто отстал», `--apply` массово; грязные деревья и worktree-таргеты движок отклоняет, они попадают в «Пропущенные».
- **Версионирование:** `.claude/.machinery-version` (коммитится) - commit шаблона, на котором собрана машинерия. Пишется при `new-project` (клон) и каждом синке. `/status` показывает отставание + счётчик.
- **Миграции данных:** `.claude/migrations/NNN-*.mjs` (идемпотентные) + журнал `.claude/.migrations-applied.json`. Прогоняются после копирования файлов, до коммита. Первая - `001-index-json-cache` (закрывает ручной шаг из ADR-013).

Тест движка: `.claude/tests/sync/run.mjs` (mirror, версия, миграция, `.gitignore`-регресс, идемпотентность, no-delete, самосинк).

## Альтернативы

- **`git pull origin main` в клиенте** (как намекал ADR-006). Отвергнуто: тянет всю историю шаблона в клиентский main, мерж-конфликты с клиентской работой, нет выборочности «только машинерия».
- **Тянуть напрямую из origin (git archive/checkout по файлам)** вместо локального `template-project`. Отложено: надёжнее против рассинхрона, но нужна сеть/доступ; вместо этого движок сверяет локальный шаблон с его origin и предупреждает, если тот отстал/не запушен. Оставлено опцией на будущее (`--from-origin`).
- **Merge-движок** (сливать клиентские правки внутри машинерии). Отвергнуто: машинерия read-only у клиентов и не кастомится (ADR-001), поэтому достаточно зеркала. Если клиент всё же кастомил - dry-run покажет, решаем вручную (движок не превращается в merge-движок).
- **Расширить `/status` вместо отдельного `/sync-all`.** `/status` оставлен только показывать версию; раскатка - отдельная команда (это действие по 19+ репо, ему нужна своя поверхность).

## Последствия

- Обновление существующих клонов из «ручного glob-copy на один проект» стало детерминированным, версионируемым, ревертабельным и массовым. Раскрыт пробел ADR-006.
- **`.gitignore` обязан входить в синкаемый набор** (находка пилота): без актуальных правил (напр. `_index.json`) миграция `git rm --cached` откатывалась назад при стейджинге - файл снова попадал в индекс как untracked. Аналогично в набор вошли `tests` (движок советует их прогнать после синка) и `.claude/settings.json` (конфиг хуков).
- Коммит синка берёт только изменения, появившиеся **в результате** синка (diff от снимка «грязного до старта»), поэтому незакоммиченная клиентская работа не затрагивается.
- **Клиенты не пушат** - у них origin = сам шаблон; синк коммитит только локально. Раскатка origin-шаблона - отдельный явный `git push` в `template-project`.
- `CLAUDE.md` сознательно не синкается авто (может содержать клиентские пометки) - движок только сигналит о расхождении.
- **Не покрыто:** родительские файлы (`~/seo-projects/.claude/`: `guide`, `new-project`, `status`, `sync-all`, `CLAUDE.md`) - движок синкает шаблон -> клиенты, а не в родителя; дубль `guide`/`CLAUDE.md` parent<->template сводится вручную. Кандидат на отдельный ADR (тонкие указатели или verify-дифф).
