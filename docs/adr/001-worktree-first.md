# ADR-001: Все задачи в отдельных worktree-сессиях

**Статус:** Принято

**Дата:** 2026-05-24

## Контекст

Изначально (версия для claude.ai) одна тема = одна длинная сессия копи-паста между чатами. При переносе в Claude Code Desktop появилась возможность реальной параллели — несколько сессий одновременно. Но возникли проблемы:

1. Две сессии в одной папке проекта **толкаются за общие служебные файлы** (`.claude/tmp/current-task.txt`, который читают хуки). Хук check-section будет смотреть «не на ту» статью.
2. Две сессии могут одновременно править `topics.xlsx` или `ЗАКАЗЧИК.md` — будут конфликты на диске и непредсказуемое поведение.
3. Если одна сессия что-то сломала в общих файлах — нет простого способа откатить только её правки.

Архитектурный выбор: где проводить границу изоляции?

## Решение

**Каждая задача (статья, аудит, генерация тем, исследование сайта) запускается в отдельной git worktree.** Единственная команда, которая работает в основной папке (main) — `/handoff-process`, она применяет накопленные результаты worktree-задач к общим файлам.

Внутри worktree разрешено менять только:
- Свою папку задачи (`articles/NNN/`)
- `.claude/tmp/` (служебные файлы сессии)
- `.claude/handoff-requests/` (запросы для main)

Защита — pre-commit hook (см. [ADR-003](003-pre-commit-whitelist.md)).

## Альтернативы

### A. Без worktree, всё в одной папке

Простейшая модель: запретить параллель в одном проекте, для параллели использовать разные проекты клиентов. Отвергнуто: пользователь явно сказал, что в будущем проект клиента содержит не только статьи, но и аудиты, коммерческие тексты — нужна реальная многозадачность внутри одного клиента.

### B. Изоляция через session_id в хуках

Каждой сессии — свой подкаталог `.claude/tmp/<session-id>/`, хуки определяют активную задачу по идентификатору сессии. Отвергнуто: требует переписывания хуков под чтение session_id из stdin Claude Code, не решает проблему конкуренции за `topics.xlsx` / `ЗАКАЗЧИК.md`.

### C. Lock-файлы на уровне статьи

`articles/NNN/.lock` блокирует папку. Отвергнуто: защищает только per-task папки, не общие файлы.

### D. Worktree только при явной потребности

Можно делать worktree, можно не делать. Решает пользователь каждый раз. Отвергнуто: создаёт смесь режимов (часть работы в worktree, часть в main), пользователь не помнит, где какие файлы — каша.

## Последствия

**Хорошо:**
- Параллель работает без конфликтов (math: разные worktree касаются разных папок, git merge всегда чистый).
- Откат целиком: не понравилась задача — `git worktree remove --force`, основная папка чистая.
- Атомарная транзакция: либо вся задача попадает в main через `/handoff` + `/handoff-process`, либо ничего.

**Плохо:**
- Лишний шаг для пользователя: после каждой задачи — `/handoff` (и при необходимости `/handoff-process` в main).
- Нужно понимать механику git worktree (хотя бы поверхностно).
- На Windows иногда `git worktree remove` упирается в file lock — нужен fallback через `prune`.
