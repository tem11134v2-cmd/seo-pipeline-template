# ADR-017: Доказательный подбор блоков (скан лидеров) + поддержка каталожных сайтов

**Статус:** Принято

**Дата:** 2026-06-06

## Контекст

ADR-015 (`/seo-tekst`) подбирал блоки по **статической** `BLOCKS.md` (матрица применимости «тип страницы × блок»). При портировании авторский шаг У5-В (competitor-driven подбор: фетч лидеров -> матрица покрытия -> «покрытие >= 2/3 = блок обязателен») был свёрнут в эту статику с пометкой «competitor fetch опционален».

На реальном прогоне (НЕФТЕХИМСЕРВИС, B2B-каталог) и в обсуждении вскрылись два пробела:
1. **Каталожные сайты** (e-commerce / B2B-каталог) имеют свои типы страниц (Главная-каталог, Категория, Карточка товара) с особой композицией - статическая матрица «сайта услуг» их не покрывает, а kit-фрагментов под листинг/фильтры/галерею не было.
2. **Набор «почти-все-используют»-блоков меняется от ниши к нише.** Универсальная матрица это усредняет и теряет нишевую специфику.

Гипотеза заказчика: смотреть **живьём через Chrome-плагин** 3-6 лучших лидеров (из структуры/анализа), снимать реальную композицию блоков по типам страниц и фишки - и для услуг, и (особенно) для каталогов.

## Решение

### 1. Звено `leader-block-scanner` (проектного уровня, в `/seo-tekst`)

Новый агент сканирует 3-6 лидеров по типам страниц и строит `leader_blocks.json` - **матрицу покрытия «блок × тип страницы»** (`coverage_pct`, `typical_order`) + `features_to_steal` (калькулятор/фильтр/конфигуратор/сравнение). Это восстановленный У5-В, но rendered-точный.

- **Chrome-плагин PRIMARY** (`mcp__Claude_in_Chrome__*`): видит rendered-композицию, lazy-load, интерактив - критично для SPA-каталогов, которые `mcp_fetch_page` почти не видит.
- **`mcp_fetch_page` FALLBACK**: сырой HTML, если Chrome не подключён.
- **Статическая матрица `BLOCKS.md` - ПОЛ**: совсем нет данных -> `page-writer` падает на неё. Деградирует изящно, не блокирует.
- **Проектный уровень, один скан, кешируется** в `texts/NNN/leader_blocks.json` - не трогает быстрый per-page веер. Флаг `--scan-leaders`/`--no-scan` (по умолчанию ВКЛ при наличии лидеров).

### 2. Доказательный подбор блоков в `page-writer`

При наличии `leader_blocks.json`: блоки с покрытием `>= 50%` лидеров для типа страницы = ОБЯЗАТЕЛЬНЫЕ (в `typical_order`); статическая матрица = пол/добор. Нет файла -> чисто по матрице. `features_to_steal` идут как идеи для блоков/CTA.

### 3. Поддержка каталожных сайтов (kit + методология)

- **`BLOCKS.md`**: секция «Каталожные сайты» - типы (Главная-каталог, Категория, Подкатегория, Карточка товара, Бренд), каталожные блоки (хлебные крошки, плитка категорий, листинг товаров + фильтры, галерея товара, характеристики, похожие/популярные, подкатегории) с метриками + рецепты по типам.
- **kit-фрагменты (5 новых):** `breadcrumbs`, `category-grid`, `product-listing` (фильтр-рейл + сетка товаров, вложенный repeat `filters[].options`), `product-gallery`, `subcategory-tiles` + их CSS в `prototype.css` (токены, responsive) + маппинг каталожных block-types в `fragments-manifest.json`. Рендерятся статичными прототип-представлениями (плейсхолдер-карточки, мокап фильтра) - прототип, не живой каталог.

### 4. Живёт в `/seo-tekst`, не в `/seo-struktura`

Подбор блоков (из чего собрать страницу) - забота ТЕКСТОВ; нужен финальный `pages.json`; скан близко к потреблению. `/seo-struktura` отвечает за СПИСОК страниц (какие есть), `/seo-tekst` - за КОМПОЗИЦИЮ (из каких блоков). Кеш в `texts/NNN/`.

## Альтернативы
- **A. Только статическая матрица** (как было). Отвергнуто: не покрывает каталоги + нишевую вариативность блоков.
- **B. Скан в `/seo-struktura`** (там сходятся лидеры + типы страниц). Рассматривалось; отложено - composition это texts-concern, и скан нужен финальному pages.json. Можно «поднять» позже отдельным решением.
- **C. Расширить `leader_scan.json` в `/seo-analiz`**. Отвергнуто: анализ идёт ДО финализации типов страниц (их даёт структура); и `leader_scan` - это посылы/фишки (Э2-лайт), не по-типовая матрица блоков.
- **D. Только `mcp_fetch_page` без Chrome.** Отвергнуто: фетч слеп к SPA-каталогам и интерактиву (фишки). Chrome PRIMARY + fetch FALLBACK.

## Последствия

**Хорошо:**
- Подбор блоков из предписательного стал доказательным (что реально у лидеров ниши), особенно ценно для каталогов и нетиповых ниш.
- Каталожные сайты теперь собираются в прототип (5 фрагментов + рецепты + типы страниц).
- Скан проектного уровня + кеш + опционален -> не ломает быстрый веер; деградирует Chrome -> fetch -> матрица.
- Chrome видит фишки-дифференциаторы (калькуляторы/фильтры/конфигураторы), которые становятся идеями для прототипа.

**Плохо:**
- **Зависимость от Chrome-плагина** (расширение подключено, логин, антибот). Митигация: fetch-фолбэк + матрица-пол; скан опционален.
- Скан 3-6 лидеров × N типов = время (9-18 загрузок). Митигация: проектный уровень + кеш + `--no-scan`.
- Точность авто-распознавания блоков по rendered-странице - эвристика. Митигация: маппинг в готовый словарь `block_to_fragment`, нет данных -> не выдумывать.
- Каталог-фрагменты рендерят статичные представления (не живой листинг/фильтр) - но это и есть прототип.
