# Точки роста алгоритма `/write-article`

**Источник:** разбор после прогона статьи 001 «Мужской костюм на свадьбу 2026» в worktree `claude/charming-lumiere-b1242a`, 2026-05-28.

**Цель документа:** собрать в одном месте всё, что нужно поправить или улучшить в конвейере написания статей. Здесь и баги, и UX-улучшения, и стратегические сдвиги. После handoff-process этот файл можно сдвинуть в `docs/` как постоянную дорожную карту — это roadmap, а не одна правка.

**Главная цель алгоритма после доработок:** запуск `/write-article N` от старта до готового пакета файлов (docx в GDrive + html + tilda при нужной платформе) без пауз и без ручных правок. Не только в `--auto`, а **базовый режим** — самодостаточный. Паузы — только опционально через флаг для тщательной отработки.

## Раздел 1. Реальные баги из прогона (must fix)

Эти проблемы воспроизводились в живой задаче — без их починки самодостаточный режим невозможен в принципе.

### 1.1 `article-finalizer` молча теряет `[ФОТО:]` и `[ТАБЛИЦА:]` метки

**Симптом.** В `tz.md` + промтах photo-promter заложено 4 фото для разделов 1, 2, 5, 7. После `article-finalizer` в `article.md` оказалась только одна метка `[ФОТО:]` — в разделе 1. Остальные три были утрачены при склейке `sections/01..09.md` → `article.md`. То же опасение для `[ТАБЛИЦА:]` и `[ИКОНКИ:]` если они стоят в неожиданных местах.

**Эффект.** Мы платим Nano Banana ~$0.07 × 4 = $0.28 за фото, а пять из шести не имеют куда вставиться в HTML. Photo-promter тоже зря работает.

**Что чинить.** Логика финализатора при склейке секций должна:
1. Сохранить порядок и точные позиции всех меток (`[ФОТО:]`, `[ТАБЛИЦА:]`, `[ДИАГРАММА:]`, `[ЦИТАТА:]`, `[ИКОНКИ:]`, `[ТАБЫ:]`) из `sections/*.md`.
2. Если в ТЗ был запланирован элемент, который в секции не появился — это warning, не silent drop. Финализатор должен либо доставить метку в нужное место, либо явно зафиксировать в `report.md` под «Расхождения с ТЗ».
3. Перепроверить число `[ФОТО:]` в финале против ожидания photo-promter и согласовать (вернее ожидание из ТЗ, photo-promter обслуживает уже расставленные метки).

### 1.2 `section-writer` галлюцинирует «проблему UTF-8» и пишет ASCII

**Симптом.** Дважды за прогон агент решил, что check-section.sh жалуется на кириллицу + локаль, и переписал раздел латиницей/транслитом «для диагностики». Раз — пустой «## Test», второй раз — целый раздел латиницей.

**Эффект.** Раздел приходится удалять вручную и перезапускать агента с жёстким стоп-сигналом.

**Что чинить.** В системный промт `section-writer` добавить блок-запрет:

> Никогда не пиши latin/ASCII/Test-файлы для «диагностики хука». Если check-section.sh вернул exit 2 — отчитайся в чат точным stderr, не записывай файл повторно. Никаких транслитов, никаких пробных вариантов. Кириллица в файлах работает, это проверено.

Также я в каждом промте этот пункт повторял руками — надо унести в системный промт агента.

### 1.3 `assemble-html.mjs` — три бага в одном

Подробно описаны в отдельном handoff-request `fix-assemble-html-script.md` (тот же worktree). Кратко:

- **Bug 1.** Плейсхолдеры `___NX_PLACEHOLDER_*___` интерпретируются markdown как «жирный курсив», подмена не срабатывает. Все таблицы, иконки и фото остаются как `<em><strong>NX_PLACEHOLDER_PHOTO_1</strong></em>`.
- **Bug 2.** Поиск маркера `<!-- CONTENT -->` строкой не находит реальный `<!-- CONTENT - основной текст статьи -->`. Fallback клеит контент к концу `<body>` и demo-блок остаётся.
- **Bug 3.** Если делать «вырезаем всё body» — теряются обёрточные классы `nx-article`, breadcrumbs/h1/TOC/FAQ ломаются стилистически.

Рабочий скрипт с правильной стратегией (точечные подмены внутри template + плейсхолдеры через HTML-комментарии) лежит в worktree как `.claude/tmp/assemble-local.mjs`. После handoff его можно скопировать в основной скрипт.

### 1.4 `text-auditor` находит, но `/edit` применяет по одной

**Симптом.** Аудит выдал 30 пунктов (3 критичных + 15 важных + 12 косметических). Применять по одной через `article-fixer` — это 30 вызовов агента. Я в прогоне делал через Edit-ы напрямую, но это в обход алгоритма.

**Что чинить.** Нужен **`article-fixer-batch`** агент: принимает `audit.md` и применяет все правки разом за один проход, возвращает дифф + список «не смог применить». Косметика — отдельным проходом по флагу `--cosmetics`.

### 1.5 API 529 Overloaded ломает прогон

**Симптом.** За прогон поймал 3 раза. Каждый раз нужно перезапускать `/write-article 1 --resume`.

**Что чинить.** Скил должен сам делать `ScheduleWakeup` (60-180с) и продолжать. Пользователь не должен видеть «упало → повторите».

## Раздел 2. Идеи пользователя — оценка и рекомендации

### 2.1 Две статьи на тему в разных жанрах + авто-проверка

**Вердикт: согласен, но реализовать через `topic_id`, а не slug.**

Сейчас проверка идёт по slug-у директории. Это хрупко: подправил заголовок темы — slug изменился — проверка не срабатывает. Правильная привязка — поле `№` из колонки topics.xlsx (это и есть topic_id, уникальный идентификатор темы).

**Что сделать:**

1. В `meta.json` каждой статьи писать:
   - `topic_id` (значение колонки № из topics.xlsx)
   - `genre` (уже есть)
   - `platform_target`: `"site"` | `"external"` | `"social"` — где статья будет публиковаться

2. Вести индекс `articles/_index.json` со списком всех статей: `[{nnn, topic_id, slug, genre, platform_target, state, completed_at}]`. Обновлять при каждом вызове `update-meta.sh`. Это даёт быструю проверку без сканирования 50 директорий.

3. Логика при повторном `/write-article N`:
   - Сканируем индекс на `topic_id == N`.
   - Нашли 0 — пишем как обычно (первый жанр из xlsx, platform = site).
   - Нашли 1 — показываем «уже сделана в жанре X на площадку Y. У темы в xlsx доступны жанры [Y, Z]. Какой брать? Или переписать существующую?»
   - Нашли все (= число жанров в колонке) — «по теме всё закрыто. Варианты: переписать одну из существующих / следующую тему / отмена».

4. Жанр на повторе берётся из колонки «Жанры (2-3)» как контрастный к уже использованному (topic-generator обычно ставит контрастные).

### 2.2 Самодостаточный режим — основной, паузы опциональны

**Вердикт: поддерживаю, но дотюнить.**

Сейчас в скиле пять пауз: после JM, после ТЗ, после финализации, после аудита, после photo-promter. Из них реально нужна **одна** — после финализации (ручной контроль над текстом перед автоматизированными шагами).

**Что сделать:**

- Сделать `/write-article N` **самодостаточным по умолчанию** — без пауз вообще, с одним условным остановом после финализации (управляется флагом).
- Добавить флаг `--auto` (или `--silent`) который убирает даже эту последнюю паузу — для пакетной обработки.
- Добавить `--review` который ВКЛЮЧАЕТ ручную проверку (для ответственных текстов).

По шагам:
- JM-неудача → авто-retry один раз с другим стоп-доменом / другой формулировкой. После двух неудач — стоп с понятной ошибкой.
- ТЗ — собирается из JM, валидация структуры автоматическая (есть ли все обязательные разделы).
- Финализация — без паузы.
- Аудит — автоматически применяются крит + важные правки через `article-fixer-batch`. Косметика — лог в `audit-skipped.md` без блокировки.
- Photo-promter + Nano Banana + Cloudinary — без пауз (см. 2.5).
- Финал — отчёт в чат с тремя ссылками (docx/html/tilda).

Это даёт сценарий «нажал → через ~15-25 минут получил пакет файлов».

### 2.3 Финальные deliverables (А: docx на GDrive, Б: html, В: tilda)

**Вердикт: принимаю целиком.**

**A. DOCX на Google Drive с метатегами в шапке.**

Шапка docx (выделено цветом фона):

| Поле | Значение |

| Title | … |
| Description | … |

**Что реализовать:**

- Inline-картинки: скачать с Cloudinary по URL → embed как Image (docx-библиотека умеет).
- SVG-иконки из nx-icons-grid: либо рендерим в PNG через Cloudinary fetch+upload, либо заменяем на маркер `■` + текст.

**Б. HTML output.html.** Уже есть, чинится через handoff-request `fix-assemble-html-script.md`.

**В. Tilda split.** Уже есть, шаг 11 скила (`tilda-split.mjs`). У mansband_ru платформа Mosaic — поэтому не сработал. Автодетект по `ЗАКАЗЧИК.md` правильный, ничего менять не нужно.

### 2.4 Расширенные первичные вопросы (жанр + площадка)

**Вердикт: согласен частично. Спрашивать всегда — раздражает.**

Базовый запрос `/write-article N` должен идти без вопросов. Расширенный диалог — только при коллизии (см. 2.1) или явных флагах.

**Флаги:**
- `/write-article N` → первый жанр, platform = site. Без вопросов.
- `/write-article N --external` → второй жанр, platform = external.
- `/write-article N --genre="Личный опыт"` → явный выбор жанра.
- `/write-article N --platform=social` → подбор под соцсети (короче, другой тон).

Диалог нужен только когда:
- Тема уже сделана в этом жанре (см. 2.1).
- Запрошенный жанр не из колонки xlsx (предложить ближайший доступный).
- Флага нет, но в xlsx у темы помечено «требует выбора» (новая опциональная колонка).

### 2.5 Авто-генерация фото

**Вердикт: 100% поддерживаю, в этой сессии уже реализовали в обход.**

Состояние `awaiting-photos` нужно убрать из state machine целиком. После `enhancer`:

1. `photo-promter` собирает промты (как сейчас).
2. Сразу параллельно `nb2` через `mcp__openrouter-image__generate_image` (`aspect_ratio: 16:9`, `output_path` в `photos/NN-slug.png`).
3. Сразу публикация в Cloudinary через `mcp__cloudinary-publish__publish_image` с папкой `articles/YYYY/MM/<slug>`, тегами, alt-текстом.
4. Скил собирает `photos/urls.json` сам.
5. Идём в сборку HTML.

Падение генерации одного фото — retry один раз, потом TODO-маркер в этой позиции, остальные не блокируются. В meta.json фиксировать URL + `assetId` Cloudinary — для последующих ребилдов.

Стоимость: $0.07 × 5-8 = $0.35-0.56 за статью. Прозрачно.

### 2.6 5-10 фото вместо 4

**Вердикт: согласен по сути, но «10 на каждую» — перебор.**

«По фотке после каждого H2» = визуальный спам. Лучше правило **«1 фото на каждые ~300-400 слов основного текста, мин 4, макс 8»**, а tz-builder + photo-promter сами решают позиции по типам разделов:

- Hero (после intro) — всегда
- Коронный раздел — всегда
- Инструктивные разделы (сравнения, чек-листы) — да
- Обзорно-списочные разделы — нет (там таблица справится)
- Заключение — опционально CTA-визуал

Для гайда 2700 слов = ~5-7 фото. Для статьи-обзора «10 типов пиджаков» — 10 (по карточке на каждый), потому что там фото = смысловой контент, не иллюстрация.

**Что важно сначала починить:** баг 1.1 (article-finalizer теряет метки). Без этого хоть 4, хоть 10 — большинство уйдёт в никуда.

## Раздел 3. Дополнительные точки роста

Из прогона я заметил несколько вещей, которые в твоём списке не звучали, но дают заметный прирост качества.

### 3.1 Audit-fixer-batch (обязательный)

Уже в Разделе 1.4. Без него самодостаточный режим невозможен — иначе 30 кликов вручную на каждую статью.

### 3.2 Auto-retry на 529 / временные ошибки

Описано в 1.5. Скил делает `ScheduleWakeup` 60-180 секунд и продолжает. Никакого `--resume` от пользователя.

### 3.3 Дедуп перелинковки

**Симптом из прогона.** `/selection/wedding` повторяется в моей статье 3 раза. Это не криминал, но плохо для распределения веса ссылок: одна страница получает три внутренних линка, остальные 14 URL из ЗАКАЗЧИК.md — ноль.

**Что сделать.** В `tz-builder` (раздел перелинковки): запретить повтор URL чаще двух раз на статью. Если плановых ссылок 8 — раздавать на 8 разных URL из ЗАКАЗЧИК.md. Учитывать тематическую релевантность анкора и URL.

### 3.4 Баланс body и FAQ

**Симптом.** У меня body 2768 слов + FAQ 1700 слов = ~4500. Это перевешивает чтение, страница превращается в стену вопросов.

**Что сделать.** В `enhancer` (FAQ-генератор): для гайда лимит 6-8 вопросов по 60-80 слов = ~500 слов FAQ. Для коммерческих карточек / страниц услуг — можно 15+ (там длинный FAQ оправдан). Тип ограничения берётся из жанра в `meta.json`.

### 3.5 Социальная картинка для og:image

**Симптом.** Schema.org JSON-LD у нас сейчас без `image`. И в html нет `og:image`/`twitter:image`. При репосте статьи в мессенджер ничего не подгрузится.

**Что сделать.** При публикации hero-фото в Cloudinary запрашивать вариант `social` (1200×630, c_fill, g_auto) — Cloudinary это умеет одним параметром. URL социалки писать в `urls.json` отдельно от hero, и в template подставлять как:

```html
<meta property="og:image" content="…/social/01-trends.jpg">
<meta name="twitter:image" content="…/social/01-trends.jpg">
```

Schema.org Article → поле `image` тоже заполнять.

### 3.6 Метрики после сборки

**Что сделать.** В отчёт `report.md` добавлять:

- Слов в body / FAQ / итого
- Inline-ссылок: всего, уникальных URL
- Частота главного запроса и кластера (вхождения)
- Оценка читаемости (формула типа Flesch-Kincaid для русского — есть готовые формулы)
- Фото: число, средний размер, alt-coverage (% картинок с непустым alt)
- Schema.org: какие сущности на странице

Это даёт быстрый чек-сигнал «здорово / нездорово» без повторного `jm_text_analyze`.

### 3.7 Регенерация одного раздела

**Что сделать.** Команда `/rewrite-section <NNN> <N> "<описание>"` — пересобирает раздел N статьи NNN с новой постановкой задачи и пересчётом N-грамм. Это не fix (правка), это полный rewrite одного раздела. Полезно когда раздел не пошёл, а остальные ок.

### 3.8 JM-кеш между прогонами одной темы

**Симптом из прогона.** Если делать вторую статью на ту же тему в другом жанре в один день — `jm_text_analyze` (5 лимитов) попадает в кеш JM на 4 часа. А `jm_text_generate` (15 лимитов) — нет, выполняется заново. На 50 статей при двух жанрах = 750 «лишних» лимитов.

**Что сделать.** При повторном `/write-article N` (другой жанр, та же тема) — спрашивать «использовать существующий jm/ из статьи NNN?». Если да — копируем cluster.json / lsi.json / analyze.json из существующей статьи в новую. Экономия 20 лимитов на повторе.

### 3.9 Стоп-сигнал section-writer от галлюцинаций

Уже в Разделе 1.2.

### 3.10 Лимит на тире и стоп-слова — на уровне check-section

**Симптом из прогона.** В разделе 7 проскочило стоп-слово «распродажи». Хук check-section.sh проверяет тире, но стоп-слова бренда не проверяет.

**Что сделать.** В `check-section.sh` добавить проверку стоп-слов бренда из `ЗАКАЗЧИК.md` секция «Бренд → Стоп-слова». Если найдено — exit 2 + список с номерами строк. Stop-words автозагрузка из ЗАКАЗЧИК.md делает хук универсальным для разных клиентов.

### 3.11 Парсер ЗАКАЗЧИК.md в один общий модуль

**Симптом.** Сейчас разные скрипты по-разному парсят ЗАКАЗЧИК.md (таблицы / bullet'ы / различные label'ы). У меня assemble-html.mjs имеет свою функцию `pickClientField`, у других скриптов своя.

**Что сделать.** Общий модуль `.claude/scripts/_client.mjs` — экспортирует:
- `parseClient()` → объект с полями (domain, region, brand_name, stop_words, link_pool, author, platform, etc.)
- `getLinkPool()` → массив { url, anchor, category }
- `getBrandFacts()` → строка с фактами для финализатора

Все остальные скрипты пользуются им. Меняется ЗАКАЗЧИК.md — пересобираются все статьи единообразно.

## Раздел 4. Что НЕ менять

Это работает хорошо, не трогать:

- **Worktree-модель + handoff-process.** Изоляция per-task, чистая история, возможность параллелить — фундамент.
- **State machine на meta.json как single source of truth.** Простой, понятный, легко возобновлять.
- **JM + ТЗ + sections как отдельные шаги.** У каждого своя цель и свой агент. Сливать не стоит.
- **Аудит как обязательный шаг** — даже если применяется автоматически. Без него тексты текут.
- **Хук check-section.sh как валидатор.** Просто расширить (см. 3.10).

### Логика повтора темы

```
input: /write-article 1
match = index.articles.filter(a => a.topic_id == 1)
genres_in_xlsx = topics.xlsx row N "Жанры (2-3)" split на запятых
genres_done = match.map(m => m.genre)
genres_available = genres_in_xlsx - genres_done

if match.length == 0:
  go with genres_in_xlsx[0], platform = site
elif genres_available.length > 0:
  ask "тема сделана в жанре(ах) X. Доступно: Y. Брать Y[0]? или переписать?"
else:
  ask "по теме всё закрыто. Переписать существующую (выбери)/следующая тема/отмена"
```

### Auto-режим — структура состояний

### build-article-docx.mjs — структура

По аналогии с `build-strategy-docx.mjs`:

1. Шапка-таблица с метатегами (фон серый)
2. Spacer
3. H1 (Heading 1 style)
4. Author byline + дата
5. Breadcrumbs как обычный параграф
6. Body: проходим article.md через marked → AST, мапим в docx-элементы
7. Inline-картинки: скачиваем с Cloudinary, embed как Image
8. Таблицы: native docx Table
9. FAQ: H2 «Частые вопросы» + Q/A список
10. Tags: маркированный список ссылок

`share-article.mjs` (новый):
- Аналог `share-strategy.mjs`: загружает docx в указанную папку GDrive, конвертирует в Google Doc, выставляет права (если в ЗАКАЗЧИК.md указан владелец).
- Возвращает share URL, записывает в `meta.json.share.docx_url`.

### Авто-генерация фото — последовательность

```
After enhancer:
  1. photo-promter → photos/prompts.md (N промтов)
  2. for each prompt в параллель:
       generate_image(prompt, model=nb2, aspect_ratio=16:9, output_path=photos/NN.png)
  3. for each сгенерированного фото в параллель:
       publish_image(local_path, folder=articles/YYYY/MM/<slug>, public_id=NN-slug,
                     tags=[topic, slug, "ai-generated"], alt_text,
                     variants=["hero", "social", "thumb"])
  4. собрать photos/urls.json (с полями hero/social/thumb на каждое)
  5. update meta.state = "photos-published"
  → assemble-html
```

Падение одного `generate_image` — retry × 1, потом TODO-маркер в `urls.json`. Не блокирует остальные.

### article-fixer-batch — контракт

```
input:
  article_dir
  audit_md_path
  severity_filter: "critical+important" | "all" | "critical-only"

output:
  diff.md в article_dir/
  лог applied.json: [{id: "1.1", title, status: "applied"|"skipped"|"failed", reason}]
  обновлённый article.md

agent behavior:
  - читает audit.md, парсит секции «Критичные», «Важные», «Косметические»
  - для каждой правки в фильтре пытается применить
  - если правка точечная (фрагмент → рекомендация) — Edit
  - если правка структурная (перестановка абзацев) — отчитаться как skipped с причиной
  - все правки в одном проходе, без 30 вызовов
```

## Раздел 7. Что я измерю чтобы понять, стало ли лучше

Метрики успеха алгоритма после доработок:

1. **Time-to-completed:** от `/write-article N` до `completed` в meta. Цель — ≤ 25 минут без участия пользователя.
2. **Pause count:** число вопросов пользователю за прогон. Цель — 0 в авторежиме, 1 в режиме `--review`.
3. **Bug recurrence rate:** доля прогонов, где случилась ASCII-галлюцинация, потеряны метки, упал API. Цель — ≤ 5%.
4. **Deliverables completeness:** доля прогонов, где на выходе есть и docx (с share-URL), и output.html, и tilda-split (где применимо). Цель — 100%.
5. **Cost per article:** OpenRouter + Cloudinary + JM лимиты. Цель — ≤ $1.5 при 5 фото.
6. **Audit auto-apply rate:** доля автоматически применённых правок из audit.md (критичные + важные). Цель — ≥ 90%.

Логировать эти метрики в `articles/_metrics.jsonl` (append-only), на отметку каждого `completed`. После 10 статей видно тренд.
