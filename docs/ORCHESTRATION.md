# Конвенция оркестрации и диеты контекста

Оркестратор скила запускает субагентов, собирает их результаты и ведёт задачу к
финалу. Чтобы контекст оркестратора не распухал (а стоимость и риск ошибок не
росли), все скилы держатся одних правил передачи данных между шагами. Ниже -
эта конвенция.

Она опирается на действующее правило `.claude/CLAUDE.md`: **«Все промежуточные
результаты - в файлы, не в чат»**. Данный документ разворачивает его в набор
конкретных требований к возвратам агентов и к чтению данных оркестратором.

## Правила

### (а) Сводка агента в чат - не длиннее 8 строк

Субагент возвращает в чат только короткую сводку - **не длиннее 8 строк**: что
сделал, куда записал результат (путь к файлу), ключевые счётчики, статус (ок или
на чём споткнулся). **Полные тексты, JSON, HTML и содержимое файлов в чат
возвращать запрещено** - они лежат в файле, оркестратору достаточно пути.

### (б) Parent-fallback запрещён

Если шаг должен создать файл, его создаёт **сам агент**. Ситуация, когда агент
вернул содержимое файла в чат, а оркестратор записал файл за него
(«parent-fallback»), **запрещена**: она протаскивает весь объём через контекст
оркестратора - ровно то, чего конвенция избегает.

Правильная реакция на «агент не записал файл» - **ре-делегация**: повторный
запуск агента с явным требованием записать файл по нужному пути (и напоминанием,
что содержимое в чат возвращать не надо). Проверку «файл создан» делает
SubagentStop-хук (см. ниже механизм expected-маркеров).

### (в) Оркестратор не читает большие JSON целиком

Большие структурированные артефакты (semantic_pack, onpage, audience и т.п.)
оркестратор **не вычитывает в контекст целиком**. Вместо этого:
- **гейты через мини-скрипты с exit-кодом** - `verify-*.mjs` / `select-*.mjs`
  возвращают `0/2/…`, оркестратор ветвится по коду, не читая данные;
- **точечные выборки** - если нужно одно-два значения, берём именно их
  (счётчик, поле, длину), а не весь файл.

### (г) Verify-скрипты печатают не более ~30 строк нарушений

Скрипты проверки печатают **не более ~30 строк нарушений**; остальное сворачивают
счётчиком - **«и ещё N»**. Это удерживает вывод гейта в пределах короткой сводки
и не даёт длинному списку проблем залить контекст оркестратора.

### (д) Объёмную фактуру передаём путями, не содержимым

Большая вводная фактура (бриф, ТЗ, каталоги блоков, данные разведки) передаётся
агентам **путями к файлам**, а не вставкой содержимого в промт делегирования.
Агент сам читает то, что ему нужно, из указанного файла. Промт делегирования
остаётся тонким контрактом: что сделать, где вход, куда писать выход.

### (е) Опора на существующие механизмы

- Правило `.claude/CLAUDE.md` **«Все промежуточные результаты - в файлы, не в
  чат»** - корень всей конвенции; пункты выше конкретизируют его.
- **Механизм expected-маркеров** (`.claude/hooks/check-file.sh`, SubagentStop):
  делегирующий промт кладёт путь ожидаемого файла в
  `.claude/tmp/expected-<agent>-<run_id>.txt`, хук после остановки агента
  проверяет, что файл создан и непуст (для веера - что папка непуста), иначе
  `exit 2`. Это и есть автоматический контроль пункта (б): parent-fallback не
  нужен, потому что «файл не создан» ловится хуком и лечится ре-делегацией.

## Область действия

Конвенция **обязательна для новых скилов** - проектируй их сразу по этим
правилам. Существующие скилы приводятся к ней **по боевым этапам 1-6**
(доработки конвейера), а не разом.
