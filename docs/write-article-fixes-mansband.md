## Сквозной принцип ревизии

ТЗ почти везде предлагает «дописать промту агента ещё текста». Для LLM-агента это
мягкая гарантия. Где можно — заменяем на **детерминированную защиту в скрипте/оркестрации**
(verify-скрипт, glob-delete, exit-код), а усиление промта оставляем вторичным.

## БАГ #8 — verify-progress ложный warning по N-граммам → ПЕРЕОЦЕНКА: это НЕ минор

Подтверждено: `verify-progress.mjs` (134-142) ищет точную подстроку lowercase. НО важнее:
при сильном расхождении (>tol*2, строка 146) severity = **error → exit 2 → блокирует
`--auto` на шаге 5**. Пример из ТЗ (used=18, факт=1, target=12: расхождение 17 > tol*2=6)
= ERROR = **остановка нормальной статьи**. Это не косметика, это потенциальный halt пайплайна.

- [ ] `verify-progress.mjs`: нормализация перед поиском (дефис→пробел, lowercase, ё→е, склейка пробелов).
- [ ] N-граммную сверку **никогда не эскалировать до error** (cap = warning): счётчик в progress.json считает леммы (агент), а скрипт — литералы, расхождение структурно неизбежно.
- [ ] В сообщение добавить пояснение: «скрипт считает точные вхождения, progress.json — леммы/варианты; расхождение по N-граммам ожидаемо».

## НОВОЕ (не из ТЗ — найдено при сверке кода)

### НОВОЕ-A (ВАЖНЫЙ) — build-article-docx: хрупкая нумерация фото + потеря инлайн-меток

`build-article-docx.mjs` (320-327) определяет номер фото через `md.indexOf(text)` —
ломается на одинаковых абзацах; и матчит только метку-целый-абзац (`^\[ФОТО:...\]$`),
а **инлайн** `[ФОТО:]` внутри абзаца не распознаёт → выводит как литеральный текст
в документ. При этом `assemble-html.mjs` (148) делает правильно: сквозной счётчик
`++tagCounter.photo` + ловит инлайн. То есть HTML и docx расходятся по фото.

- [ ] Переписать нумерацию в docx на сквозной счётчик во время обхода lexer (как в assemble-html), а не `indexOf`.
- [ ] Обрабатывать инлайн-метки `[ФОТО:]` так же, как html. Это и есть механизм многих «лишних/недостающих» фото из #5/#6.

### НОВОЕ-B (ВАЖНЫЙ) — нет лимита ретраев на петли финализатора

section-writer имеет fail-fast (max 3, строки 152-166). А петли финализатора в скиле
(шаг 5, п.1 report.md и п.2 verify-markers) **без лимита** → упрямый финализатор = вечный
цикл и сжигание токенов в `--auto`.

- [ ] `write-article` шаг 5: лимит = 2 ретрая на каждую петлю, дальше — детерминированный fallback (для report.md — скил пишет сам, см. #2) или стоп с диагностикой (для verify-markers).

### НОВОЕ-C — аудит assemble-html на согласованность с docx

assemble-html нумерует/рендерит метки правильно, но «Photos used» (398) считает записи
urls.json, а не реально вставленные `<img>`. Это тот же класс, что #5.

- [ ] После фикса #5 (`verify-photos.mjs`) — единый источник истины снимает вопрос; отдельный фикс assemble-html не нужен, только держать в одном инварианте.

### НОВОЕ-D — дубли NN-файлов отравляют verify-markers

verify-markers читает `sections/*.md` в порядке сортировки (55-57); два файла с одним NN
(#4) раздувают `sectionMarkers` и ломают позиционную сверку тел (86). Фикс #4 чистит это
автоматически — отметить как связку, отдельной правки не нужно.
