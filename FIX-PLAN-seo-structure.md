## Принципы (корень почти всех правок)

1. **Алгоритм принимает SEO-решения сам и пишет их в ДАННЫЕ + журнал, а не прозой-рекомендацией.**
   Клиенту - только бизнес-вопрос «производите/оказываете? да/нет/обсудить». Разделение:
   SEO-механика = алгоритм; бизнес-реальность = клиент.
2. **Нет молчаливых скипов.** Любой условно-пропущенный или деградировавший шаг пишет поле-причину
   в свой JSON + surface в сводке + строкой в A6.md. (Закрывает деградацию JM, Казахстан, спаривание,
   gap-valve - один класс бага, один механизм.)
3. **Полнота через охват, калибровка спросом через лидеров.** Раскрывать нишу полно ВЕЗДЕ, но «полно»
   = глубина лидеров региона, не безгранично. Фикс «жадности» НЕ должен перелетать в раздувание мусором.
4. **Повышенный blast radius страхуется `--review`, не пассивным списком.** Раз клиент больше не
   SEO-арбитр, low-confidence СТРУКТУРНЫЕ решения ставят паузу в `--review`; `--auto` остаётся автономным.

## ФАЗА 2 - журнал + page_id + стадированное авто-применение (ВЫПОЛНЕНО 2026-05-30) `[ядро философии]`

> **Статус:** внедрено. `page_id` (slug названия) генерит master-list-builder, протянут через
> markers/semantic_pack/top10 (select-top10 выводит `id` + fallback на slug для legacy). Журнал
> `decisions.json`: marker-finder создаёт, cannibalization-resolver дописывает, structure-writer читает
> (раздел «Наши SEO-решения» + low-confidence чек-лист). n11-кейс: роль авто -> umbrella (физически в
> markers.json), инфо-запрос -> блог и коммерция -> страница-дом записаны решениями в журнал. marker_swap
> 1:1 журналируется (confidence high). `--review` пауза на low-confidence структурных. Идемпотентность по
> `id`. Smoke 11/11; `id` подтверждён в top10/cannibalization.
>
> **ОСОЗНАННАЯ ГРАНИЦА (отложено за пределы Фазы 2):** агенты ЗАПИСЫВАЮТ структурные решения
> (`split_page`/`add_page`/`merge_lowfreq`) в журнал, но НЕ переписывают физически `master_list.json`
> (добавление/удаление страниц = перенумерация `n` = риск). «Решение в данных» = журнал (аудируемо,
> обратимо). Физическую перестройку master_list по журналу включим отдельным шагом, когда проверим Фазу 2
> вживую. Это отвечает эталонному кейсу «решение в данных, не в прозе»: роль применена, маршруты в журнале,
> клиенту - ноль SEO-прозы.
>
> **Не покрыто smoke-тестами:** поведение агентов (роль/журнал) тестами не ловится - они гоняют только
> 3 скрипта. Реальная проверка - на живом прогоне `/seo-structure`.

### 2.1. Журнал решений `[ALGO разд.5]`
`decisions.json`: каждая запись `id, что_изменили, почему(данные), источник(агент/шаг), обратимо, confidence(high/low)`.
Наполняется на шагах маркеров/семантики/каннибализации; потребляется structure-writer + xlsx.

### 2.2. page_id - предусловие идемпотентности `[скрытая цена ALGO разд.5]`
Весь конвейер кеится по позиционному `n` (6 артефактов + 3 скрипта мапят по `p.n`). Расщепление/добавление
страницы перенумеровывает всё ниже. Ввести стабильный slug-id, протянуть master_list -> markers ->
semantic_pack -> top10 -> xlsx -> import. Без него идемпотентность хрупкая.

### 2.3. Авто-применение 1:1 (безопасная половина) `[ALGO разд.2]`
Смена маркера на синоним + смена роли (`umbrella/navigational/info/article`) - НЕ перенумеровывают ->
применять сразу, писать в журнал. Регресс-гард: 4 рабочие авто-замены 001 продолжают работать.

### 2.4. Авто-применение структурное `[ALGO разд.2, после 2.2]`
Расщепление/добавление страниц - ТОЛЬКО после page_id. До него - оставлять рекомендацией.

### 2.5. Идемпотентность + confidence-аудит `[ALGO разд.5]`
Перед применением - проверка журнала по `id`. `--review` пауза на low-confidence структурных (принцип 4).

Рычаг - глубина per_page, НЕ «больше конкурентов» (это вторично и дороже).

> (каркас категории, SKU - по линейке клиента). 3.5 (кэп конкурентов в /seo-analysis) НЕ трогал - вторично.

### 3.1. Глубже per_page `[главный рычаг, апстрим не нужен]`
`domain_pages` per_page=100-150 + второй проход по другому `sort` (трафик/top10), объединять.

### 3.2. Фильтр по качеству URL, не по видимости
Снять отбраковку «top50_count=0» (нишевая категория с 0 видимости валидна как посадка). Резать
технический мусор: пагинация, фильтры/сортировки, query-параметры, теговые облака, дубли со/без слеша.

### 3.3. Своя ниша vs чужая
Броадер-ниша клиента (теплообменники, газгольдеры) -> кандидат с бизнес-флагом. Чужая ниша мультинишевого

### 3.4. Анти-компрессия в промпт + adjacent_directions
«аккуратность/краткость списка НЕ причина сокращать охват»; «никогда не сливать разные виды оборудования».
Медиана глубины лидеров - мягкая проверка («объединение >> медианы -> разобраться»), не жёсткая цель.
`serp.adjacent_directions` с `competitor_count>=2` -> кандидаты с бизнес-флагом.
- **Граница полноты:** ширину (набор категорий) зеркалим; глубину (конкретные SKU/модели) НЕ выдумываем -
  строим каркас категории + шаблон карточки, помечаем «карточки по линейке клиента».

### 3.5. (опц., вторично) кросс-скильная развилка

недоделка, а сознательная очередь: материализация поверх непроверенной автономии умножает риск.
