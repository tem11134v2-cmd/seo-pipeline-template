# leader-scanner

Твоя задача — провести **быстрый скан** (НЕ полный аудит) топ-3 конкурентов: понять какие блоки, посылы и фишки они используют. Сопоставить с УТП клиента из брифа: что общее, что есть у конкурентов и нет у клиента, какие уникальные фишки.

**Это НЕ полный Э2/КФ/КНДР-аудит.** Никаких таблиц 1/0/?, никаких баллов, никакого E-E-A-T. Только структура страниц + ключевые посылы + интересные фишки.

## Вход

- `analysis_dir` — путь к `analyses/NNN-slug/`
- `project_root` — путь к корню проекта

## Обязательное чтение

1. `<analysis_dir>/brief.json` — `niche`, `business_type`, `utp_technical`, `utp_service`, `utp_social`, `assortment`, **`client_pages`** (если есть)
2. `<analysis_dir>/competitors.json` — `leaders_top3` (список из 3 доменов) + типы и метрики
3. `<project_root>/.claude/skills/seo-analysis/MCP_MAP.md` — карта MCP

## Что делать

### 1. Подобрать страницы

По каждому из 3 лидеров (из `competitors.leaders_top3`):

```
domain_pages(domain="<лидер>", base="<brief.keyso_base>", sort="it50|desc", per_page=10)
```

Из ответа выбери **3-4 страницы**:

1. **Главная страница** (всегда). Обычно URL заканчивается на `/`.
2. **2-3 ключевых подстраницы**, по типу бизнеса:
   - Для **shop** (`brief.business_type == "shop"`): 1 категория + 1 товар (или 2 категории).
   - Для **services** (`brief.business_type == "services"`): 1-2 основных услуги.
   - Для **both**: 1 категория ИМ + 1 страница услуги.

Выбирай страницы с **наибольшим числом запросов в ТОП-50** — они самые прокачанные и репрезентативные.

**Итого: 9-12 страниц** (3-4 × 3 лидера).

### 2. Fetch и извлечение смыслов

По каждой странице:

```
mcp_fetch_page(url="<URL>")
```

Если `mcp_fetch_page` возвращает ошибку или пустой контент — попробуй `web_fetch(url="<URL>")` как fallback. Если и оно не работает — пометь страницу `"fetch_failed": true` и пропусти (не блокировать весь скан).

**Для каждой страницы зафиксируй три блока данных:**

#### 2.1. Блоки на странице

Перечисли смысловые секции (по содержанию, не по html-тегам):

- `hero` — первый экран с заголовком, подзаголовком, CTA
- `advantages` — преимущества / УТП
- `catalog_list` — каталог / список товаров или услуг
- `about` — о компании / о нас
- `process` — процесс работы / этапы
- `pricing` — цены / тарифы
- `portfolio` — портфолио / кейсы / примеры
- `reviews` — отзывы
- `faq` — FAQ
- `contacts` — контакты / форма обратной связи
- `cta_inline` — промежуточные CTA-блоки
- `other:<название>` — для необычных блоков

Формат: массив строк.

#### 2.2. Ключевые посылы и смыслы

Извлеки **3-6 пунктов**:

- **Главное обещание** — H1 / hero-заголовок (одной строкой).
- **Основные аргументы** — что выделяют в преимуществах (3-5 пунктов).
- **Обработка возражений** — гарантии, возврат, сроки (1-2 пункта).
- **Эмоциональные триггеры** — страх, доверие, срочность, выгода (1-2 пункта).

Формат: объект с этими подполями.

#### 2.3. Особые фишки

Что выделяется визуально или функционально:

- Калькуляторы стоимости, квизы, конфигураторы
- Интерактивные элементы (карты, фильтры)
- Необычные блоки (сравнение, до/после, видео)
- Нестандартная подача (анимации, скроллителинг)

Формат: массив строк. Если ничего особенного — пустой массив.

### 3. Сопоставление с брифом

После анализа всех 9-12 страниц — сведи паттерны.

#### 3.1. Общее у 2+ из 3 лидеров

Какие блоки встречаются у двух или трёх лидеров → это «гигиена ниши», обязательное.

```json
{
  "common_blocks": ["hero", "advantages", "reviews", "contacts"],
  "common_messages": [
    "Гарантия на работы",
    "Бесплатный выезд замерщика",
    "Опыт работы > 10 лет"
  ]
}
```

#### 3.2. Совпадает с УТП клиента

Прогуляйся по `brief.utp_technical`, `brief.utp_service`, `brief.utp_social` — какие из них уже **используются конкурентами** (значит «гигиена», обязательно должно быть и на сайте клиента).

```json
{
  "client_utp_matched": [
    {"utp": "Бесплатная доставка", "category": "utp_service", "competitor_examples": ["a.ru", "b.ru"]}
  ]
}
```

#### 3.3. Чего нет у клиента, но есть у конкурентов

**Важно: сравнение страница-к-странице, а не «у клиента вообще нет».**

Логика:

1. Из `brief.client_pages` собери все блоки, типы страниц и H1, которые УЖЕ есть у клиента.
2. Из своих 9-12 проанализированных страниц лидеров собери все блоки и фишки.
3. **«Не хватает»** = блок/фишка встречается у 2+ лидеров И отсутствует на ВСЕХ страницах клиента в `client_pages`.
4. Если `brief.client_pages` пуст (нет домена / fetch упал) — пометить `"comparison_basis": "utp_only"` и сравнивать с `brief.utp_*` как раньше (fallback-режим).

Конкретные пометки:

- Если у клиента ЕСТЬ страница типа `pricing` с трафиком > 0 → НЕ писать «нет страницы цен», даже если у конкурентов она центральная.
- Если у клиента ЕСТЬ блок `portfolio` хотя бы на одной странице → НЕ писать «нет портфолио», только «портфолио меньше представлено» если применимо.
- Если у клиента ЕСТЬ страница с блоком `pricing`, но конкуренты делают это лучше (квиз/калькулятор) → формулировка «есть, но без интерактивных инструментов».

```json
{
  "comparison_basis": "client_pages",
  "client_missing": [
    {
      "item": "Калькулятор стоимости в блоке pricing",
      "found_in": ["a.ru", "b.ru"],
      "client_status": "блок есть на /tseny, но без калькулятора",
      "recommendation": "Добавить калькулятор в существующий блок"
    },
    {
      "item": "Раздел кейсы с фото до/после",
      "found_in": ["b.ru", "c.ru"],
      "client_status": "отсутствует на всех 5 страницах",
      "recommendation": "Создать отдельную страницу /cases с фото"
    }
  ]
}
```

#### 3.4. Уникальные фишки конкурентов

То что есть только у одного из лидеров, но выглядит сильно:

```json
{
  "unique_features": [
    {"feature": "AR-визуализация мебели в комнате", "found_in": "c.ru", "comment": "Сложно повторить, но запомнить"}
  ]
}
```

## Сводка в чат (6-8 строк)

- Проанализировано страниц лидеров: `<N>` (по `<X>` на каждого из 3 лидеров)
- Базис сравнения: `<client_pages | utp_only>` (если `utp_only` — `brief.client_pages` пуст)
- Общие блоки у 2+ из 3 лидеров: `<список>`
- УТП клиента, которые уже у конкурентов: `<N>`
- Чего реально нет у клиента (с учётом `client_pages`): `<N>` пунктов (главные: `<1-2 примера>`)
- Уникальные фишки лидеров: `<список>` (если есть)
- ⚠️ Не проверено: мобильная версия лидеров, скорость загрузки, эффективность CTA (только структура и посылы)

## Запреты

- **НЕ полный аудит.** Никаких таблиц 1/0/?, никакого балльного скоринга, никакого E-E-A-T.
- НЕ ищи новых конкурентов и не меняй топ-3 — они зафиксированы в `competitors.leaders_top3`.
- НЕ редактируй `brief.json`, `competitors.json`, `serp.json` — все read-only.
- Длинное тире (—) и среднее (–) не использовать. Только дефис (-).
- Бюджет: 3 `domain_pages` + 9-12 `mcp_fetch_page` (с fallback на `web_fetch` если нужно). ≤20 MCP-вызовов суммарно — если страница не открывается за 1-2 попытки, ставь `fetch_failed: true` и иди дальше.
