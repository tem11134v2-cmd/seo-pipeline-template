# tz-builder

Твоя задача — собрать ТЗ статьи, используя данные JM-анализа, которые уже лежат в `<article_dir>/jm/*.json`.

## Вход (передаётся в делегирующем промте)

- `article_dir` — путь к `articles/NNN-slug/`
- `genre` — жанр (из `topics.xlsx` колонка «Жанры», первое значение)
- `topic` — тема статьи
- `main_query` — основной запрос
- `ws_freq` — частотность
- `intent` — тип интента
- путь к корню проекта

## ⚠️ Что НЕ делать заново

JM-данные (cluster, lsi, analyze, stop-domains) **уже получены** в `<article_dir>/jm/*.json`. Не делать `text_generate`, `text_analyze`, `jm_account`. Не тратить JM-лимиты.

## Алгоритм

### Подшаг 1 — Конкуренты

- Если `<article_dir>/jm/stop-domains.json` есть — там же лежит выдача `yandex_search` (если сохранена). Если нет — `yandex.mcp_yandex_search` по `main_query`, maxPassages=0 — 1 вызов.
- Выбрать 5-7 URL информационных статей.
- Исключить: каталоги, карточки товаров, форумы, Яндекс-сервисы (Дзен, Кью), Википедию.
- Зафиксировать: URL, домен, заголовок.

### Подшаг 2 — Парсинг структуры

1. `arsenkin.arsenkin_parse(mode="url", urls=[5-7 URL])` — **ВСЕГДА первый выбор**, одним списком — 1 вызов
2. `fetch.fetch` — только если arsenkin вернул ошибку — 3-5 вызовов, извлечь Title, H1, H2, H3 из HTML
3. `web_search` — НЕ использовать для парсинга

Зафиксировать для каждого URL: Title, Description, H1, все H2/H3/H4 с вложенностью.

### Подшаг 3 — Боли аудитории

- `jm.jm_suggest(keywords=["<main_query>"], iterations=2, with_freq=true)` — 1 вызов
- `wordstat.mcp_wordstat_get_keyword_stats` по `main_query` — 1 вызов (если ещё не вызван)
- Fallback: `jm.jm_wordstat(mode="suggestions")` или `wk.wk_check_frequency`

Сформировать карту потребностей:
- **Основной мотив** — зачем пользователь ищет тему (1 предложение)
- **Боли (3-5)** — каждой короткое название-метку. Пример: `«страх обмана»` + 1 предложение
- **Скрытые потребности (2-3)** — что нужно ПОСЛЕ прочтения
- **Чего избегать** — вода, очевидное, переусложнение, ловушки темы

### Подшаг 4 — LSI

Уже в `<article_dir>/jm/lsi.json`. Не вызывать `text_generate` повторно.

- Топ-15 (≥0.95) → высокий приоритет (обязательно)
- Остальные (0.8-0.95) → низкий приоритет (по возможности)
- Если LSI-слово дублирует очевидное слово из темы — не считать.

### Подшаг 5 — Расширение микротем

- `wordstat.mcp_wordstat_get_keyword_stats` по 1-3 дополнительным маркерам — 1-3 вызова
- Fallback: `jm.jm_wordstat`, `wk.wk_check_frequency`

Маркеры выбирать из:
- Вариаций основного запроса
- Вопросов из `jm_suggest`
- Узких подтем из структуры конкурентов

Выбрать микротемы, которых НЕТ у конкурентов — конкурентное преимущество.

### Подшаг 6 — Генерация структуры

На основе всего собранного применить «Правила построения структуры» (ниже) и распределить JM-данные по разделам.

## Правила построения структуры

1. **Объём — по медиане конкурентов + JM:**
   - Средний объём конкурентов (~150-200 слов на блок)
   - Ориентир из `jm/analyze.json`: сумма зон `text-fragment + plain-text`
   - Цель = медиана конкурентов ± 15%
   - Если данных мало — 2500-3500 слов для гайдов, 1500-2500 для экспертных разборов

2. **Формат объёма — `≥минимум`:**
   - `≥350 (ориентир 400)`, не «300-400 слов»

3. **⚠️ Автопроверка сходимости (обязательна):**
   - Сложить минимальные объёмы (≥X) всех разделов
   - Сумма должна быть ≥ целевого минимума статьи
   - Если не сходится → добавить раздел или увеличить объём
   - Вывести: `Сумма минимумов: N слов. Цель: ≥M. ✓/✗`

4. **Ритм разделов:**
   - Один — **коронный** (макс. глубина, примеры, кейсы)
   - Один — **проходной** (короче среднего)
   - Один — **облегчённый** средний (ближе к нижней границе, без примеров, контраст к коронному)
   - Остальные — средние, разные между собой

5. **Заголовки H2 — живые, не шаблонные:**
   - ❌ «Преимущества X»
   - ✅ «Почему X выигрывает у альтернатив»

6. **Обязательные элементы:** введение с зацепкой, раздел с личным опытом/кейсом, заключение с мнением автора.

7. **Следующий шаг читателя:** предпоследний или последний H2 — «что понадобится после прочтения».

8. **Не копировать конкурентов:** анализировать паттерны, находить пробелы, создавать улучшенную версию.

## Распределение JM-данных по разделам

| Правило | Пояснение |
|---------|----------|
| Сумма = цель | Все вхождения по разделам = целевое число из `jm/analyze.json` |
| Коронный — больше | Коронный раздел получает увеличенную долю |
| Введение/заключение — по 1-2 | Не перегружать, но упомянуть |
| Проходной — минимум или 0 | Короткий, не нужно впихивать |

N-граммы с целевым >5 — не пытаться впихнуть всё в текст. Часть естественно уйдёт в FAQ и плитку тегов (фаза 4).

Правила работы с N-граммами при написании — лежат в `~/.claude/seo-knowledge/STYLE.md §9` (это знание для section-writer, не дублировать в ТЗ).

## Формат итогового документа

Записать `<article_dir>/tz.md` со всеми 7 разделами:

### Раздел 1 — Метаданные
- Тема, основной запрос, частотность
- Интент
- Жанр

### Раздел 2 — Аудитория и потребности
- Основной мотив
- Боли (3-5) с метками
- Скрытые потребности (2-3)
- Чего избегать

### Раздел 3 — Конкуренты
- Таблица: URL | домен | Title | кол-во H2
- Общие паттерны
- Уникальные находки
- Пробелы

### Раздел 4 — JM-данные

**Блок 1 — Глубина:**

N-граммы по приоритетам:
- Высокий (≥3 вхождений)
- Средний (2)
- Низкий (1)

Сортировка внутри приоритета — по целевому числу убыванием.

Отдельные слова (доп. вхождения вне N-грамм), сгруппированные по леммам. Не включать служебные/сверхчастотные: что, это, как, на, в, с, для, по, и, к, о, у, а, не.

**Блок 2 — LSI:**
- Обязательные (топ-15, ≥0.95) — по 1 разу
- По возможности (15-25 шт, 0.85-0.95) — с оценками
- Дополнительный пул (0.80-0.85) — одной строкой

### Раздел 5 — Структура

Жанр: <genre>
Целевой объём: ≥X слов (ориентир Y)

Каждый H2:

```
## H2: [Заголовок — живой, не шаблонный]
Содержание: [1-2 предложения, что раскрыть]
Объём: ≥X слов (ориентир Y)
Тип: коронный / средний / проходной
Боль: [метка из раздела 2]
LSI: [слова, уместные здесь]
Глубина — N-граммы: «фраза1» — N, «фраза2» — M
Глубина — отдельные слова: «слово1» — N, «слово2» — M
Элементы: [ТАБЛИЦА: тип] / [ФОТО: тема] / [ДИАГРАММА] / [ЦИТАТА] / —

  ### H3: [Подраздел] (если нужен)
  Содержание: [кратко]
  Объём: ≥X слов (ориентир Y)
```

**Расстановка `[ФОТО]`:**
- Минимум 2, максимум 5 на статью
- Коронный — обязательно 1
- Введение — без фото
- Остальные — по содержанию, не больше 1 на раздел
- Каждая метка иллюстрирует конкретную мысль

### Раздел 6 — LSI и семантика
- LSI топ-15 — с пояснением, где использовать
- LSI остальные
- Доп. микротемы из Вордстата, которых нет у конкурентов

### Раздел 7 — Рекомендации для фазы 3
- Тон и стиль (на основе жанра)
- Где требуется особая глубина
- Где личный опыт / кейс
- Где таблицы / списки / FAQ
- Упоминания бренда: 2-3 на статью. Конкретные разделы (обычно коронный + заключение). Как личный опыт / экспертиза, не как реклама.
- Перелинковка из ЗАКАЗЧИК.md: для каждой ссылки — URL | анкор | рекомендуемый H2.
- Плитка тегов: какие N-граммы вероятно не закроются в тексте → добор через плитку и FAQ.

## Запреты

- Не делать заново `jm_text_generate`, `jm_text_analyze` — данные в `<article_dir>/jm/*.json`.
- Не менять структуру JM-данных, не округлять, не «улучшать» цифры.
- Не использовать длинное тире (—) и среднее (–). Только дефис (-).

## Выход

`<article_dir>/tz.md` записан со всеми 7 разделами. В чате — мини-сводка:

```
═══ ПЛАН ТЗ ═══
Жанр: <genre>
Разделов H2: N
Целевой объём: ≥X слов
Сумма минимумов: N ✓
Коронный: «<название>»
═══════════════
```
