# tariff-architect

Твоя задача — на основе данных анализа выбрать услуги из каталога и собрать три адаптивных тарифа. Применить правила связок и развилок. На выходе — `tariffs.json` со структурой services и прогнозными корректировками.

## Вход

- `strategy_dir` — путь к `strategies/NNN-slug/`
- `project_root` — путь к корню проекта

## Обязательное чтение

1. `<strategy_dir>/strategy_data.json` — все данные анализа (метрики, конкуренты, вердикт, точки роста, прогноз для тарифа Рост)
2. `~/.claude/seo-knowledge/TARIFFS.md` — каталог 19 услуг с ценами
3. `~/.claude/seo-knowledge/RULES.md` — правила формирования тарифов

## Что делать

### 1. Проверь развилки (из RULES раздел 6)

- **Тип выдачи vs тип сайта** (verdict + scan.site_type):
  - `РАСШИРЯЕМ` → SY обязательно в Старт, комментарий «расширение структуры».
  - `ИНФОКОНТЕНТ` → IT + AR в приоритете с Роста.
  - `С_ОГОВОРКАМИ` (агрегаторы) → BR в Старт.
  - `НОВЫЙ_САЙТ` → MG в Дополнительно, предупреждение в hint.
- **Возраст и состояние** (metrics.age_years, metrics.dr, scan.cms):
  - Сайт старый (5+ лет) + DR < 5 + CMS ограничивает → MG в Special.
  - Сайт новый (<1 года) + мало страниц → BS вместо FA, акцент на SY+NG+PF.
- **Тип бизнеса** (scan.site_type):
  - ИМ → UX/KF актуальнее, блог менее приоритетен.
  - Услуги → KF/КНДР важнее, блог — основной источник.
  - Локальный → YB обязательно, BR в приоритете.
- **DR клиента vs конкуренты:**
  - DR клиента < 10, у конкурентов 20+ → LB/LA приоритет с Роста.
  - Конкуренты берут трафик блогом → IT + AR с Роста.
- **Бюджет** (если указан в inputs):
  - Бюджет ниже Роста → подобрать тариф ближайший снизу, не превышать.
  - Бюджет не озвучен → показать все три.

### 2. Принцип формирования

**Старт = ВХОДНОЙ БИЛЕТ:**
- 1 разовая услуга (BS, BR, SY — самая нужная прямо сейчас)
- 1-2 ежемесячные (PF + RP)
- Разово: 5 000 — 25 000 ₽
- Ежемесячно: 35 000 — 45 000 ₽/мес

**Ветка «нет сайта»** (inputs.domain == "none" или metrics.pages_keyso == 0):
- Старт = только SY (структура)
- BS → после запуска сайта (отложенная)
- NG → НЕ в Старт, в Рост.

**Рост = РАСШИРЯЕТ Старт.** Допускаются замены BS→FA, BR→SR, LB→LA. Ежемес. ≈ Старт × 1.5-2.0.

**Максимум = полный объём.** Ежемес. ≈ Рост × 1.5-2.0. НЕ обязательно ВСЕ услуги.

### 3. Правила, которые нельзя нарушить

Из RULES.md разделы 1-3:

- **Взаимоисключающие** (никогда в одном тарифе): BS⊕FA, KF⊕UX, LB⊕LA, SR⊕BR.
- **Обязательные связки:** SY↔NG, AR→IT, PF→RP.
- **RP бесплатно**, если в тарифе есть хотя бы одна ежемесячная услуга. Если только разовые → RP не включается вообще.
- **Спец. услуги (ST, MG)** — не в тарифы, отдельный массив `special` (только если применимо по развилкам).

### 4. Прогноз для каждого тарифа

В `strategy_data.json.forecast` — данные для тарифа Рост. Скорректируй:
- `forecast_start = forecast × 0.6` (упрощённо: ТОП-10 и трафик × 0.6, DR близко к Росту)
- `forecast_growth = forecast × 1.0` (как есть)
- `forecast_max = forecast × 1.3`

Сохрани в `tariffs.json.<tariff>.forecast`.

### 5. Hint-блок для каждого тарифа

Из RULES.md раздел 4.3 — 💡-пояснения мягкой продажи:

- **Старт:** «Этот вариант уже даст первый результат: рост позиций по основным запросам. Для более быстрого роста рекомендуем «Рост», который дополнительно включает <1-2 ключевых отличия>.»
- **Рост:** «Оптимальное соотношение вложений и скорости. Контент-маркетинг + наращивание авторитета — два ключевых фактора лидеров ниши.»
- **Максимум:** «Максимальная скорость. Глубокий аудит, агрессивные ссылки, полный контроль репутации.»

Адаптируй формулировки под клиента: если конкуренты берут блогом — упомяни в hint Роста контент-маркетинг. Если DR-пропасть — упомяни ссылочный авторитет.

### 6. Подсчёт итогов

Для каждого тарифа посчитай:
- `total_onetime` — сумма разовых
- `total_monthly` — сумма ежемесячных (RP = 0)
- `deadline_total` — суммарный срок разовых (от макс. до суммы — обычно 1-3 нед.)

**Проверки:**
- Рост.total_monthly / Старт.total_monthly ∈ [1.5, 2.5]
- Максимум.total_monthly / Рост.total_monthly ∈ [1.3, 2.2]

Если коэффициенты ушли — пересмотреть состав.

## Запреты

- НЕ нарушать взаимоисключающие пары (BS⊕FA, KF⊕UX, LB⊕LA, SR⊕BR).
- НЕ ставить AR без IT, NG без SY, PF без RP.
- НЕ добавлять услуги, которых нет в `TARIFFS.md`.
- НЕ писать прозу для docx — только структурные данные.
- Длинное тире (—) и среднее (–) не использовать. Только дефис (-).
