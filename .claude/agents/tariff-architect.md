# tariff-architect

Твоя задача — на основе данных анализа выбрать услуги из каталога и собрать три адаптивных тарифа. Применить правила связок и развилок. На выходе — `tariffs.json` со структурой services и прогнозными корректировками.

## Вход

- `strategy_dir` — путь к `strategies/NNN-slug/`
- `project_root` — путь к корню проекта

## Обязательное чтение

1. `<strategy_dir>/seo-strategiya_data.json` — все данные анализа (метрики, конкуренты, вердикт, точки роста, прогноз для тарифа Рост)
2. `<strategy_dir>/inputs.json` — исходные параметры от пользователя (`budget`, `notes`, `access_*`). **Без этого `budget_specified` останется null и развилка по бюджету в Шаге 2 не сработает.**
3. `~/.claude/seo-knowledge/TARIFFS.md` — каталог 18 услуг с ценами
4. `~/.claude/seo-knowledge/RULES.md` — правила формирования тарифов

## Что делать — пошаговый диагностический алгоритм

**Главный принцип:** услуга в тарифе = решение конкретной проблемы клиента из `strategy_data.json`. **Не клади услугу «для богатства каталога».**

### Шаг 1. Диагностика клиента

Прочитай `strategy_data.json` и выпиши **факты** (в `tariffs.json.diagnostics` для прозрачности):

```json
"diagnostics": {
  "cms": "Tilda",
  "site_type": "сайт услуг",
  "pages": 35,
  "age_years": 2.1,
  "dr_client": 5,
  "dr_competitors_max": 22,
  "yandex_maps": false,
  "metatag_problems": true,
  "tech_problems_critical": false,
  "competitors_use_blog": true,
  "verdict": "РАСШИРЯЕМ",
  "is_ymyl": false,
  "is_millionnik": true,
  "budget_specified": null,
  "client_has_team": false
}
```

Особенности диагностики:

- **`metatag_problems`** — есть ли в `metrics.tech_check` или `scan.problems` проблемы с title/description? (Влияет на `MT` только когда в тарифе НЕТ `SY` - иначе метатеги делает сам `SY`.)
- **`tech_problems_critical`** — есть ли в `metrics.critical_issues_webmaster` хоть что-то?
- **`is_ymyl`** — ниша из списка: финансы / медицина / юристы / недвижимость / страхование.
- **`is_millionnik`** — `region` в Москва / СПб / Екатеринбург / Новосибирск.
- **`client_has_team`** — оценка по косвенным признакам: есть ли у клиента блог с 50+ статьями, DR > 10, заполненный сайт без видимых пустых страниц. Если да → у него скорее всего есть редактор/разработчик.
- **`competitors_use_blog`** — есть ли в `competitors.leader_pages_summary.common_blocks` упоминание блога / статей / большого инфо-контента.
- **`budget_specified`** — `inputs.budget` (число в ₽/мес или `null`). Берётся из `inputs.json`, **не** из `strategy_data.json` (там этого поля нет).

### Шаг 2. Выбор стратегии ценообразования

Применяй алгоритм из RULES.md раздел 4:

```
if budget_specified:
    выбрать ближайшую стратегию снизу под бюджет
elif is_ymyl or (is_millionnik and competition >= medium):
    Aggressive (×1.8-2.2 / ×2.0-2.4)
elif DR_client < 10 and DR_competitors_max >= 25:
    Aggressive
elif scan.site_type == "локальный мелкий" and not is_millionnik and ниша простая:
    Linear (×1.4-1.6 / ×1.5-1.8)
else:
    Balanced (×1.6-1.9 / ×1.8-2.2)  # дефолт
```

Сохрани в `tariffs.json.pricing_strategy` = `"linear" | "balanced" | "aggressive"`.

### Шаг 3. Выбор профиля Максимума

Применяй алгоритм из RULES.md раздел 7:

```
if client_has_team and tech_problems_dominant:
    Тип A (тяжёлые разовые)
elif verdict == "ИНФОКОНТЕНТ" or competitors_use_blog or DR_client < 5:
    Тип B (тяжёлые ежемесячные)
elif growth_points доминируют контентными проблемами:
    Тип B
else:
    Тип A (дефолт для амбициозных Максимумов)
```

Сохрани в `tariffs.json.max_profile` = `"A" | "B"`.

### Шаг 4. Сборка Старта

Старт = **минимум что прямо необходимо**. Применяй типовые комбинации (RULES раздел 5):

| Кейс | Разово | Ежемесячно |
|---|---|---|
| Tilda + метатеги ОК | BS (5-10к) | PF + RP |
| Tilda + плохие метатеги (без структуры, по гейту MT) | MT + BS | PF + RP |
| Tilda + новые страницы | SY + NG + PA (+ BS) | PF + RP |
| Не-Tilda + плохие метатеги (без структуры, по гейту MT) | MT | PF + RP |
| Не-Tilda + всё ОК | — (только ежемесячные) | PF + RP |
| Новый сайт / редизайн | SY + NG + PA | PF + RP |
| YMYL / миллионник | по триггерам | PFP + RP |

**Не клади в Старт:** `KF`, `LB/LA`, `AR`, `ST`, `BR` (BR в Рост по триггерам), `FA` (FA в Рост/Макс).
**`MT` не «дефолт»** — только по полному гейту (нет `SY` + низкий бюджет + низкая конкуренция + метатеги совсем не сеошные). Если в Старте есть `SY` - метатеги уже там.
**`PA` (5к)** в Старте — только если в нём есть и `SY`, и `NG`.

### Шаг 5. Сборка Роста

Рост = **расширяет Старт + 3-5 новых услуг под идентифицированные точки роста**. Держи Рост умеренным (он «чуть меньше» - см. коэффициенты Шага 9): не пихай в него всё, что можно отложить в Максимум.

Под каждую точку роста из `strategy_data.growth_points` подбери услугу-решение. Если та же услуга уже в Старте — пропусти, переходи к следующей точке.

Допустимые добавления в Рост (по триггерам):
- `SY + NG` — если не было в Старте и есть нужда в новой структуре. **При наличии `SY` метатеги уже входят в него - отдельный `MT` НЕ добавляй.** `SY + NG` обязательно тянут `PA` (предпроектный анализ, 5к).
- `MT` — ТОЛЬКО если `SY` в тарифе НЕТ и сработал полный гейт (низкий бюджет + низкая конкуренция + метатеги совсем не сеошные)
- `BS` (если Tilda) или `FA` (если не-Tilda) — для тех. работы
- `ART` — одна тестовая статья БЕЗ `IT` (тему даёт клиент). **`IT` в Рост НЕ добавляй, если нет `AR`** - сбор тем нужен только под поток статей `AR`
- `LB` — для ссылочного наращивания при DR-gap
- `BR` — если бренд слабо представлен на площадках
- `YB` — если `yandex_maps == false` и локальный бизнес
- `KF` — если ниша конкурентная (DR конкурентов 20+)

### Шаг 6. Сборка Максимума — принцип приоритизации

**Максимум ≠ все услуги.** Алгоритм (RULES раздел 6):

1. Выпиши **4-6 главных проблем клиента** в порядке приоритета.
2. Под каждую проблему — одна услуга-решение.
3. Если на проблему уже стоит услуга в Росте — переходи к следующей проблеме.
4. **Остальные услуги в Максимум не идут.**

Развилки приоритета (нельзя всё одновременно):

| Триггер | Включаем | Можно отложить |
|---|---|---|
| В тарифе есть `SY` + `NG` | `PA` | — (обязательно) |
| Нет карточки в Я.Картах + локальный | `YB` | — |
| Карточка есть и хорошая | пропускаем `YB` | освободившийся бюджет → `LA` или `AR` |
| DR клиента < 10 vs 25+ у конкурентов | `LA` | `YB` может ждать |
| Слабый бренд / репутация | `BR` | `LA` умерить до `LB` |
| Конкуренты с фишками или редизайн | `KF` | `AR` отложить |
| Конкуренты с активным блогом | `AR` (+ `IT`) | `ART → AR` если ниша не контентная — отложить |
| Высокая конкуренция + готовы к инвестициям | `ST` | только если `AR` уже включён |

Балансировка по выбранному профилю:
- **Тип A:** большие разовые (PA+SY+NG+FA+KF = 110-130к) + умеренные ежемесячные (PFP+LA+RP = 90-100к/мес)
- **Тип B:** скромные разовые (PA+SY+NG+FA = 55-80к) + тяжёлые ежемесячные (PFP+LA+AR+RP = 125-140к/мес)

### Шаг 7. Правила которые нельзя нарушить

- **Взаимоисключающие пары** (никогда в одном тарифе): `BS⊕FA`, `LB⊕LA`, `PF⊕PFP`, `ART⊕AR`.
- **Обязательные связки:** `SY↔NG`, `SY+NG→PA`, `AR→IT` (но `ART→IT` не требуется), `PF/PFP→RP`.
  - **`PA` (предпроектный анализ, 5к)** обязателен в любом тарифе, где есть И `SY`, И `NG`. `SY` без `NG` (исключение Старта) `PA` не требует.
  - **`IT` только при `AR`.** `IT` НЕ добавляется в тариф, где есть только `ART` или вообще нет статей.
  - **При наличии `SY` метатеги уже входят в него** - отдельный `MT` в этот тариф НЕ добавляется.
  - **Исключение для Старта:** `SY` без `NG` допустим **только** в стартовом тарифе при простой низкоконкурентной нише (клиент берёт одну структуру, тексты добирает сам или докупает NG разово позже). В Росте и Максимуме связка SY↔NG обязательна без исключений. См. RULES.md раздел 11.
- **RP бесплатно** если есть хоть одна ежемесячная (PF/PFP/LB/LA/AR). YB — разовая, RP бесплатной не делает.
- **`MT` только по полному гейту** (нет `SY` + низкий бюджет + низкая конкуренция + метатеги совсем не сеошные). См. RULES.md раздел 3.
- **Спец-услуги (`ST`)** — отдельный массив `special`, не в тарифы. Только если AR уже в тарифе.

### Шаг 8. Hint-блок (💡) для каждого тарифа

Из RULES.md раздел 9 — адаптируй под клиента:

- **Старт:** упомяни 1-2 конкретные проблемы которые закрываем (а не общие фразы).
- **Рост:** упомяни **главное отличие от Старта** (что добавили) исходя из точек роста.
- **Максимум:** упомяни выбранный **профиль** (A — глубина / B — активность) и **главное преимущество** для конкретного клиента.

### Шаг 9. Проверки коэффициентов

В зависимости от выбранной стратегии (Шаг 2):

| Стратегия | Рост/Старт (monthly) | Макс/Рост (monthly) |
|---|---|---|
| Linear | 1.4 - 1.6 | 1.5 - 1.8 |
| Balanced | 1.6 - 1.9 | 1.8 - 2.2 |
| Aggressive | 1.8 - 2.2 | 2.0 - 2.4 |

Если коэффициент ушёл от диапазона — пересмотри состав. Если выбранная стратегия не клеится с реальными ценами — переход на соседнюю (например, Balanced → Aggressive).

## Запреты

- НЕ нарушать взаимоисключающие пары (`BS⊕FA`, `LB⊕LA`, `PF⊕PFP`, `ART⊕AR`).
- НЕ ставить AR без IT, NG без SY, PF/PFP без RP.
- **НЕ добавлять `IT` в тариф без `AR`.** `ART` (тестовая статья) сбор тем не требует.
- **НЕ добавлять `PA` без связки `SY`+`NG`** и НЕ забывать `PA`, если `SY`+`NG` в тарифе есть (обязательная связка).
- **НЕ добавлять отдельный `MT`, если в тарифе есть `SY`** — метатеги уже входят в структуру.
- **НЕ требовать `MT` как обязательную пару к BS.** BS работает с тем что есть. `MT` — только по полному гейту (см. RULES раздел 3).
- **НЕ класть услугу в тариф «для богатства каталога».** Услуга = решение конкретной проблемы. Если проблемы нет — услуги нет.
- **НЕ собирать Максимум как «все услуги подряд».** Максимум = 4-6 ключевых услуг под приоритетные проблемы. Развилки приоритета — см. RULES раздел 6.
- **НЕ включать `MT` без полного гейта** (нет `SY` + низкий бюджет + низкая конкуренция + метатеги совсем не сеошные). `MT` — не дефолтная услуга.
- НЕ добавлять услуги, которых нет в `TARIFFS.md` (актуальный список 18 услуг).
- НЕ использовать удалённые ID: `UX`, `SC`, `PL`, `SR` (как отдельный аудит), `MG`, `MF`, `MD` — они выпилены из линейки (`MF`/`MD` объединены в `MT`).
- НЕ забывать `price_note` для услуг с переменной ценой (`BS`). Без него клиент не поймёт почему именно эта цифра в смете.
- НЕ писать прозу для docx — только структурные данные.
- **НЕ использовать `rost` как ключ среднего тарифа.** Только `growth`. Это ключ JSON, не лейбл. Лейбл «Рост» отрисует скрипт сборки xlsx сам. Использование `rost` → скрипт молча пропустит вкладку.
- Длинное тире (—) и среднее (–) не использовать. Только дефис (-).
