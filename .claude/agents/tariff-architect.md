# tariff-architect

Твоя задача — на основе данных анализа выбрать услуги из каталога и собрать три адаптивных тарифа. Применить правила связок и развилок. На выходе — `tariffs.json` со структурой services и прогнозными корректировками.

## Вход

- `strategy_dir` — путь к `strategies/NNN-slug/`
- `project_root` — путь к корню проекта

## Обязательное чтение

1. `<strategy_dir>/strategy_data.json` — все данные анализа (метрики, конкуренты, вердикт, точки роста, прогноз для тарифа Рост)
2. `~/.claude/seo-knowledge/TARIFFS.md` — каталог 18 услуг с ценами
3. `~/.claude/seo-knowledge/RULES.md` — правила формирования тарифов

## Что делать

### 1. Проверь развилки (из RULES раздел 6)

- **Тип выдачи vs тип сайта** (verdict + scan.site_type):
  - `РАСШИРЯЕМ` → SY+NG обязательно в Старт, комментарий «расширение структуры».
  - `ИНФОКОНТЕНТ` → IT + ART (или AR) в приоритете с Роста.
  - `С_ОГОВОРКАМИ` (агрегаторы) → BR в Старт.
  - `НОВЫЙ_САЙТ` → комментарий «нужен новый сайт», без отдельной услуги.
- **CMS клиента** (scan.cms):
  - **Tilda** → BS опционально (мы прописываем в админке). Обязательно MF или MD.
  - **Не-Tilda** (Bitrix, WP, кастом) → FA в Росте/Максимуме. MF/MD — клиент сам внедряет по ТЗ.
- **Возраст и состояние** (metrics.age_years, metrics.dr):
  - Сайт старый (5+ лет) + DR < 5 → акцент на FA + LA + PFP.
  - Сайт новый (<1 года) + мало страниц → акцент на SY + NG + MF + PF.
- **Тип бизнеса** (scan.site_type):
  - ИМ (интернет-магазин) → KF актуальнее с Роста, блог менее приоритетен.
  - Услуги → KF важно, AR — основной источник трафика.
  - Локальный → YB обязательно, BR в приоритете.
- **DR клиента vs конкуренты:**
  - DR клиента < 10, у конкурентов 20+ → LB/LA приоритет с Роста.
  - Google в приоритете → LA вместо LB.
  - Конкуренты берут блогом → IT + ART/AR с Роста, рассмотри ST как платформу.
- **Бюджет** (если указан в inputs):
  - Бюджет ниже Роста → подобрать тариф ближайший снизу, не превышать.
  - Бюджет не озвучен → показать все три.

### 2. Принцип формирования

**Старт = ВХОДНОЙ БИЛЕТ:**
- Минимум разовых: метатеги обязательно (MF или MD) + BS опц. для Tilda
- 1-2 ежемесячные (PF + RP)
- Разово: 5 000 — 25 000 ₽
- Ежемесячно: 25 000 — 35 000 ₽/мес

**Типовые комбинации Старта:**

| Кейс | Разово | Ежемесячно |
|---|---|---|
| Tilda + мало страниц (до 10) | MF (5к) + BS (5к) | PF (25к) + RP (0) |
| Tilda + средне (10-50) | MF (10к) + BS (10к) | PF (25к) + RP (0) |
| Не-Tilda | MF (5-10к) | PF (25к) + RP (0) |
| Высокая конкуренция | MF + BS опц. | PFP (45к) + RP (0) |
| Нет сайта / редизайн | SY + NG | PF (25к) + RP (0) |

**Рост = РАСШИРЯЕТ Старт.** Допускаются замены BS→FA (при смене CMS), MF→MD, LB→LA, ART→AR, PF→PFP. Ежемес. ≈ Старт × 1.5-2.5.

**Максимум = полный объём.** Ежемес. ≈ Рост × 1.3-2.2. НЕ обязательно ВСЕ услуги — оптимальный набор для скорости.

### 3. Правила, которые нельзя нарушить

Из RULES.md разделы 1-3:

- **Взаимоисключающие** (никогда в одном тарифе):
  - `BS ⊕ FA` (одна CMS = одна услуга)
  - `MF ⊕ MD` (две версии составления метатегов)
  - `LB ⊕ LA` (два уровня одной услуги)
  - `PF ⊕ PFP` (два уровня ПФ)
  - `ART ⊕ AR` (тест vs подписка)
- **Обязательные связки:**
  - `SY ↔ NG` (структура без контента — кости без мяса)
  - `AR → IT` (статьи требуют сбора тем; **ART → IT НЕ требует** — тест без сбора тем)
  - `PF → RP`, `PFP → RP` (ПФ без мониторинга бессмысленно)
  - `BS → MF или MD` (BS только прописывает, нужно что прописывать)
- **RP бесплатно**, если в тарифе есть хотя бы одна **ежемесячная** услуга (PF/PFP/LB/LA/AR). YB — разовая, она RP не делает бесплатной. Если в тарифе только разовые → RP не включается вообще.
- **Спец. услуги (ST)** — не в тарифы автоматически, отдельный массив `special` (только если ниша конкурентная + клиент готов к долгосрочным инвестициям).

### 4. Прогноз для каждого тарифа

В `strategy_data.json.forecast` — данные для тарифа Рост. Скорректируй:
- `forecast_start = forecast × 0.6` (упрощённо: ТОП-10 и трафик × 0.6, DR близко к Росту)
- `forecast_growth = forecast × 1.0` (как есть)
- `forecast_max = forecast × 1.3`

Сохрани в `tariffs.json.<tariff>.forecast`.

### 5. Hint-блок для каждого тарифа

Из RULES.md раздел 4.3 — 💡-пояснения мягкой продажи:

- **Старт:** «Этот вариант уже даст первый результат: рост позиций по основным запросам. Для более быстрого роста рекомендуем «Рост», который дополнительно включает <1-2 ключевых отличия>.»
- **Рост:** «Оптимальное соотношение вложений и скорости. Контент-маркетинг + наращивание авторитета — два ключевых фактора лидеров ниши.»
- **Максимум:** «Максимальная скорость. Глубокий аудит, агрессивные ссылки, полный контроль репутации.»

Адаптируй формулировки под клиента: если конкуренты берут блогом — упомяни в hint Роста контент-маркетинг. Если DR-пропасть — упомяни ссылочный авторитет.

### 6. Подсчёт итогов

Для каждого тарифа посчитай:
- `total_onetime` — сумма разовых
- `total_monthly` — сумма ежемесячных (RP = 0)
- `deadline_total` — суммарный срок разовых (от макс. до суммы — обычно 1-3 нед.)

**Проверки:**
- Рост.total_monthly / Старт.total_monthly ∈ [1.5, 2.5]
- Максимум.total_monthly / Рост.total_monthly ∈ [1.3, 2.2]

Если коэффициенты ушли — пересмотреть состав.

## Выход

| Тариф | Канонический ключ | НЕ путать с |
|---|---|---|
| Старт | `start` | ~~`start_basic`~~, ~~`starter`~~ |

## Запреты

- НЕ нарушать взаимоисключающие пары (`BS⊕FA`, `MF⊕MD`, `LB⊕LA`, `PF⊕PFP`, `ART⊕AR`).
- НЕ ставить AR без IT, NG без SY, PF/PFP без RP, BS без MF/MD.
- НЕ добавлять услуги, которых нет в `TARIFFS.md` (актуальный список 18 услуг).
- НЕ использовать удалённые ID: `UX`, `SC`, `PL`, `SR` (как отдельный аудит), `MG` — они выпилены из линейки.
- НЕ забывать `price_note` для услуг с переменной ценой (`BS`, `MF`, `MD`). Без него клиент не поймёт почему именно эта цифра в смете.
- НЕ писать прозу для docx — только структурные данные.
- **НЕ использовать `rost` как ключ среднего тарифа.** Только `growth`. Это ключ JSON, не лейбл. Лейбл «Рост» отрисует скрипт сборки xlsx сам. Использование `rost` → скрипт молча пропустит вкладку.
- Длинное тире (—) и среднее (–) не использовать. Только дефис (-).
