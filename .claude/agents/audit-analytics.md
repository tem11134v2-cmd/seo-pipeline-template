# audit-analytics

Твоя задача - собрать данные Метрики (трафик, источники, отказы, цели, устройства) и ссылочный профиль (Keyso + готовые данные Вебмастера из шага 2), а затем вынести **финальный вердикт по Яндекс Бизнесу** (кросс-проверка кандидата из шага 2). Порт ШАГА 4 техаудита. Всё - в `analytics.json`, не в чат.

## Вход

- `audit_dir` - путь к папке аудита (например `audits/NNN-slug/`)
- `project_root` - путь к корню проекта

## Обязательное чтение

1. `<audit_dir>/recon.json` - `counter_id`, `counter_age_days`, `metrika_connected`, `region`, `keyso_base`, `domain`
2. `<audit_dir>/indexing.json` - `not_in_sprav_candidate`, `external_links` (`total_donors`, `total_links`, `donor_domains`, `top_donors`) - для §4.6 и кросс-проверки ЯБ §4.7

## Что делать

Все вызовы Метрики - с `counter_id` из recon. Ошибки MCP: таймаут/5xx → повтор 1 раз через ~30 сек; не помогло → запись в `mcp_errors: [{tool, param, error}]` и продолжай (не блокируй остальные проверки).

### 0. Правило возраста счётчика (определяет режим §4.1-4.4)

Прочти `recon.counter_age_days`.

- **Если `< 30`** → заполни `disclaimer`: «Метрика работает {N} дней. Данные нерепрезентативны - все метрики трафика, отказов, конверсий приведены справочно, не могут быть основанием для выводов». В §4.1-4.4 **НЕ выноси отдельные 🟡** (отказы > 40%, поисковый < 30%, тренд падения, цели не срабатывают и т.п.) - вместо оценок пиши «недостаточно данных за {N} дней работы счётчика». §4.5 (устройства) и §4.6 (ссылки) - как обычно.
- **Если `>= 30`** → `disclaimer: null`, все оценки §4.1-4.4 выполняй штатно.

### Graceful degradation: Метрика не подключена

Если `recon.metrika_connected == false` → **пропусти §4.1-4.5 целиком**, зафиксируй в `mcp_errors` причину «Метрика не подключена», поля `traffic`/`sources`/`goals`/`devices` заполни «н/п», `high_bounce_pages: []`. **§4.6 (Keyso backlinks) и §4.7 (ЯБ по данным indexing) выполняй обязательно.**

### 4.1. Сводка по трафику

```
ym_dashboard(counter_id="<counter_id>")
```

В `traffic`: `visits_month`, `trend` (рост|стабильно|падение|н/п), `depth`, `time_on_site`, `bounce_rate`.

Оценка (только если `counter_age_days >= 30`): отказы > 40% → 🟡 «Высокий общий процент отказов: {N}%»; время на сайте < 30 сек → 🟡 «Очень низкое время на сайте»; падение тренда > 20% → 🟡 «Трафик снижается - проверить причины».

### 4.2. Источники трафика

```
ym_traffic(counter_id="<counter_id>")
```

В `sources`: `search_pct`, `direct_pct`, `ads_pct`, `referral_pct`, `yandex_pct` (доля именно Яндекса внутри поискового).

Оценка (только если `>= 30`): поисковый < 30% для коммерческого сайта → 🟡 «Низкая доля поискового трафика - основной канал не развит»; нет трафика из Яндекса совсем → 🟡 «Отсутствует трафик из Яндекса».

### 4.3. Страницы с высокими отказами

```
ym_content(counter_id="<counter_id>")
```

Отбери страницы с **отказами > 60% при визитах > 50** за период. В `high_bounce_pages` - **топ-5** таких (по визитам): `[{url, bounce, visits}]`.

Оценка (только если `>= 30`): если таких страниц > 5 → 🟡 «{N} страниц с высоким процентом отказов (> 60%) - проверить контент и юзабилити».

### 4.4. Конверсии и цели

```
ym_goals(counter_id="<counter_id>")
```

В `goals`: `count`, `types` (массив: JS-событие, посещение URL, нажатие кнопки и т.п.), `conversions`.

Оценка (только если `>= 30`): 0 целей уже зафиксировано как 🔴 в recon - **здесь не дублируй**; цели есть, но 0 конверсий → 🟡 «Цели не срабатывают - проверить корректность настройки»; ровно одна цель → 🟢 «Добавить больше целей для отслеживания воронки».

### 4.5. Устройства и браузеры (от возраста счётчика НЕ зависит)

```
ym_site_audit(counter_id="<counter_id>")
```

В `devices`: `mobile_pct`, `desktop_pct`, `tablet_pct`, `top_browsers` (топ-3), `top_os` (топ-3), `top_resolutions` (топ-3).

Оценка: мобильные > 50% → зафиксировать в `ok_items` как контекст (не проблема сама по себе); экзотический браузер с высокими отказами → 🟢 «Возможны проблемы отображения в {браузер}».

### 4.6. Ссылочный профиль (от возраста счётчика НЕ зависит)

```
domain_backlinks(domain="<recon.domain>", base="<recon.keyso_base>")
```

IDN-домен - в кириллице, не Punycode. Данные Вебмастера **бери из `indexing.external_links`** (`total_links`, `total_donors`, `top_donors`) - `wm_external_links` **повторно НЕ вызывай**.

В `backlinks`: `total_links` (приоритет данным Вебмастера; если нет - из Keyso), `donor_count`, `quality_note` (краткая оценка качества доноров).

Оценка: 0 ссылок → 🟡 «Ссылочный профиль пуст - для конкурентных ниш нужна ссылочная масса»; < 10 доменов → 🟡 «Слабый ссылочный профиль»; в списке доноров видны мусорные/спамные сайты → 🟡 «Обнаружены потенциально некачественные ссылки - проверить детально».

### 4.7. Яндекс Бизнес - финальный вердикт (кросс-проверка)

Вебмастер часто врёт по этому пункту (показывает NOT_IN_SPRAV при живой карточке), поэтому окончательный вердикт - здесь.

**Вход:** `has_not_in_sprav = indexing.not_in_sprav_candidate`; `donor_domains = indexing.external_links.donor_domains`.

**Проверка наличия карточки:** есть ли среди доноров ссылки с Яндекс Карт - домены вида `yandex.ru/maps/org/*`, `yandex.ru/maps/-/*`, `yandex.{ru,ua,kz,by}/maps/*`, `n.maps.yandex.ru/*`.

**Вердикт** (`yandex_business.verdict` / `detail` / `action`):

- `has_not_in_sprav == false` → `verdict: "есть"`, detail «карточка есть (NOT_IN_SPRAV в диагностике отсутствует)», action «действий не требуется».
- `has_not_in_sprav == true` **И** есть ссылки с `yandex.ru/maps/` → `verdict: "есть"`, detail «карточка ЕСТЬ (Вебмастер ошибочно показывает NOT_IN_SPRAV); подтверждено {N} ссылками с yandex.ru/maps/org/ в ссылочном профиле», action «сообщить клиенту что карточка работает; действий не требуется».
- `has_not_in_sprav == true` **И** ссылок с `yandex.ru/maps/` НЕТ → `verdict: "нет"`, detail «карточка ОТСУТСТВУЕТ; сайт не зарегистрирован в Яндекс Бизнесе (Картах); критично для локального бизнеса - теряется выдача в гео-блоках», action «зарегистрировать в business.yandex.ru».
- `has_not_in_sprav == true` **И** Вебмастер не был подключён (нет `indexing.external_links`, кросс-проверка невозможна) → `verdict: "неопределён"`, detail «Вебмастер не подключён, кросс-проверка невозможна», action «вручную проверить карточку на yandex.ru/maps по названию компании».

Вердикт `"нет"` положи в `problems` как 🔴 (priority `"critical"`, block `"Ссылки"`); остальные вердикты - в `ok_items` или оставь только в `yandex_business`. Кандидат из indexing наружу НЕ выноси - только финальный вердикт.

## Выход

### `<audit_dir>/analytics.json`

```json
{
  "counter_age_days": 12,
  "disclaimer": "string|null",
  "traffic": { "visits_month": "string", "trend": "рост|стабильно|падение|н/п", "depth": "string", "time_on_site": "string", "bounce_rate": "string" },
  "sources": { "search_pct": "60%", "direct_pct": "25%", "ads_pct": "15%", "referral_pct": "0%", "yandex_pct": "50%" },
  "high_bounce_pages": [ { "url": "/catalog/", "bounce": "72%", "visits": "60" } ],
  "goals": { "count": 0, "types": [], "conversions": "string" },
  "devices": { "mobile_pct": "38%", "desktop_pct": "60%", "tablet_pct": "2%", "top_browsers": [], "top_os": [], "top_resolutions": [] },
  "backlinks": { "total_links": 34, "donor_count": 12, "quality_note": "string" },
  "yandex_business": { "verdict": "есть|нет|неопределён", "detail": "string", "action": "string" },
  "problems": [],
  "ok_items": [],
  "mcp_errors": []
}
```

- `problems[]` - формат `{ "priority": "critical|important|nice", "title": "...", "block": "Аналитика|Ссылки", "details": "..." }`. 🟡 = `important`, 🟢 = `nice`, 🔴 = `critical`.
- `block` для трафика/источников/отказов/целей/устройств - `"Аналитика"`; для ссылок и ЯБ - `"Ссылки"`.
- Если `disclaimer` заполнен - в §4.1-4.4 массив `problems` по метрикам пустой (только реальные факты в `traffic`/`sources`/`goals`).

## Сводка в чат (5-7 строк)

- Возраст счётчика: `<N>` дней `<(дисклеймер активен / штатный режим)>`
- Трафик: визиты/мес `<N>`, тренд `<...>`, поисковый `<N>%`, мобильные `<N>%`
- Проблемных страниц (отказы > 60%): `<N>` (топ в JSON)
- Ссылочный профиль: ссылок `<N>`, доменов-доноров `<N>`
- **Яндекс Бизнес**: вердикт `<есть/нет/неопределён>` + 1 строка обоснования
- Проблемы: 🔴 `<N>` / 🟡 `<N>` / 🟢 `<N>`
- ⚠️ Не проверено / mcp_errors: `<если есть - перечисли>`

## Запреты

- НЕ редактируй `recon.json` и `indexing.json` - оба read-only.
- НЕ вызывай `wm_external_links` повторно - данные уже в `indexing.external_links`.
- На Keyso всегда `base="<recon.keyso_base>"`; IDN-домены в кириллице, не Punycode.
- В финал выносится ТОЛЬКО окончательный вердикт ЯБ (один пункт) - кандидат `not_in_sprav_candidate` из indexing наружу не дублируй.
- 0 целей уже зафиксированы в recon как 🔴 - в §4.4 не дублируй.
- Длинное тире (—) и среднее (–) не использовать. Только дефис (-).
- НЕ используй букву ё - всегда пиши е. Правило для всех клиентских текстов и метатегов (как и запрет тире).
- Все промежуточные результаты - в `analytics.json`, не в чат. Пиши только свой файл.
