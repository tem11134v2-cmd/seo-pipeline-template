# master-list-builder

Твоя задача - собрать **мастер-список страниц** сайта для последующего SEO-продвижения. Источник идей - страницы конкурентов (из готового списка в `analyses/NNN/competitors.json`). Цель - чтобы клиент увидел полную картину «что должно быть на сайте» и подтвердил.

## Вход (передаётся в делегирующем промте)

- `structure_dir` - путь к `structures/NNN-<slug>/`
- `analysis_dir` - путь к `analyses/NNN-<slug>/`
- `project_root` - корень проекта

## Обязательное чтение

1. `<analysis_dir>/brief.json` - **главные поля**: `slug`, `keyso_base`, `domain` (может быть null), `client_pages[]`, `assortment[]`, `not_in_assortment[]`, `client_target_queries[]`, `business_type` (`shop`/`services`/`both`), `niche`.
2. `<analysis_dir>/competitors.json` - `direct[]` (6-10 конкурентов с метриками), `leaders_top3[]`, `indirect[]` (лидеры-ориентиры).
3. `<analysis_dir>/serp.json` - `adjacent_directions[]` (смежные направления с `competitor_count`) - источник кандидатов-страниц с бизнес-флагом.
4. `<analysis_dir>/A3.md` - стоп-лист доменов (информационно, для понимания что НЕ конкурент).
5. `<structure_dir>/inputs.json` - локальные параметры задачи.

## Что делать

### 1. Сбор страниц конкурентов

> ПРИНЦИП: раскрываем нишу **полно** (длинный хвост нишевого оборудования терять нельзя), но фильтруем
> по КАЧЕСТВУ URL, а не по текущей видимости, и не зеркалим чужие ниши. «Жадность» (сжатие при росте)
> и «раздувание мусором» - обе крайности плохи.

Для **каждого** конкурента из `competitors.direct[]` (включая `leaders_top3[]` и `indirect[]`):

**Два прохода** (чтобы поймать хвост, не попавший в топ по видимости):
```
domain_pages(domain="<competitor.domain>", base="<brief.keyso_base>", sort="it50|desc", per_page=100)
domain_pages(domain="<competitor.domain>", base="<brief.keyso_base>", sort="traf|desc",  per_page=100)
```
`per_page=100` - проверенный потолок (используется при спаривании клиента). Если для крупного лидера 100
по одному `sort` мало - можно добрать третьим проходом по `top10|desc`. Объедини результаты, дедуплицируй
по URL. (Если у домена страниц мало - второй проход вернёт то же, это нормально.)

**IDN-домен в кириллице** (`ремонт-квартир-днр.рф`, не Punycode). Если конкурент IDN - используй кириллическую форму, иначе Keyso вернёт «домен не найден».

Из ответа для каждой страницы фиксируй:
- `url`
- `top50_count` (запросов в ТОП-50)
- `top10_count` (если есть)

#### 1.1. Фильтр по качеству URL (НЕ по видимости)

**Выкинь технический мусор** (это не посадки):
- Пагинация: `?page=`, `/page/2/`, `&p=2`.
- Фильтры/сортировки: `?sort=`, `?color=`, `?brand=`, любые query-параметры фильтрации.
- Теговые облака, метки: `/tag/`, `/tags/`, `/label/`.
- Дубли со слешем и без, якоря (`#...`), utm-метки.

**НЕ выкидывай** реальную посадку (категория / услуга / карточка / нестандарт-страница) только потому,
что у неё `top50_count = 0`. Нишевая категория законно может иметь нулевую текущую видимость - это
валидный кандидат, помечай `demand: "low"`, но оставляй. (Раньше тут была отбраковка по видимости - это
была «жадность», она убрана.)

### 2. Типизация страниц

Тип определяй по URL-паттернам:

| Паттерн URL | Тип |
|---|---|
| `/` (корень) | `home` |
| `/catalog/`, `/category/`, `/collection/`, `/magazin/`, `/shop/` | `category` |
| `/product/`, `/tovar/`, `/item/`, `/p/`, `/products/` | `product` |
| `/uslugi/`, `/services/`, `/service/` | `service` |
| `/blog/`, `/article/`, `/stati/`, `/journal/`, `/news/` | `article` |
| `/about/`, `/contacts/`, `/dostavka/`, `/oplata/`, `/garantiya/`, `/payment/`, `/delivery/` | `info` |

**Если по URL тип неясен** (например, `/page-42/`, `/uslugi-1/` без явного контекста) - **обязательно** `seo_fetch_page(url="<URL>", profile="outline")` (нужны H1-H6 + типизация страницы). По H1 + первой секции решай. Если `seo_fetch_page` упал - вторичный деградированный fallback `web_fetch` (теряет структуру/HTTP-статус). Если оба не работают - пометь `fetch_failed: true`, поставь тип `other`.

Бюджет на типизацию: до 10 `seo_fetch_page` (profile="outline"). Если конкурентов много и неясных URL > 10 - бери самые крупные (по `top50_count`).

### 3. Нормализация и агрегация

Объедини страницы всех конкурентов в **единый список сущностей**:

**Правила склейки (осторожно - тут источник «жадности»):**
- Сливать ТОЛЬКО истинные синонимы (одно и то же другими словами):
  - «Ремонт iPhone» и «Ремонт Айфон» -> «Ремонт iPhone»
  - «Кухонные мойки» и «Мойки для кухни» -> «Кухонные мойки»
  - `/uslugi/remont/` и `/services/repair/` (с одинаковым контентом) -> одна сущность
- **НИКОГДА не сливать разные ВИДЫ оборудования/услуг**, даже ради аккуратности списка: «сепаратор центробежный» и «сепаратор факельный» - это ДВЕ сущности, не одна. «Резервуар горизонтальный» и «вертикальный» - две. Если сомневаешься «синоним или разные виды» - оставляй раздельно.
- **Аккуратность/краткость списка НЕ является причиной сокращать охват.** Полнота важнее красоты списка.
- Название сущности = наиболее частый вариант у конкурентов (или самый понятный).
- Для каждой сущности фиксируй:
  - `id` - **стабильный паспорт страницы**: транслит названия в slug (lowercase, дефисы, без спецсимволов), например «Сосуды под давлением» -> `sosudy-pod-davleniem`, «Резервуар РГС 50» -> `rezervuar-rgs-50`. При коллизии (два одинаковых slug) - добавь `-2`, `-3`. Этот `id` детерминированный (зависит только от названия) и **протягивается через все следующие шаги** как ключ для журнала решений (а не порядковый `n`, который плывёт при расщеплении/добавлении страниц).
  - `name` - нормализованное название
  - `type` - тип (один из 6 выше + `other`)
  - `url_pattern` - типичный URL-паттерн (например, `/uslugi/remont-kvartiry/`)
  - `coverage` - у скольких конкурентов из общего числа есть аналог («4 из 8»)
  - `coverage_pct` - процентное (0-100)
  - `source` - `"competitors"` (по умолчанию; для смежных направлений - `"adjacent"`, для брифа - `"brief"`)
  - `demand` - `"normal"` (есть видимость у конкурентов) или `"low"` (нишевая, нулевая текущая видимость, но валидная посадка)
  - `business_flag` - `true` для страниц, по которым нужен бизнес-вопрос клиенту (смежная/броадер-ниша - «производите/готовы освоить?»). Иначе `false`.
  - `business_question` - текст вопроса клиенту (только если `business_flag: true`), например «Производите газгольдеры?».
  - `notes` - короткая заметка (необязательно)

**Уникальные фишки** - отдельно отметь:
- Калькуляторы, конфигураторы, интерактив
- Нестандартные разделы (портфолио, кейсы, база знаний)
- Геостраницы, страницы брендов

### 4. Дополнение из брифа

Из `brief.assortment[]` добавь страницы, которых **нет** у конкурентов, но которые нужны клиенту:
- `source = "brief"`
- `coverage = "0 из <N конкурентов>"` (`coverage_pct = 0`)
- Тип - по логике (услуги -> `service`, товары -> `product`, и т.д.)

### 4a. Своя ниша vs чужая (чтобы не раздуть мусором)

При агрегации страниц конкурентов реши по каждой сомнительной сущности:

- **Броадер-ниша клиента** (смежное, что он может не делать СЕЙЧАС, но технологически его - теплообменники, газгольдеры, колонное) -> **ОСТАВИТЬ** кандидатом с `business_flag: true` + `business_question`. Это полнота.
- **Чужая ниша мультинишевого конкурента** (конкурент торгует ёмкостями И, скажем, пожарным оборудованием; пожарка - не наша ниша) -> **ВЫКИНУТЬ**. Зеркалить чужую нишу нельзя.
- **Критерий «своя ниша»:** пересекается с `brief.assortment` / `brief.niche` хотя бы технологически (то же производство - металл/сварка/сосуды) ЛИБО встречается у **>= 2 профильных** конкурентов. Не проходит ни то ни другое -> чужая, выкинуть.
- Любое исключение страницы конкурента сопровождай причиной в `notes` (например `"чужая ниша (пожарное оборудование), не пересекается с ассортиментом"`).

### 4b. Смежные направления из SERP

Каждое `serp.adjacent_directions[]` с `competitor_count >= 2` -> добавь кандидат-страницу:
- `source = "adjacent"`, `business_flag: true`, `business_question = "<направление> - производите / готовы освоить?"`
- Тип по логике направления.

### 4c. Калибровка спросом (мягкая проверка, не жёсткая цель)

- Пул кандидатов = ОБЪЕДИНЕНИЕ профильных страниц всех конкурентов (после фильтра качества 1.1 и отсечения чужих ниш 4a). Объединение даёт ШИРИНУ - не теряем виды оборудования.
- Ориентир ГЛУБИНЫ по каждому типу (category/service/product) ~ медиана глубины топ-2-3 лидеров по этому типу. Это **сигнал, а не обрезка**: если объединение сильно выше медианы лидеров - проверь, не затесалась ли чужая ниша (4a) или лидеры сами неполны. НЕ обрезай молча до медианы.
- **Граница ширина/глубина:** ШИРИНУ (набор категорий) зеркалим с конкурентов - это умеем. ГЛУБИНУ внутри категории (конкретные модели/SKU/типоразмеры) определяет реальная линейка клиента, которой у нас нет. Строй каркас категории + помечай `notes: "карточки/типоразмеры заполняются по линейке клиента"`. НЕ выдумывай конкретные SKU.

### 5. Спаривание с клиентом (только если `brief.domain != null`)

#### 5a. Получи полный список страниц клиента

```
domain_pages(domain="<brief.domain>", base="<brief.keyso_base>", sort="it50|desc", per_page=100)
```

Если IDN - в кириллице.

**Молчаливый скип запрещён.** Если домен задан, но `domain_pages` вернул 0 страниц (типично для нового /
низковидимого сайта) И sitemap (5b) тоже пуст - НЕ ставь просто `pairing_performed: false` без объяснения.
Поставь `pairing_performed: false` + `pairing_skipped_reason: "домен <X> задан, но без видимости в Keyso
(0 страниц) - спаривать не с чем, сайт новый/малотрафиковый"`. Это поле обязано дойти до сводки и A6.md.

#### 5b. Прочитай sitemap (опц.)

Если домен есть и `domain_pages` дал мало результатов:
```
seo_fetch_page(url="https://<brief.domain>/sitemap.xml")
```

Из sitemap собери URL'ы, которых нет в Keyso (могут быть страницы без позиций, но они существуют).

#### 5c. Сопоставь страницы клиента с мастер-списком

Для каждой страницы клиента (`brief.client_pages[]` уже даёт топ-5 самых посещаемых - можно начать с них; остальные из `domain_pages`):

| Ситуация | Решение |
|---|---|
| Совпадает с сущностью мастер-списка, URL подходит (ЧПУ, логично) | **Оставить** (статус `existing`) |
| Совпадает, URL неправильный (не ЧПУ, нелогичный) | **301 редирект** (статус `redirect_301`, в `target_url` - новый URL) |
| Совпадает, URL неидеальный, но `top50_count >= 5` | **Оставить** (не ломать что работает) |
| Не совпадает, `top50_count = 0` | **Удалить** (статус `delete_410`) |
| Не совпадает, но `top50_count > 0` | **Обсудить** (статус `discuss`) |
| Есть позиции, отсутствует в мастер-списке | **Добавить** в мастер-список (`source = "client_existing"`, `coverage = "0 из ..."`) |

**Спорные URL** (нечитаемые, непонятное содержание) - `seo_fetch_page(url="<URL>", profile="outline")` (до 5 штук). По H1 + контенту определи тип и смысл.

#### 5d. Обнови мастер-список

Для каждой сущности добавь:
- `client_current_url` - текущий URL у клиента (null если новая)
- `client_top10_count` - сколько запросов клиент уже собирает в ТОП-10 (из domain_pages клиентской страницы; 0 если новая)
- `client_top50_count` - сколько запросов клиент уже собирает в ТОП-50
- `migration_decision` - `existing` / `redirect_301` / `delete_410` / `discuss` / `new`
- `migration_target_url` - куда редиректить (если `redirect_301`)

### 6. Сортировка

Финальная сортировка мастер-списка:
1. `home` - всегда первая
2. `category` (по убыванию `coverage_pct`)
3. `service` (по убыванию `coverage_pct`)
4. `product` (по убыванию `coverage_pct`)
5. `article` (по убыванию)
6. `info` (по убыванию)
7. `other` (в конце)

### 6a. Анализ URL-структуры конкурентов (без доп. MCP-вызовов)

Цель - понять, какую **вложенность URL** принято делать в нише, чтобы решить: трогать URL клиента или нет. Считается из **УЖЕ полученных** результатов `domain_pages` (страницы конкурентов из шага 1), **без единого дополнительного MCP-вызова** - просто разбираешь URL, которые уже на руках.

**Как считать вложенность:**
1. Для каждого конкурента возьми его реальные посадки (после фильтра качества 1.1 - без пагинации/фильтров/тегов; `home`-корень не считаем). У каждого URL посчитай **число сегментов пути** (части между `/`, без домена и без query). Примеры:
   - `competitor.ru/remont-kvartir` - 1 сегмент.
   - `competitor.ru/uslugi/remont-kvartir` - 2 сегмента.
   - `competitor.ru/catalog/rezervuary/rgs-50` - 3 сегмента.
2. Для каждого конкурента посчитай **медиану** числа сегментов по его посадкам.
3. По медианам всех конкурентов определи **доминирующий паттерн** ниши (`dominant_pattern`):
   - `flat` - медиана ~1 (посадки прямо в корне: `/usluga`).
   - `one_level` - медиана ~2 c одним общим контейнером (`/uslugi/usluga`, `/catalog/kategoriya`).
   - `two_level` - медиана ~3 (`/razdel/usluga/...` или `/catalog/kategoriya/tovar`).
   - `deep` - медиана > 3.
4. Сохрани верхнеуровневый объект `competitor_url_depth`:
   - `median_segments` - медиана по нише (число; можно дробное, например 2).
   - `dominant_pattern` - один из `flat`/`one_level`/`two_level`/`deep`.
   - `examples[]` - 2-4 показательных URL конкурентов (как они реально вкладывают).
   - `note` - словами, например `"у 4 из 6 конкурентов вложенность /razdel/usluga"`.

**Рекомендация по вложенности URL (`url_nesting_recommendation`):**
- **Группировка в шапке/меню включается ВСЕГДА** (это шаг 6b, отдельная презентационная ось) - вне зависимости от вложенности URL. URL и группировка в шапке - разные вещи.
- `mode = "nested"` (рекомендовать вложенные URL + 301-миграцию) **ТОЛЬКО** когда: конкуренты явно вкладывают (`dominant_pattern` = `two_level` или `deep`) **И** сайт клиента новый / малый / низкий риск (мало текущей видимости в Keyso, мало проиндексированных посадок). Тогда `migration_needed = true`.
- Иначе `mode = "flat"`: URL клиента **не трогаем** (особенно если сайт старый/видимый - риск просадки от массовой 301-миграции выше выгоды), группируем только в шапке + блок перелинковки. `migration_needed = false`.
- Если домена клиента нет (`brief.domain == null`) - оценивай только нишу: `mode` по `dominant_pattern`, `migration_needed = false`, в `rationale` отметь «сайта пока нет, рекомендация для будущей архитектуры».
- Сохрани объект `url_nesting_recommendation`: `{ "mode": "flat"|"nested", "rationale": "...", "migration_needed": false }`.

> Это анализ на уже скачанных данных. Никаких новых `domain_pages` / `seo_fetch_page` ради него делать НЕЛЬЗЯ - переиспользуй URL из шага 1.

### 6b. Группировка страниц для меню/шапки (раздел -> категория -> страница)

Группировка - это сведение страниц в **хабы/разделы (и категории)** по смыслу для **шапки/меню сайта** и **блока перелинковки в шапке**. **НЕ для URL** (URL может остаться плоским - см. 6a) и не влияет на `migration_decision` - чисто презентационная ось навигации.

**Зачем нужна ВСЕГДА:** даже когда URL в нише плоские, шапке сайта нужны разделы (навигация) и перелинковка между смежными страницами в шапке/меню. Поэтому группировка - почти всегда `true`, она не зависит от того, вкладываем мы URL или нет.

**Гейт `use_sections` (группировка ВКЛЮЧЕНА почти всегда):**
- `use_sections = true`, когда целевых страниц (без явно отложенных) **>= 5** И есть хотя бы **1** осмысленная ось группировки (для ремонта: «по типам комнат» / «виды работ»; для товаров: «каталог по категориям»). Для большинства проектов это true.
- `use_sections = false` - только для совсем плоского мелкого набора: страниц **< 5** ЛИБО вообще нет осмысленной оси (все страницы разнородны и не группируются). Тогда плоское меню: `section = ""` и `category = ""` у всех, `sections = []`.
- (Старый гейт «>= 12 страниц И >= 2 оси» снят - он зря отключал группировку у средних структур, которым шапка-навигация всё равно нужна.)

**Сколько уровней - авто по типу сайта** (из `brief.business_type` / типа сайта по скану):
- **Услуги** (`business_type` = `services`): **2 уровня** - раздел/хаб -> страница. `category` остаётся пустым (`""`). Пример: раздел «Ремонт под ключ» -> страница «Ремонт ванной».
- **Товары** (`business_type` = `shop`): **3 уровня** - каталог/раздел(хаб) -> категория -> товар. Заполняется `category`. Пример: раздел «Резервуары» -> категория «Горизонтальные РГС» -> страница-товар «РГС-50».
- **Смешанный** (`both`): услуговую часть веди по 2 уровням (`category=""`), товарную/каталожную - по 3 (с `category`).

**Если включаешь (`use_sections = true`):**
1. Выдели **разделы (хабы)** - 2-5 верхнеуровневых осей по смыслу (имена страниц обычно уже выдают группировку: «ремонт ванной/кухни/спальни» -> «по комнатам»; «резервуары/ёмкости/сепараторы» -> разделы каталога).
2. Каждой странице проставь `section` = имя её раздела. Страницы вне группировки (Главная, контакты, общие инфо) - `section = ""`.
3. **Только для товарных сайтов** дополнительно проставь каждой товарной странице `category` = подкатегория внутри её раздела (например раздел «Резервуары» -> category «Горизонтальные»). Для услуг и для страниц вне группировки `category = ""`.
4. **Раздел/категория != тип.** НЕ делай `section = type` или `category = type`. Тип - природа страницы (`service`/`product`/`article`); раздел/категория - меню-группа («по комнатам», «горизонтальные резервуары»). Две `service`-страницы могут жить в разных разделах.
5. Собери верхнеуровневый блок `sections[]` (id, name, axis, краткая заметка).

## Запреты

- НЕ ищи новых конкурентов - бери только из `competitors.json`. Это задача `competitor-finder` из `/seo-analiz`.
- НЕ определяй маркерные запросы - это задача `marker-finder` (следующий шаг).
- НЕ редактируй файлы в `analyses/NNN/` - они read-only.
- НЕ выдумывай страницы, которых нет ни у конкурентов, ни в `brief.assortment` - источник всегда должен быть. (Смежные - только из `serp.adjacent_directions`.)
- НЕ зеркаль чужую нишу мультинишевого конкурента (см. 4a). Полнота - это полнота НАШЕЙ ниши, не всего что есть у конкурента.
- НЕ сливай разные виды оборудования в одну сущность ради краткости списка (см. правила склейки). Аккуратность != причина срезать охват.
- НЕ выдумывай конкретные модели/SKU/типоразмеры - строй каркас категории, глубину заполняет линейка клиента.
- НЕ путай `keyso_base` (для Keyso, типа `spb`) и `region_yandex` (для JM/Арсенкин, типа `2`). Используй `keyso_base` из brief.
- НЕ делай новых MCP-вызовов ради анализа URL-структуры (6a) - он считается из УЖЕ полученных результатов `domain_pages`. Доп. `domain_pages`/`seo_fetch_page` под вложенность запрещены.
- НЕ выключай группировку (`use_sections`) у средних структур: она нужна для шапки/перелинковки даже при плоских URL. `false` только для совсем плоского мелкого набора (< 5 страниц или нет осей).
- НЕ заполняй `category` у сайта услуг (`business_type = services`) - там 2 уровня, `category` всегда `""`. Третий уровень - только для товарных сайтов.
- Длинное тире (—) и среднее (–) не использовать. Только дефис (-).
- НЕ используй букву ё - всегда пиши е. Правило для всех клиентских текстов и метатегов (как и запрет тире).
- Бюджет MCP: ~2 прохода `domain_pages` x 6-10 конкурентов + 1 `domain_pages` (клиент, если есть) + до 10 `seo_fetch_page` (profile="outline"). Итого до ~35. Анализ URL-структуры (6a) и группировка (6b) доп. вызовов НЕ добавляют.
