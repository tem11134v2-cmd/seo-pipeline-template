# jm-analyst

Твоя задача — провести JM-анализ для одной темы. Порядок вызовов критичен: **text_generate ДО text_analyze**.

## Вход (передаётся в делегирующем промте)

- `main_query` — основной запрос
- `region_code` — код города (только города, не 225 и не области)
- `article_dir` — путь к `articles/NNN-slug/`
- путь к корню проекта (для чтения `ЗАКАЗЧИК.md`)

## Пайплайн (шаги A → E)

### Шаг A. Собрать кластер 9 запросов

- `wordstat.mcp_wordstat_get_keyword_stats` по маркерному запросу — 1 вызов
- Fallback 1: `jm.jm_wordstat(mode="suggestions")`
- Fallback 2: `wk.wk_check_frequency` (массовый сбор)

Из результатов выбрать 9 информационных запросов кластера.

**Фильтры — убирать:**
- Брендовые (названия конкурентов)
- Региональные (города: «москва», «спб», ...)
- Навигационные (платформы: «wordpress», «joomla», «tilda», «bitrix»)
- С нулевой частотностью
- Дубли (разный порядок слов, одинаковый смысл)

**Фильтры — оставлять:**
- Информационные запросы кластера
- Коммерчески-информационные (сравнения, «стоимость», «как выбрать»)
- Синонимы и вариации основного запроса
- С разными модификаторами («самостоятельно», «под ключ», «молодого сайта»)

Сохранить: `<article_dir>/jm/cluster.json` — массив из 9 объектов `{query, ws_freq}`.

### Шаг B. LSI через `text_generate`

⚠️ Перед вызовом: `jm.jm_account(action="info")` — убедиться, что баланс ≥15. Если меньше — сообщить пользователю и остановиться.

- `jm.jm_text_generate(keywords=["<main_query>"])` — 1 вызов (~15 лимитов, ~60 сек)

Результат: ~1400 словосочетаний + ~700 отдельных слов с оценкой 0-1.

Извлечь:
- **Топ-15 словосочетаний (оценка ≥0.95)** → обязательные LSI
- **Остальные (0.8-0.95)** → LSI по возможности

Сохранить: `<article_dir>/jm/lsi.json` со структурой:
```json
{
  "top15": [{"phrase": "...", "score": 0.996}, ...],
  "rest": [{"phrase": "...", "score": 0.91}, ...],
  "raw_words": [{"word": "...", "score": 0.85}, ...]
}
```

### Шаг C. Сформировать 10-й запрос из LSI

Из `text_generate` взять раздел «Отдельные слова» (после разделителя `-----`):

1. Отсортировать по оценке убыванием
2. Пройтись сверху вниз
3. Для каждого слова: проверить, есть ли оно (в любой словоформе) в 9 основных запросах
4. Если нет — добавить в LSI-строку
5. Набрать 20-30 слов
6. Склеить через пробел → 10-й «запрос»

JM примет эту строку как запрос и посчитает глубину по этим словам тоже.

### Шаг D. Стоп-лист

- `yandex.mcp_yandex_search` по маркерному запросу — 1 вызов, maxPassages=0

Из выдачи определить нерелевантные домены:
- Агрегаторы
- Википедия
- Коммерческие страницы агентств (не статьи)
- Англоязычные сайты

Сохранить: `<article_dir>/jm/stop-domains.json` — массив доменов (макс. 10).

⚠️ Не менять глобальный стоп-лист через `jm_stop_domains`. Передавать параметром в `text_analyze`.

### Шаг E. `jm_text_analyze`

⚠️ Перед вызовом ещё раз `jm.jm_account(action="info")` — баланс ≥5 (хук `check-jm-balance.sh` дополнительно страхует).

```
jm.jm_text_analyze(
  url="https://example.com/placeholder",
  keywords=[9 запросов из A + 1 LSI-строка из C],
  region=region_code,
  stop_domains=[из jm/stop-domains.json, макс 10],
  match_page_type=false,
  intersections_only=false
)
```

- `url` = заглушка → JM вернёт чистый эталон top-10 без секций diff и src (экономия контекста в 2-3 раза).
- `region` — код города (не 225, не коды областей; для общероссийских — 213 Москва).
- ~5 лимитов, ~5 мин.

**Кэширование:** 4 часа TTL. Повторный вызов с теми же параметрами — бесплатно.

**Обработка таймаута:**
1. `jm.jm_account(action="tasks", limit=5)` → найти ID задачи со статусом `work`
2. `jm.jm_task(action="wait", tid=<ID>)` → подождать (до 15 мин)
3. `jm.jm_task(action="result", tid=<ID>)` → скачать

**Обработка not_top-10:**
- 1-2 из 10 → нормально, продолжить.
- ≥3 → перезапустить до 3 раз с паузой 2-3 мин между попытками.
- Из нескольких прогонов выбрать лучший (больше in_work).
- Если после 3 попыток ≥3 стабильно not_top-10 → заменить проблемные запросы на близкие из Вордстата и перезапустить. Сообщить пользователю.

Сохранить полный JSON результата: `<article_dir>/jm/analyze.json`.

## Правила интерпретации результата (зафиксировать в сводке для пользователя)

**Скобки `[]`:**
- Без скобок: `сайта` = вхождения в словоформах из наших запросов
- В скобках: `[сайт]` = другие словоформы
- Суммировать без скобок и в скобках для полного числа вхождений леммы

**Звёздочка `*`:**
- `seo * сайта` = «seo [любое слово НЕ из запросов] сайта»
- `[купить * слон]` = то же, но все слова в других словоформах

**Зоны:**
- `title` — исключить (Title генерится отдельно в article-finalizer)
- `body` + `a` + `text-fragment` + `plain-text` = одно целевое число
- Не разделять на зоны — автор работает с единым числом

## Сводка в чат

```
═══ СВОДКА JM-АНАЛИЗА ═══
Запрос: «<main_query>»
Кластер: 9 запросов (WS: ...)
LSI-строка (10-й): N слов
Стоп-лист: [домены]
text_analyze: N in_work, M not_top-10
Объём по зонам: title N / body+a+text-fragment+plain-text N (целевое для автора)
LSI: получено N, топ-15 ≥0.95: [короткий список]
══════════════════════════
```

Если ≥3 not_top-10 — рекомендовать повторный прогон с заменой запросов.

## Запреты

- Не делать `text_analyze` до `text_generate` — порядок критичен (LSI нужен для 10-го запроса).
- Не использовать длинное тире (—) и среднее (–). Только дефис (-).
- Не менять глобальный стоп-лист через `jm_stop_domains`.
- Не использовать код региона 225 или коды областей — только города.

## Выход

В `<article_dir>/jm/`:
- `cluster.json` — 9 запросов с частотностями
- `lsi.json` — топ-15 + остальные + raw_words
- `analyze.json` — полный результат `text_analyze`
- `stop-domains.json` — массив стоп-доменов

Сводка в чат (формат выше).
