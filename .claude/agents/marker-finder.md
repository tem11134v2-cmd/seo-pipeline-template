# marker-finder

Твоя задача - для каждой страницы из мастер-списка определить **маркерный запрос** - самый частотный целевой запрос, по которому эта страница должна продвигаться. Это «якорь» страницы; на следующем шаге JM расширит каждый маркер до топ-30.

## Вход

- `structure_dir` - путь к `structures/NNN-<slug>/`
- `analysis_dir` - путь к `analyses/NNN-<slug>/`
- `project_root` - корень проекта

## Обязательное чтение

1. `<structure_dir>/master_list.json` - список страниц от `master-list-builder`.
2. `<analysis_dir>/brief.json` - `keyso_base`, `client_target_queries[]`, `niche`, `region`.
3. `<analysis_dir>/competitors.json` - `direct[]` (для каскада), `leaders_top3[]` (первый источник).
4. `<analysis_dir>/A3.md` - стоп-лист (информационно, для понимания «не оттуда»).

## Что делать

### 1. Для каждой страницы из master_list

#### Информационные страницы

Если `type == "info"` (`/about/`, `/contacts/`, `/delivery/`, `/oplata/`, и т.п.) - **маркер не нужен**. Поставь `marker = null`, `ws_exact = null`, `source = "none (info page)"`. Эти страницы не продвигаются по запросам.

#### Главная

`type == "home"` - маркер = основной запрос ниши + регион:
- Если `brief.client_target_queries` непуст и первый запрос содержит регион (`«ремонт квартир спб»`) - бери его, проверь через `keyword_info(keyword="...", base="<keyso_base>")`.
- Иначе сформулируй маркер из ниши + региона: `"<niche> <region-в-локативе>"`. Например `niche="ремонт квартир", region="Санкт-Петербург"` -> `"ремонт квартир спб"`.
- Проверь через `keyword_info` - если частотность > 0, это маркер.
- Если 0 - попробуй варианты через `keyword_similar`.

#### Категории / Услуги / Товары

##### 1.1. Основной путь - через конкурента-лидера

1. Найди аналогичную страницу у одного из `leaders_top3[]`. URL-паттерн из `master_list.pages[i].url_pattern` + `competitors_with_page` подсказывают где искать.
2. Если в `master_list.pages[i].competitors_with_page` указан лидер - бери его URL.
3. Получи запросы страницы:
```
domain_keywords(domain="<leader.domain>", url="<leader_url>", base="<keyso_base>")
```
4. Отфильтруй:
   - Только с `ws_exact > 0` (точная частотность)
   - Исключи бренд лидера (если запрос содержит название его домена/бренда)
   - Исключи откровенно нерелевантные запросы (другие услуги)
5. Самый верхний по `ws_exact` = **маркер**.

##### 1.2. Фолбэк 1 - другие конкуренты

Если у лидера нет аналогичной страницы (или `domain_keywords` вернул пусто) - перебери остальных конкурентов из `master_list.pages[i].competitors_with_page` по убыванию метрик (`competitors.direct[]` отсортирован по `top50_count` или `top10_count`).

Для каждого - `domain_keywords(<их домен>, url=...)`. Первый успешный -> маркер.

##### 1.3. Фолбэк 2 - keyword_info по названию

Если у никого нет аналога (`master_list.pages[i].source == "brief"` или клиентская новая) - проверь название страницы как запрос:
```
keyword_info(keyword="<page.name в lowercase>", base="<keyso_base>")
```
Если `ws_exact > 0` и в SERP сайты нашего типа - это маркер.

##### 1.4. Фолбэк 3 - keyword_similar

Если `keyword_info` показал 0 - ищи синоним:
```
keyword_similar(keyword="<page.name>", base="<keyso_base>")
```
Из результатов выбери самый близкий по смыслу с `ws_exact > 0`.

##### 1.5. Фолбэк 4 - ручное

Ничего не нашлось - назначь маркер сам по логике ниши (например, `"<page.name> <region-локатив>"`). **Обязательно пометь:** `"manual_warning": "без данных конкурентов - проверить вручную"`.

### 2. Резервный источник частотности

Если **Keyso не даёт частотность** по найденному маркеру (`keyword_info` вернул 0 или нет в базе) - переключись на резерв.

**Вариант A (пакетный, предпочтительный):**
Собери все маркеры с проблемной частотностью в массив:
```
jm_wordstat(mode="frequency", keywords=["маркер1", "маркер2", ...], region=0, freq_types=["base", "exact"])
```
`region=0` = вся страна (для предпроектного построения структуры достаточно).

**Вариант B (массовый, до 1000 фраз):**
```
wk_check_frequency(keywords=["маркер1", "маркер2", ...])
```

Используй резерв только пакетом - один-два вызова на все проблемные маркеры (не по одному).

**Фиксируй ОБЕ частотности (5.2):** `ws_exact` (точная, «в кавычках») И `ws_base` (базовая Wordstat). keyword_info отдаёт обе; jm_wordstat - через `freq_types=["base","exact"]`. `ws_base` критичен downstream: для B2B `ws_exact` часто 0 при живом `ws_base`, и semantic-expander решает «расширять или нет» именно по base (иначе живые страницы уходят голыми).

В записи каждой страницы фиксируй `frequency_source`:
- `"keyso"` - данные из Keyso (основной)
- `"jm_wordstat"` - резерв через JM
- `"wk_check_frequency"` - резерв через WK

### 3. Обязательная проверка коммерциализации маркеров

**Цель.** Не допустить классическую ошибку: «маркер выбран, частотность ОК, а в ТОП-10 - блог-статьи». Если выдача информационная, а мы делаем коммерческую посадку (категория/услуга/товар) - **страница не выйдет в ТОП** даже с идеальной семантикой. Лучше узнать это до того как потратили JM-лимит на расширение.

#### 3.1. Пакетный вызов arsenkin_commerce

Собери все маркеры **коммерческих страниц** (`type ∈ {home, category, service, product}`, `marker != null`) в один массив. Информационные (`type == "info"`) пропускаем - там коммерциализация не важна.

Один вызов на всю задачу:
```
arsenkin_commerce(
  queries=["маркер1", "маркер2", ..., "маркерN"],
  se=1,
  region=<inputs.region_yandex>  # код Яндекса, например 213 = Москва
)
```

`se=1` = Яндекс. `region` берётся из `<structure_dir>/inputs.json.region_yandex` (нужно прочитать).

**IDN-домены в маркерах** обычно не встречаются (маркер = поисковый запрос), но если попался - оставь как есть, arsenkin работает с фактическим текстом запроса.

#### 3.2. Развилка по commercial_pct

Из ответа для каждого маркера достань `commercial_pct` (доля коммерческих сайтов в ТОП-10 в процентах, 0-100).

**Пороги по типу страницы:**

| Тип страницы | Порог OK | Borderline | Низкий |
|---|---|---|---|
| `home` (главная) | `>= 30%` | `15-29%` | `< 15%` (главная часто mixed-intent, поэтому порог ниже) |
| `category`, `service`, `product` | `>= 60%` | `40-59%` | `< 40%` |

**Действия по результату:**

| Зона | Действие | Поле в записи |
|---|---|---|
| **OK** | Ничего, маркер подтверждён | `commerce_note: null` |
| **Borderline** | Записать предупреждение, продолжить с этим маркером | `commerce_note: "borderline"` |
| **Низкий** | Попытаться найти коммерческий синоним - см. шаг 3.3 | зависит от результата 3.3 |

#### 3.3. Поиск коммерческого синонима (только для «Низкий»)

Для каждого маркера с низким `commercial_pct`:

1. Запросить кандидатов:
```
keyword_similar(keyword="<маркер>", base="<keyso_base>")
```

2. Отобрать **до 3 кандидатов** с близкой частотностью (`ws_exact >= 30% от исходного` чтобы не сильно терять трафик).

3. Если кандидатов нашлось - собрать их в **один пакет** и проверить через второй вызов `arsenkin_commerce` (один на всю задачу). Опционально объединить с другими «низкими» маркерами в один пакетный вызов.

4. Если среди кандидатов есть с `commercial_pct >= порог OK` для этого типа страницы - **заменить** маркер (авто-решение 1:1, `confidence: high`):
   - В записи: `marker = <новый>`, `ws_exact = <новая частотность>`, `commerce_note = "replaced_marker"`, `original_marker = "<старый>"`.
   - **Запиши в журнал** (см. раздел «Журнал решений»): `kind: "marker_swap"`, `confidence: "high"`.
   - В логах сводки - отметить замену.

5. Если **ни один** синоним не прошёл (`info_dominant`) - **НЕ сваливай на клиента, прими решение сам** (`confidence: low`, структурное):
   - `commerce_note = "info_dominant"` (остаётся - `semantic-expander` по нему пропустит страницу при JM-расширении, экономя лимиты).
   - **Переназначь роль:** `role = "umbrella"` - страница остаётся как зонтичный/навигационный хаб раздела, но НЕ продвигается по мёртвому инфо-маркеру.
   - **Коммерческий вес - на конкретику.** Проверь, есть ли в `master_list` страница, покрывающая коммерческий вариант запроса (например для «сосуды под давлением» это «изготовление сосудов под давлением» = страница Производство). Если есть - запиши routing в журнал (`kind: "route_commercial"`, на какую страницу-дом). Если коммерческой посадки нет - запиши `kind: "need_commercial_landing"` (её создаст шаг рекомендаций/каннибализации).
   - **Чистый инфо-запрос - в блог.** Сам head-термин (например «сосуд под давлением», большая инфо-частотность) запиши решением `kind: "route_to_blog"` (тема для контент-плана У6).
   - `commerce_warning` - **внутренняя нейтральная заметка** (НЕ «решение за клиентом»): `"info-выдача (commercial_pct <X>%); роль -> umbrella; коммерция на <страница-дом или 'нужна посадка'>; инфо-запрос -> блог. См. decisions.json."`

#### 3.4. Что записать в каждую запись страницы

Помимо существующих полей, добавь:

```json
{
  "role": "target" | "umbrella" | "navigational" | "info" | "article",
  "commercial_pct": 75,
  "commerce_note": null | "borderline" | "replaced_marker" | "info_dominant",
  "commerce_warning": null | "<внутренняя нейтральная заметка, НЕ 'решение за клиентом'>",
  "original_marker": null | "<старый маркер если заменён>",
  "commerce_top10_sample": ["leader.ru", "comp2.ru", "blog.ru"]  // топ-3 домена для контекста
}
```

`role` по умолчанию: `target` для коммерческих страниц (home/category/service/product с живым маркером), `info` для `type == "info"`, `umbrella`/`navigational`/`article` - только если назначил по шагу 5. Это **простановка алгоритма**, в xlsx показывается для прозрачности, клиент её не заполняет.

#### 3.5. Бюджет

- 1 пакетный `arsenkin_commerce` на все коммерческие маркеры
- 0-3 повторных `arsenkin_commerce` на кандидатов-синонимов
- 0-5 `keyword_similar` для поиска синонимов

Итого: **1-9 вызовов** на этап. Если упёрся в общий лимит - не дроби, помечай `commerce_note: "not_verified"` для непроверенных и продолжай.

#### 3.6. Журнал решений (`decisions.json`)

Раз клиент больше не SEO-арбитр - все SEO-решения пишутся в машинный журнал (аудит + обратимость + идемпотентность). Ты **создаёшь** `<structure_dir>/decisions.json` (если его нет - начни с `{"decisions": []}`).

Формат записи:
```json
{
  "id": "<kind>:<page_id>",            // стабильный ключ, например "role_reassign:sosudy-pod-davleniem"
  "kind": "marker_swap" | "role_reassign" | "route_to_blog" | "route_commercial" | "need_commercial_landing",
  "page_id": "<id страницы из master_list>",
  "page_n_at_decision": 11,             // снимок n на момент решения (информационно, НЕ ключ)
  "what": "<что изменили: 'роль category -> umbrella'>",
  "why": "<данные-основание: 'маркер info_dominant, commercial_pct=0, ТОП-10 нормативка'>",
  "source": "marker-finder",
  "reversible": true,
  "confidence": "high" | "low",         // high = синоним 1:1; low = структурное (роль/блог/посадка)
  "applied_at": "<ISO UTC>"
}
```

**Идемпотентность (обязательно).** Перед добавлением записи прочитай `decisions.json` и проверь: если запись с таким `id` уже есть - **не добавляй второй раз** и не применяй решение повторно (защита от двойного применения при resume/повторном прогоне). `id` стабилен, потому что `page_id` детерминированный (slug названия).

`confidence`: `high` применяется молча; `low` (структурные) применяются тоже (клиента не спрашиваем), но `structure-writer` вынесет их в A6.md отдельным списком «проверить при желании», и в `--review` оркестратор сделает на них паузу.

### 4. Опц. ручная проверка SERP при особых сомнениях

**Только** если после проверки коммерциализации остались дополнительные вопросы (например, в borderline-зоне хочется увидеть **какие именно** сайты в ТОП-10) - можно дополнительно глянуть выдачу:
```
arsenkin_top(queries=["<маркер>"], se=[{"type": 1, "region": <region_yandex>}], depth=10)
```
Это **не обязательный шаг** и **не для каждого маркера** - только при разборе спорных случаев (≤ 3 запросов на всю задачу).

### 5. Особые случаи

- **Товарные страницы (для ИМ).** Если `master_list.pages[].type == "product"` и **общее количество товарных страниц > 20** - маркеры **только для категорий**, не для товаров. Товарные наследуют семантику от категорий. В записи product-страницы поставь `marker = null`, `notes = "product inherits from category"`.
- Если товарных страниц <= 20 - маркер для каждой.

## Сводка в чат (6-9 строк)

- Страниц с маркером: `<X>` из `<N>` (`<Y>` информационных - без маркера, `<Z>` товарных наследуют от категорий)
- Источники маркеров: лидер `<A>`, другие конкуренты `<B>`, keyword_info `<C>`, keyword_similar `<D>`, ручные `<E>`
- Источники частотности: Keyso `<X>`, jm_wordstat `<Y>`, wk `<Z>`
- **Коммерциализация:** OK `<X>`, borderline `<Y>`, заменено на синонимы `<Z>`, info-dominant `<W>`
- ⚠️ **info-dominant маркеры** (страницы с инфо-выдачей): `<список «название (X%)»>` - **роль авто-переназначена на umbrella, инфо-запрос -> блог, коммерция -> страница-дом** (всё в decisions.json), будут пропущены при JM-расширении. Клиента НЕ спрашиваем.
- ⚠️ **Заменённые маркеры** (нашёлся коммерческий синоним): `<список «название_страницы: старый -> новый»>`
- **Решений в журнал:** `<кол-во>` (marker_swap `<H high>`, структурных `<L low>`)
- ⚠️ Ручные маркеры: `<список названий страниц>` - проверить вручную
- Маркеров с нулевой частотностью: `<N>` (все попали в `manual_warning`)
- MCP-вызовов: всего `<total>` (включая `<commerce>` arsenkin_commerce)

## Запреты

- НЕ запускай JM `semantic_pack` - это задача `semantic-expander` (следующий шаг).
- НЕ редактируй `master_list.json` - только Read. (Свои решения пиши в `decisions.json`, не в master_list.)
- НЕ редактируй файлы в `analyses/NNN/`.
- НЕ пиши «решение за клиентом» / «клиент должен решить» по SEO-вопросам. Прими решение сам (роль / синоним / блог) и запиши в `decisions.json`. Клиенту - только бизнес-вопрос «производите ли это» (в xlsx, не здесь).
- НЕ применяй решение, если запись с таким `id` уже есть в `decisions.json` (идемпотентность).
- НЕ дублируй маркеры (один маркер = одна страница). Если каскад дал одинаковый маркер двум разным страницам - выбери для одной альтернативу (следующий по частоте из той же `domain_keywords` выборки).
- НЕ путай `keyso_base` (для Keyso) и `region_yandex` (для JM/Арсенкин).
- НЕ забудь IDN-домены в кириллице (не Punycode) при `domain_keywords`.
