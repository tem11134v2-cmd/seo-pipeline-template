# seo-faq

Скил-оркестратор SEO-нормализации текстовой релевантности. Запускается **в worktree-сессии**. Порт авторской фазы У6-Ф2 (claude.ai) с JM-анализом.

**Что делает:** добавляет к готовой странице **один SEO-блок** (вставляемый в конец) - FAQ с микроразметкой Schema.org FAQPage. Внутри ответов - **контекстные ссылки** на смежные страницы (анкор = уже существующая в ответе фраза). Содержимое не «вода», а **недостающие по семантике ключи/N-граммы**, вшитые естественным языком - так нормализуется текстовая оптимизация страницы под Яндекс без переспама. «Правильная перелинковка» здесь - это ссылки ВНУТРИ текста ответов, а не отдельный блок-список.

**Дополняет /seo-tekst:** там пишут конверсию (Ф1, для людей), тут - SEO-слой (Ф2, для поиска). Но `/seo-faq` самостоятелен: работает и на чужих живых страницах, не только на свежесгенерённых.

**Главный клиентский артефакт** - единый Google Doc из 2 разделов: (1) Текстовый FAQ (человекочитаемо, с настоящими гиперссылками-в-словах + адрес в скобках), (2) Schema.org (FAQPage) - моноширинный JSON-LD для разработчика. `faq.html` - СЛУЖЕБНЫЙ файл (гейт для verify), заказчику НЕ предъявляется.

## Аргументы
```
/seo-faq [--from-tekst <NNN>] [--from-table <путь>] [--url <URL>] [--concurrency N] [--review|--auto] [--resume]
```
- Без источника - **спросит**.
- `--from-tekst <NNN>` - страницы из `texts/<NNN>-*/` (текст из page.json + запросы из pages.json). NNN faq зеркалит NNN текстов.
- `--from-table <путь>` - таблица URL/Маркер/Запросы[/Текст] (csv/tsv).
- `--url <URL>` - одна живая страница (faq-builder сам спарсит текст; дай маркер/запросы).
- `--concurrency N` (по умолчанию **1**) - сколько faq-builder запускать одновременно. **Дефолт 1 = последовательно** (см. шаг 4). N>1 - на свой риск (cross-talk + перерасход лимитов на одном JM-аккаунте).
- `--auto` (по умолчанию) - автономно. `--review` - пауза перед загрузкой в Drive.

## State machine
```
init -> pages-ready -> built -> verified -> shared -> completed
```
`meta.json` - источник истины (`bash .claude/hooks/update-meta.sh <faq_dir> <state>`).

## Артефакты
```
faq/NNN-<slug>/
├── meta.json
├── inputs.json            # slug/регион/бренд/стоп-домены/источник + interlink_pool[]
├── pages.json             # страницы: text + queries + marker + url
├── pages/<page-slug>/
│   ├── faq_blocks.json     # FAQ (q/a/links) + faq_overflow + detected_existing_faq + normalized_keywords (faq-builder)
│   ├── faq.html            # СЛУЖЕБНЫЙ гейт для verify (аккордеон + Schema.org); НЕ клиентский артефакт
│   └── faq.md              # читабельный дамп
├── FAQ_<slug>.docx         # КЛИЕНТУ -> Google Doc (2 раздела: Текстовый FAQ + Schema.org)
└── share.json
```

## Алгоритм

### 0a. worktree-проверка / 0b. parse args
Как в /seo-tekst (предупредить если main). `from_tekst / from_table / url / concurrency (дефолт 1) / review|auto / resume`.

### 1. Setup
- `--resume`: найти `faq/<NNN>-*/`, прочитать `meta.json`, спросить продолжить.
- Источник (фрэш): если не задан - спросить (1: из текстов NNN; 2: таблицей; 3: живой URL).
- slug/NNN/регион/бренд/стоп-домены -> `inputs.json`. Из `--from-tekst`: взять slug/регион/бренд из `texts/NNN/inputs.json`, **NNN зеркалит**. Иначе из `ЗАКАЗЧИК.md` (`_client.mjs --field`) или спросить. NNN - следующий свободный в `faq/`.
- **Собрать `interlink_pool`** (реальные смежные URL) в `inputs.json.interlink_pool`. Источники по приоритету: (1) `structure_dir/A6.*`; (2) соседние страницы того же `texts/NNN`; (3) sitemap сайта (best-effort: `seo_fetch_page(<origin>/sitemap.xml)` или `wm_sitemaps`; распарсить `<loc>`; недоступно - пропустить); (4) при `--from-table` - URL других строк. **Лимит** 15-20 наиболее тематически близких URL (большой sitemap целиком не вываливать). Форма элемента - `{url, anchor?}` (`anchor` - НЕобязательная подсказка). **Self-фильтр пула делается на шаге 4** (пул общий для всех страниц; URL текущей страницы исключает писатель/verify по `pages.json[].url`).
- Создать `faq/<NNN>-<slug>/`. **`.claude/tmp/current-task.txt = faq/<NNN>-<slug>/`** (критично).

### 2. JM-баланс (гейт)
`jm_account` - если баланс < 5, **стоп** с сообщением «пополни JM, затем /seo-faq --resume» (faq-builder без анализа бесполезен). Иначе дальше.

### 3. Страницы (state == init)
```
.claude\scripts\_node.cmd .claude\scripts\read-faq-input.mjs <faq_dir> --from-tekst <texts_dir> | --from-table <путь> | --url <URL> [--marker "..."] [--queries "a|b"]
```
Exit 2 - нет страниц (стоп). Предупреждения «без текста/без запросов» - показать. `update-meta.sh <faq_dir> pages-ready`.

### 4. Сборка FAQ-блоков - ПОСЛЕДОВАТЕЛЬНО, по странице (state == pages-ready)
Делегировать `faq-builder` **по ОДНОЙ странице за раз** (concurrency = 1 по умолчанию): следующий стартует ПОСЛЕ завершения предыдущего. **Почему не параллельно:** тяжёлые JM-модули (`jm_text_analyze` ~5 мин, Акварель `jm_text_check`) сериализуются на сервере одного JM-аккаунта - параллельный веер НЕ ускоряет (сервер всё равно в очередь), но даёт **cross-talk** (синхронная обёртка возвращает результат ЧУЖОЙ страницы) и перерасход лимитов на ретраях после таймаутов. Подтверждено живым тестом 2026-06-24 (6 параллельных -> 3+ агентов получили чужой gap-анализ/Акварель -> «тихо неправильный» FAQ).
- Промт: `faq_dir`, `project_root`, `page_slug`; пул смежных URL - в `inputs.json.interlink_pool`. Повтори правила ссылок: доля со ссылкой ~0,5 (коридор [0,3; 0,6]); max 1 ссылка/ответ; анкор - уже существующая в ответе фраза по границам слов; url - ТОЛЬКО из пула; **НИКОГДА на текущий URL** (self); `a` - чистый текст (Schema.org).
- `--concurrency N` (дефолт 1): N>1 запускает по N writer'ов одновременно - **на свой риск** (cross-talk + перерасход на одном JM-аккаунте; на 6 проверено - небезопасно). Практический потолок 2, и ТОЛЬКО потому, что faq-builder обязан верифицировать свою JM-задачу по tid/url (см. агента - «Надёжная работа с JM»). Writer'ы без expected-маркеров - полноту проверит шаг 5.
**СКИП:** страницы с пустыми И `url`, И `text` исключить (анализировать нечего) - залогировать.
После всех: проверить наличие `pages/<slug>/faq_blocks.json`; отсутствующие - пере-делегировать (до 2). `update-meta.sh <faq_dir> built`.

### 5. Рендер + проверка (state == built)
По каждой странице - цепочка build -> verify (гейт перед docx):
```
.claude\scripts\_node.cmd .claude\scripts\build-faq.mjs <faq_dir>/pages/<slug>/
.claude\scripts\_node.cmd .claude\scripts\verify-faq.mjs <faq_dir>/pages/<slug>/
```
Exit 2 verify - показать нарушения, пере-делегировать страницу faq-builder с инструкцией исправить (до 2). **Матрица отказа:** если после 2 пере-делегирований остаётся та же V именно из-за ссылки (self / url-вне-пула) - НЕ виснуть: выкинуть нарушающую `link` из `faq_blocks.json`, пересобрать `faq.html` без неё (`build-faq.mjs`), понизить до W и идти дальше. `update-meta.sh <faq_dir> verified`.

### 6. Документ клиенту + Drive (state == verified)
```
.claude\scripts\_node.cmd .claude\scripts\build-faq-docx.mjs <faq_dir>
```
**Стоп-условие:** если НИ У ОДНОЙ страницы нет `faq[]` (полный провал веера) - не генерить пустой docx и не грузить; стоп с подсказкой `/seo-faq --resume`.
Загрузить `FAQ_<slug>.docx` в Drive (Google Doc), ключ **`faq_folder_id`** (отдельная папка «FAQ»); если `faq_folder_id` не задан / `TODO_*` - **фолбэк на `texts_folder_id`** (обратная совместимость); нет и его - скип с подсказкой. mimeType docx + `convertToGoogleFormat:true`. Ссылка в `share.json`. `--review`: спросить перед загрузкой. `update-meta.sh <faq_dir> shared`.

### 7. Финал (state == shared)
`update-meta.sh <faq_dir> completed`. Коммит:
```bash
git add -A && git commit -m "FAQ <NNN> for <slug>: FAQ на <N> страниц"
```
Вывести:
```
═══ SEO-БЛОКИ ГОТОВЫ ═══
Клиент: <domain|slug>   Страниц: <N>   FAQ-вопросов: <total>
📄 FAQ (Google Doc, 2 раздела: Текстовый FAQ + Schema.org): <ссылка>
🔑 Нормализовано ключей/N-грамм: <total>
Дальше: /handoff - перенести в main.
═══════════════════════════
```
Подчеркни: вопросы-ответы - это то, что ДОБАВЛЯЕТСЯ в конец страницы (не заменяет контент). Документ из 2 разделов - канонический клиентский артефакт; `faq.html` - служебный (для verify), заказчику не предъявляется.

## Конкурентность / API-ошибки
**JM-аккаунт один; тяжёлые модули (`txt_anlz`, Акварель) сериализуются на сервере (~5 мин/задача).** Параллельный веер не ускоряет (сервер всё равно в очередь), но перепутывает ответы между страницами (cross-talk) и плодит дубли submit -> по умолчанию **последовательно** (concurrency 1). Каждый gap-анализ/Акварель тогда гарантированно про свою страницу.
`529/503/rate_limit` - `ScheduleWakeup` 90 сек, `/seo-faq --resume <NNN>`, до 3 попыток (страховка; разовый `503` nginx проходит с ретрая). **Timeout обёртки JM != провал** - задача создана на сервере, faq-builder добирает результат по tid с проверкой принадлежности (см. агента «Надёжная работа с JM»), а не пере-submit вслепую.

## Запреты
- НЕ пиши в корень/общие файлы - только в `faq/<NNN>/`, `.claude/tmp/`.
- НЕ редактируй kit (`seo-tekst/assets/`) - read-only (читаем блок 33 FAQ + Schema.org-правило + COPY).
- НЕ переспамь ключами (Акварель/тошнота в норме) - faq-builder контролирует.
- НЕ выдумывай цифры/URL - `[ЗАПОЛНИТЬ: ...]` / url только из `interlink_pool`, никогда self.
- Длинное/среднее тире (— –) запрещено - только дефис (-).
- НЕ запускай другие скилы из этой сессии.
