# REFERENCE.md - редкие ветки скила seo-statya

Ядро алгоритма живет в `SKILL.md`. Этот файл - вынесенные редкие/аварийные ветки и
сводные таблицы. Оркестратор читает его **точечно по якорю-разделу** (Read нужного
`## <якорь>`), а не целиком - иначе теряется смысл разреза. Каждый раздел ниже вызван
из `SKILL.md` строкой `см. REFERENCE.md#<якорь>`.

Разделы (якоря):

- `#rebuild-docx` - recovery-режим пересборки docx готовой статьи (перенос шага 0c).
- `#collision-genre` - детальные развилки выбора жанра/площадки и вторая статья на тему.
- `#api-errors` - полная схема обработки временных ошибок Anthropic API (ScheduleWakeup).
- `#retries` - сводная таблица всех гейтов-проверок конвейера (exit-код -> действие -> лимит).
- `#photo-offline` - MCP генерации/публикации фото оффлайн: ожидание, аварийный режим, data:-fallback.
- `#tilda` - ветка tilda-split (разбор HTML под Тильду).
- `#serial-final` - серийный финал (папка-экспорт, сводная таблица, архив, Drive, сообщение).
- `#serial-mode-details` - подробности серийного режима (изоляция сбоев, --finalize-batch, --resume).

Правило письма этого файла: в перенесенных дословно блоках сохранена исходная орфография
(включая букву е с точками и тире оригинала); в новых соединительных абзацах - только
дефис и буква е без точек.

## rebuild-docx

Recovery-режим `--rebuild-docx` (улучшение #3). Перенос шага 0c из старого ядра.

Если передан `--rebuild-docx` — это не полный прогон, а быстрая пересборка docx уже готовой статьи. Позиционный аргумент = `NNN` статьи (не номер темы). State machine не запускается.

1. Резолвить папку детерминированно (NNN после точки 2 не уникален):
   ```
   .claude\scripts\_node.cmd .claude\scripts\resolve-article-dir.mjs articles <id>
   ```
   - `found == false` → стоп «Статья `<id>` не найдена».
   - `ambiguous == true` → показать `candidates` (имя/жанр/площадка/state), попросить уточнить полный id папки. НЕ брать первую молча.
   - иначе `dir = <ответ>.dir`.
2. Прочитать `meta.json`. Если `state` < `assembled` — стоп «Статья ещё не дошла до сборки, запусти обычный `/seo-statya N --resume`».
3. Пересобрать docx:
   ```
   .claude\scripts\_node.cmd .claude\scripts\build-article-docx.mjs <dir>
   ```
   Обработать exit-код как в шаге 12 (exit 3 = docx неполный → ретрай, лимит 2; см. баг #6). Не продолжать с неполным docx.
4. Если в `meta.share.drive_id` есть значение — удалить старый файл в Drive: `mcp__gdrive-piotr__deleteItem(itemId=<drive_id>)` (если упало — предупредить, продолжать).
5. Перезалить новый docx как в шаге 13 (явный `mimeType`, sanity-check). Обновить `meta.share` целиком, включая свежий `build_script_commit` (= `git log -1 --format=%h -- .claude/scripts/build-article-docx.mjs`).
6. Вывести: «🔄 docx статьи NNN пересобран и обновлён в Drive: <docx_url>». СТОП.

Recovery-сценарий «нашли баг скрипта - надо перегенерить N статей»: прогнать цикл `/seo-statya <NNN> --rebuild-docx` по каждой статье с устаревшим `build_script_commit`.

## collision-genre

Детальные под-сценарии выбора жанра/площадки (развилка «>=1 запись» из шага 1a) и
сценарий второй статьи на ту же тему. В ядре шаг 1a оставляет только трехходовую
развилку; точные под-случаи и формулировки вопросов - здесь.

**Если `--resume` (полная логика):**
- Найти все `existing_articles` с `state != "completed"`. Если их несколько — взять самую недавнюю по `updated` (или по NNN, как fallback). Прочитать её `meta.json`, продолжить с её состояния. В `--auto` — без вопросов. В `--review` — короткое «Найдено NNN в state `<state>`, продолжать? [Y/n]».
- Если всех таких нет (все completed) — сказать пользователю: «Все статьи по теме N завершены. Запусти `/seo-statya N` без --resume, чтобы написать новую (в другом жанре).» Стоп.

**Развилка «>=1 запись, есть `genres_available`» (частичный повтор). Возможные сценарии:**
- **`--genre="X"` передан и `X` в `genres_available`** → используем `X`, без вопросов. (Номер папки детерминирован = номер темы, см. 1b; `max+1` больше не нужен.)
- **`--genre="X"` передан, но `X` уже в `genres_done`** → сообщить «По теме N в жанре `X` уже есть статья NNN. Доступные: `genres_available`. Что делать?» и ждать (даже в --auto, потому что коллизия скоупа — пользователь явно дал противоречащий флаг, нужно его решение).
- **флаг не передан** в `--auto` → взять `genres_available[0]` автоматически. Для `platform_target` — взять первый не использованный из приоритета `["site", "external", "social"]` (т.е. `["site", "external", "social"] \ existing.map(e => e.platform_target)`). Если все три уже использованы — использовать `external` как самый частый случай. В чате уведомить одной строкой: «По теме N уже есть статья(и) в жанре(ах) `<genres_done>`. Беру жанр `<genres_available[0]>`, platform=`<выбранная>`.»
- **флаг не передан** в `--review` → спросить:
  > «По теме №N уже сделана статья в жанре(ах) <X>. Доступные из topics.xlsx: <Y>. Какой жанр для новой?»

**Развилка «`genres_available` пуст (все жанры уже использованы)»** — спросить (в обоих режимах):
> «По теме №N все жанры из topics.xlsx уже использованы (<X>). Варианты: 1) переписать существующую (укажи NNN), 2) следующая тема (N+1), 3) отмена. Что делать?»

**Вторая статья на ту же тему.**

Типичный сценарий: одна тема — две статьи разных жанров:
- Статья на сайт клиента (жанр из `topics.xlsx` колонка «Жанры», первый)
- Статья на внешнюю площадку (Дзен, сателлит) — другой жанр (второй из той же колонки)

Шаг 1a автоматизирует это. Просто запусти `/seo-statya N` второй раз — скил увидит коллизию по `topic_id` в `articles/_index.json`, возьмёт следующий доступный жанр и проставит `platform_target=external`. В `--auto` режиме всё произойдёт без вопросов; в `--review` — с подтверждением.

Явное управление:
```
/seo-statya 1                              # первая статья: первый жанр, platform=site
/seo-statya 1                              # вторая: следующий жанр из xlsx, platform=external (по умолчанию)
/seo-statya 1 --genre="Личный опыт"        # явный выбор жанра
/seo-statya 1 --platform=social            # для соцсетей (короче, другой тон)
```

**JM-кеш экономит лимиты:** `jm_text_analyze` кешируется на 4 часа по тем же параметрам, поэтому второй прогон в тот же день не тратит ≥5 лимитов на JM-анализ повторно. Тонкая ручная оптимизация — см. волну 6 (JM-кеш между прогонами).

## api-errors

Полная схема обработки временных ошибок Anthropic API. В ядре - три строки сути,
здесь - тайминги и эскалация.

Любой вызов субагента или MCP-тула может вернуть ошибку `529 Overloaded` / `503 Service Unavailable` / `rate_limit_error` / `tcp reset` от Anthropic API. **Это не баг и не повод останавливать прогон** — поведение скила:

1. Поймал ошибку → не выводить пользователю «упало, перезапустите`/seo-statya --resume`».
2. `ScheduleWakeup` на 90 секунд (можно 60-180 в зависимости от подозреваемой нагрузки) с тем же `/seo-statya N --resume`, чтобы продолжить с текущего состояния.
3. После пробуждения — повторить упавший вызов. Если ошибка та же — `ScheduleWakeup` ещё раз, но уже на 180 секунд. Максимум 3 попытки подряд.
4. Если три попытки подряд провалились — только тогда сказать пользователю и попросить вмешательства.

Так пользователь не видит флапающих ошибок Anthropic и не должен сам жонглировать `--resume`.

В серийном режиме временная 529/503 - **не** повод помечать тему `failed` в batch-queue:
это ScheduleWakeup + `--resume`, тема остается `in-progress` и продолжится с того же
state после пробуждения.

## retries

Сводный индекс всех гейтов-проверок конвейера. Подробная логика каждого гейта описана
по месту в соответствующем шаге `SKILL.md`; эта таблица нужна оркестратору для быстрого
поиска «какой exit-код что значит и каков лимит». Ни один гейт не убран из ядра - это
указатель, а не единственный дом инструкции.

| Гейт (шаг) | Проверка | exit 2 (или иной) -> действие | Лимит | При исчерпании |
|---|---|---|---|---|
| Шаг 3 | `verify-photo-budget.mjs` | ре-делегировать `tz-builder` (добрать метки `[ФОТО:]` до нижней границы вилки; таблицы - другой бюджет, фото не уменьшают) | 2 | предупреждение в чат, продолжить (фото добираются позже) |
| Шаг 4 | `check-section.sh` (SubagentStop) | разобрать stderr, удалить `sections/<NN>-*.md`, повторить `section-writer` | fail-fast: два одинаковых exit 2 подряд | стоп (в серии - тема `failed`, к следующей) |
| Шаг 5 | наличие `article.md` / `report.md` / `metatags.json` | ре-делегировать `article-finalizer` (записать файл через Write по абсолютным путям, содержимое в чат НЕ возвращать) | 2 | стоп, проверить `article-finalizer` вручную. Parent-fallback ЗАПРЕЩЕН (ORCHESTRATION.md п.б) |
| Шаг 5 | `verify-markers.mjs` | ре-делегировать `article-finalizer` (сохранить метки 1-в-1, включая текст в скобках; сверить тела перед записью) | 2 | стоп, ручная правка `article.md`, дальше с битыми метками не идти |
| Шаг 5 | `verify-article-metatags.mjs` | exit 2 -> ре-делегировать `article-finalizer` (перепиши `metatags.json` и «## Метатеги» в `report.md`); exit 1 -> `metatags.json` не записан, вернуться к проверке наличия файлов | 2 | предупреждение в чат, продолжить (метатеги поправимы через сводную таблицу серии) |
| Шаг 9 | `verify-photos.mjs` (после `photo-promter`) | ре-делегировать `photo-promter` (число блоков «## Фото N» = число меток `[ФОТО:]` в `article.md`) | 2 | стоп с диагностикой |
| Шаг 9b | `verify-photos.mjs` (после `photo-producer`) | рассинхрон `urls.json` vs метки: если MCP жив - ре-делегировать `photo-producer` по остатку либо пересобрать `urls.json` под актуальные метки | 1 | записи с `todo` допустимы и не блокируют; MCP оффлайн -> REFERENCE.md#photo-offline |
| Шаг 10c | вердикт `article-verifier` | `needs-fix` -> `article-fixer-batch` (source=verify_report.json, роутинг по `where`) + повторный `assemble-html.mjs` + повторный `article-verifier` | 2 цикла | остаток в `meta.warnings` (`--auto`) / вопрос (`--review`); `fail` -> стоп (в серии - тема `failed`) |
| Шаг 12 | `build-article-docx.mjs` | exit 3 (`Photos embedded X/Y`, X<Y) -> повторить сборку (транзиентный сбой Cloudinary обычно проходит); exit 1 -> ошибка ввода, разобрать stderr | 2 | стоп, показать `Photos embedded X/Y` + stderr, попросить проверить URL в `photos/urls.json` |
| Шаг 13 | Drive upload + sanity-check | нет якоря «Статьи» в DRIVE.md или MCP `gdrive-piotr` недоступен -> skip с `skip_reason`; пустой Google Doc после аплоада -> 1 ретрай | 1 | статья остается `docx-built`, `/share-article <NNN>` догрузит позже |

Общий принцип: гейт возвращает exit-код, оркестратор ветвится по коду и НЕ вычитывает
данные проверки в контекст (ORCHESTRATION.md в/г). Verify-скрипты сами печатают не более
~30 строк нарушений.

## photo-offline

Аварийная ветка шага 9b: MCP генерации (`openrouter-image`) и/или публикации
(`cloudinary-publish`) недоступны. Обычную генерацию и публикацию делает агент
`photo-producer` (прямые MCP-вызовы `mcp__openrouter-image__generate_image` модели
nano-banana-2 и `mcp__cloudinary-publish__publish_image`); ветку ожидания и аварийный
режим держит **оркестратор** - агент не умеет ScheduleWakeup.

`cloudinary-publish` (`publish_image`) и `openrouter-image` (`generate_image`) — локальные Python-серверы с тяжёлым cold-start (импорт `cloudinary`/`pydantic` и т.п.). Под параллельной нагрузкой (много worktree-сессий стартует разом → конкуренция за диск/CPU) они иногда не успевают подняться к началу прогона. Если `ToolSearch «cloudinary»` / `«generate_image»` ничего не вернул — НЕ останавливай прогон сразу, действуй по порядку:

1. **Сначала подожди — в ~87% случаев сервер не упал, а медленно стартует** (по логам проекта: из ~290 сессий ровно 0 переподключений после падения, но большинство медленных коннектов дозревают за 20-30с). `ScheduleWakeup` на 60-90с с тем же `/seo-statya N --resume`, после пробуждения повтори `ToolSearch`. Лимит 2 пробуждения. Обычно на 2-й проверке тул уже на месте — сервер «дозрел».
2. **Переподключения ПОСЛЕ падения не бывает.** Если сервер реально упал по таймауту (в логе подключения `%LOCALAPPDATA%\claude-cli-nodejs\Cache\...\mcp-logs-cloudinary-publish\*.jsonl` строка `Connection failed` / `timed out`) — в этой сессии он уже не оживёт, сколько ни жди. Тогда: в `--review` — стоп, попроси пользователя перезапустить сессию; в `--auto` — перейди в аварийный режим (п.3), не вешай прогон бесконечными пробуждениями.
3. **Аварийный режим (чтобы статья не встала из-за CDN):**
   - Упал только **publish** (картинки сгенерированы локально, но не публикуются): впиши в `urls.json` для каждого PNG `data:`-URL (base64) вместо CDN-ссылки + поле `"todo": "CDN-публикация отложена (MCP publish оффлайн)"`. docx и HTML получат картинки локально. Минус: нет CDN-URL для живого сайта и `og:image`. (Практически это делает сам `photo-producer` per-photo; оркестратор лишь фиксирует пропуск, если агент вернул отложенные фото.)
   - Упала **генерация** (`openrouter-image`): пометь все фото в `urls.json` как `{"photo": N, "todo": "генерация отложена (MCP оффлайн)", "prompt": "..."}` и иди дальше — статья соберётся без изображений, фото добиваются позже.
   - В обоих случаях зафиксируй пропуск: `update-meta.sh <dir> photos-published skip_reason="MCP CDN/generate оффлайн — фото отложены"`. После восстановления MCP до-опубликовать перезапуском шага 9b (новая сессия) или вручную.
4. **Профилактика (разовая, на стороне пользователя):** базовый фикс уже применён глобально — `MCP_TIMEOUT=120000` (User env Windows) переводит большинство «падений» в «медленный успех». Если оффлайн повторяется часто — не стартуй пачку worktree залпом (разнеси старт по времени, чтобы убрать пик одновременных cold-start) и проверь, что Claude Code Desktop перезапускали после установки переменной (иначе она не подхватится).

**Стоимость на статью.** Nano Banana 2 ≈ $0.07 за изображение × 4-7 фото = $0.28-0.49. Cloudinary в free-tier бесплатно до 25k transformations / 25GB bandwidth — реальный расход на одну статью пренебрежимо мал.

## tilda

Ветка tilda-split (шаг 11 ядра выполняется по этому разделу). Запускается, когда статья
уже прошла верификацию (`state == "verified"`) и платформа клиента - Тильда.

Прочитать `ЗАКАЗЧИК.md`, секция «Платформа и хостинг», поле «Платформа». Если значение содержит «Тильда» (case-insensitive):

```
.claude\scripts\_node.cmd .claude\scripts\tilda-split.mjs <dir>
```

`update-meta.sh <dir> tilda-split`

Если платформа другая — шаг пропустить **с логированием**:
```
update-meta.sh <dir> verified skip_reason="Tilda-split: платформа <X>, не Тильда"
```
(state не меняем - остаётся `verified`, просто фиксируем skip в `meta.skips`).

## serial-final

Серийный финал (шаг 15 ядра). Запускается автоматически, когда `batch-queue.mjs next`
вернул `{"done":true}` (exit 3) - то есть пройдены все темы очереди - или сразу при
`--finalize-batch`. Собирает один самодостаточный набор - **без отдельных просьб
пользователя**. Никаких пауз.

**Переменные шага** (зафиксируй явно, чтобы не путать подстановку):
- `spec` = исходный аргумент `N` (`11-20` / `11,12,15` / `all`).
- `articles_root` = `articles` (относительный путь от корня проекта; скрипты делают `resolve()` от cwd).

0. **Имя клиента - из MAIN-worktree, НЕ из текущей папки** (мы внутри worktree, `--show-toplevel` вернёт имя worktree, а не клиента):
   ```bash
   COMMON_DIR=$(git rev-parse --git-common-dir)
   case "$COMMON_DIR" in /*|[a-zA-Z]:*) ;; *) COMMON_DIR=$(cd "$COMMON_DIR" && pwd) ;; esac
   MAIN_WT=$(dirname "$COMMON_DIR")
   proj=$(basename "$MAIN_WT")     # слаг клиента, напр. traffic_masters_ru
   ```
   (Если запущено в main, `proj` = имя клиентской папки напрямую - тоже верно.)

1. **Освежить индекс** (на всякий случай, экспорт читает meta напрямую, но индекс полезен):
   ```
   .claude\scripts\_node.cmd .claude\scripts\rebuild-index.mjs articles
   ```

2. **Путь папки-экспорта** (на Рабочем столе - вне репо, переживает удаление worktree при `/handoff`):
   - `spec_safe` = spec с заменой `,`→`_` (напр. `11-20` или `11_12_15`).
   - `export_dir = %USERPROFILE%\Desktop\Stati_<proj>_<spec_safe>_<YYYY-MM-DD>` (`proj` из шага 0, сегодняшняя дата).

3. **Собрать папку + манифест:**
   ```
   .claude\scripts\_node.cmd .claude\scripts\export-articles.mjs articles "<export_dir>" "<spec>"
   ```
   Копирует `output-NNN.html` каждой готовой статьи как `NNN - Тема.html` (имена санитайзены под Windows) и пишет `manifest.json`. Пропущенные/не-completed темы печатает - перенеси их в финальное сообщение как «требуют внимания».

4. **Сводная таблица метатегов:**
   ```
   .claude\scripts\_node.cmd .claude\scripts\build-articles-xlsx.mjs "<export_dir>"
   ```
   Создаст `<export_dir>\Метатеги_<spec_safe>.xlsx` (№ · Тема · Жанр · H1 · Title · симв. · Description · симв. · Анонс · файл-черновик · Google Doc; превышения лимитов подсвечены).

5. **Архив** (UTF-8 имена записей - PowerShell/.NET, а не `Compress-Archive`, который ломает кириллицу):
   ```powershell
   Add-Type -AssemblyName System.IO.Compression.FileSystem
   [System.IO.Compression.ZipFile]::CreateFromDirectory("<export_dir>", "<export_dir>.zip", [System.IO.Compression.CompressionLevel]::Optimal, $false, [System.Text.Encoding]::UTF8)
   ```
   Архив `<export_dir>.zip` ложится рядом с папкой на Рабочем столе.

Sanity-check непустого `id`.

## serial-mode-details

Подробности серийного режима (в ядре - краткий цикл на очереди; здесь - изоляция сбоев,
`--finalize-batch`, семантика `--resume`, несколько статей одной темы).

**Изоляция сбоев.** Если на какой-то теме прогон упал или требует ручного вмешательства (например, verify не прошёл лимит ретраев, или коллизия жанра в `--auto`) - **не валить весь батч**: пометить тему `failed` в очереди (`batch-queue.mjs set <N> failed --reason "<кратко>"`), перейти к следующей. Временные API-ошибки 529/503 обрабатываются как обычно (ScheduleWakeup + --resume, REFERENCE.md#api-errors), это не «проблемная тема» - тема остаётся `in-progress`.

**`--resume` для серии:** повторный `/seo-statya 11-20 --resume` = идемпотентный `batch-queue.mjs init "11-20"` (сохраняет уже проставленные `done`/`failed`) + цикл `next`. Для каждой темы, где уже есть `completed`-статья, `done` в очереди уже стоит - написание пропускается; недоделанные (`pending`/`in-progress`) дописываются. В конце - тот же серийный финал. Если все статьи диапазона уже `done` - `next` сразу вернёт `{"done":true}` (exit 3), это идемпотентная пере-сборка финала, аналог `--finalize-batch`.

**`--finalize-batch`:** пропустить цикл написания, сразу шаг 15 (REFERENCE.md#serial-final) по уже готовым статьям диапазона. С `all` - по всем `completed`-папкам `articles/*` (список тем считает `batch-queue.mjs`, не оркестратор - диета контекста).

**Несколько статей одной темы в серии.** По одной теме можно написать несколько статей в разных жанрах для разных площадок (см. REFERENCE.md#collision-genre и шаг 1a). Скил сам разруливает коллизию по `topic_id` через производный индекс (`rebuild-index.mjs` → `articles/_index.json`; ключ записи = имя папки, поэтому несколько статей одной темы не схлопываются).

**Коллизия двух серий в одной worktree.** `batch-queue.mjs init` при ДРУГОМ spec с незавершёнными темами возвращает exit 1 «активна другая серия <spec>, заверши или удали batch-queue.json» - явная остановка, чтобы не смешать две очереди. Если все темы прошлой серии `done` - init перезаписывает очередь новым spec.
